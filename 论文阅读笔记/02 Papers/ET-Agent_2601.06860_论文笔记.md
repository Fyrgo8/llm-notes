# ET-Agent（arXiv:2601.06860）论文笔记

Yifei Chen, Guanting Dong, Zhicheng Dou | arXiv v2, 2026 | [PDF](https://arxiv.org/pdf/2601.06860v2) | `Agentic RL` `Tool Use` 

## 问题与核心判断

ET-Agent 关注的不是 agent 能否调用工具，而是其调用行为的**有效性**。现有 TIR（Tool-Integrated Reasoning）训练大多把最终答案正确性当作主目标——这会掩盖四种彼此相关、却不会被 final accuracy 充分惩罚的失败模式：① 重复检索或无信息增益的冗余调用；② 格式或代码错误导致的中止执行；③ 该查却过早结束的工具调用不足；④ 工具轨迹中冗长或错误的推理。论文的预实验不仅举例说明了这些问题，还给出了量化证据：数学与知识任务中都存在大量冗余调用；对错误轨迹追加提示并允许继续推理时，约 20.8% 的失败能够被修复。这说明一部分错误并非能力完全缺失，而是**策略过早收束**（premature convergence）。

论文的判断是：TIR 的训练目标应从“答对”扩展为“以正确、可执行、紧凑的轨迹答对”。这一判断本身不新颖，ET-Agent 的实质增量在于把它落实为一个两段式闭环：先制造比当前 policy 更广的工具使用轨迹分布，再在这个分布上做组内相对强化。作者反对直接以 DPO 一类二元偏好训练处理该问题，理由是同一问题可对应不同 query、调用顺序和次数，动作空间远大于一对 chosen/rejected 轨迹所能覆盖的范围；也不满足于仅按答案 reward 做 RL，因为那会让“多调用换正确率”的策略继续存活。

因此，ET-Agent 更接近一个面向 agent 行为的 on-policy RL pipeline，而不是新的 policy objective。它想解决的关键矛盾是：**行为校正需要候选轨迹之间有足够差异，但未校正模型的采样又可能同质或局部**。

行为校正需要候选轨迹之间有足够的差异：你想让模型学会"少调用、紧凑推理"，得先让它看到"多调用 vs 少调用""堆砌推理 vs 精简推理"这些做法之间的对比。如果所有候选轨迹都一样，模型没有比较对象，学不到东西。

而未校正模型的采样又可能同质或局部：刚开始训练的模型本来就是不完美的——它生成的轨迹往往差不多烂，模式单一，多样性不足，没法提供足够的差异信号。

因此**先扩探索、后按相对质量收敛**是整篇论文唯一真正承重的叙事——其他细节（f_{tool} 函数、sigma curriculum 等）都是围绕这个核心叙事服务的工具。

## 机制与训练闭环

ET-Agent 的训练流程围绕一条核心叙事展开：**先扩探索 → 后按相对质量收敛**。具体分为三个环节：

```
飞轮（离线造数据，不更新参数） → RFT（监督学习热身） → RL 收敛（组内相对比较）
                                    ↑
                            Pareto sampling 挑题过滤
```

---

### 环节一：Self-Evolving Data Flywheel ⭐ 核心创新

**核心思想**：改造比丢弃更有价值。对正确轨迹砍冗余，对错误轨迹补探索，而非简单丢弃。

**整个飞轮不需要人工标注或外部大模型**，全靠规则 + 当前模型自身完成自动化闭环。

#### 步骤 1：生成原始轨迹

对每个问题，用当前模型采样多条轨迹，按答案对错（规则比对标准答案）分为 Correct / Incorrect Set。

#### 步骤 2：两组走不同改造路径

| 分组 | 改造方法 | 作用 |
|:----|:---------|:-----|
| **Correct Set**（答对） | `Redundant Modification`：定位最早冗余步骤，从该处截断重写 | 剪掉"知道答案还要多查一次"的浪费 |
| | `Global Refinement`：保留工具序列和逻辑，只压缩思考内容 | 去掉啰嗦推理，留必要逻辑 |
| **Incorrect Set**（答错） | `Self-Correction`：找到首次出错位置，从该处修正后续推理 | 让模型学会"哪里错了、怎么改" |
| | `Hint Injection`：在错误处/末尾插入"再查一下"提示 | 反制 premature closure（过早停止） |

**实现机制**：
- **划分 Correct / Incorrect**：规则比对标准答案（MATH、AMC、AIME 等有确定答案的任务）
- **定位冗余**：逐个移除工具调用，若答案仍正确则该步冗余
- **定位错误**：从前往后检查每步输出，找到首个导致最终答案出错的步骤
- **修改/重写**：用**当前模型自己**（Qwen2.5-7B）生成新内容，不用 GPT-4 等外部模型

#### 步骤 3：迭代得到 D_aug

所有改造后轨迹回灌池中，**迭代多轮**。模型越好→修改质量越高→正向循环，这就是 **Self-Evolving** 的含义。最终产出 `D_aug`——**不是偏好对**（chosen/rejected），而是覆盖了"删冗余"与"补探索"两个方向的多样化训练分布。

### 环节二：RFT 热身 ⭐ 核心创新配套

对通过答案、格式和执行质量控制的 `D_aug` 做 **RFT**（Reward-Guided Fine-Tuning）：

> **RFT 不是采样，是监督学习**。模型看到的是「原始问题 → 改造后的轨迹」（如"长8宽6求对角线 → 被砍掉冗余后的紧凑版""2024诺贝尔化学奖 → 被插入 hint 后继续检索的版本"），通过 MLE 学会输出这些行为模式。目标函数是：

$$ \max_\theta\; \mathbb{E}_{(x,y)\sim D_{aug}}\left[\sum_{t=1}^{|y|}\log P_\theta(y_t\mid x,y_{<t})\right]. $$

**为什么必须做 RFT，不能直接上 RL？**  
如果模型根本不会生成飞轮扩展出的多样化轨迹，RL 阶段采到的候选还是老一套，同题 16 条轨迹都相似，group-relative advantage ≈ 0，RL 训不动。RFT 就是让模型先**学会"能生成"**这些新行为模式（紧凑的、会纠错的、不会过早停的），为后续 RL **制造组内差异基础**。

Figure 7 的 t-SNE 佐证了这一设计：从 Instruct（基座）到 RFT，轨迹嵌入分布**变散**（多样性增加）；进入 RL 后**又收紧**（收敛到高效行为）。

---

### 桥梁：Pareto Sampling 挑题过滤

不是所有问题都适合做组内比较——如果一道题的 16 条轨迹全都差不多，组内比较没有信号。因此 RL 前先筛选问题：

| 指标 | 计算方法 | 含义 |
|:----|:---------|:-----|
| **`S_corr`**（正确性离散度） | 熵：`S = -[p·log(p) + (1-p)·log(1-p)]` | `p=0.5`（一半对）时最大；全对/全错时=0 |
| **`S_tool`**（调用次数离散度） | 变异系数：`CV = σ/μ` | 归一化均值差异，调用 2~8 次（μ=5, σ=3）时≈0.6 |

**Pareto sampling** 保留两个维度上**非支配**（不存在另一个问题同时在两项上都更高）的 query，辅以 crowding distance 保证分布均匀。**目标不是挑高分题或容易题，而是挑组内既有结果差异、也有行为差异的题**——避免 RL 因候选同质而梯度衰减。

---

### 环节三：RL 收敛（非核心创新）

> 以下 reward 设计和 ARPO 更新**不是论文的核心创新**，只是常规的 RL 工具。真正的创新在前面——飞轮 + RFT + Pareto sampling 为这个常规优化器提供了"有差异可比较"的候选轨迹。

**Reward 公式**（乘法结构：正确性是门槛，效率是方向，格式是惩罚）：

$$ f_{tool} = \frac{1}{1+e^{\sigma_{tool}(v_{i,tool}-\bar v_{tool})}},\qquad
f_{len} = \frac{1}{1+e^{\sigma_{len}(v_{i,len}-\bar v_{len})}}, $$
$$ R_i = R_i^{corr} \cdot f_{tool} \cdot f_{len} + R_i^{format}. $$

- `f_tool` / `f_len`：logistic 函数，奖励**相对同题更省**的轨迹（不是绝对阈值）
- **乘法 vs 加法**：答错时 `R_corr ≈ 0`，整体归零——确保正确性是前提，效率是加分项
- **⚠️ 风险**：效率惩罚过强会把模型推向过早停止→论文用 **curriculum sigma**（三轮逐步减小 σ）缓解

**Policy update（ARPO）**：组内归一化的 advantage + PPO clipping

$$ A_i = \frac{R_i - \bar R}{\text{std}(R)} $$

不训练 value model，不依赖显式 KL 约束。ARPO 本身是常规的 GRPO-family 优化器。

## 证据、定位与边界

### 主实验结果

| 方法 | 平均正确性 | 效率（正确性/调用次数） |
|:----|:---------|:---------------------|
| **ET-Agent** | **60.1** | **46.0** |
| Tool-Light（最强 baseline） | 55.0 | 38.5 |
| Tool-Star | 52.3 | 37.5 |

- **测试任务**：数学（AIME24、AMC23、MATH500）+ 知识问答（2WikiMultiHopQA、Bamboogle、MuSiQue）
- **额外指标**：成功执行率、无冗余调用比例也领先
- **⚠️ 谨慎看待**：一次检索和一次代码执行在现实中成本不同，论文等同计数；且知识任务只用本地 Wikipedia，不等于真实系统成本优化

### 消融实验（各组件贡献）

| 去掉什么 | 正确性 | 效率 | 解读 |
|:--------|:-----|:----|:-----|
| **完整模型** | **60.1** | **46.0** | 基线 |
| ❌ 去掉飞轮 | **47.3** ↓13 | — | **影响最大**——飞轮是核心 |
| ❌ 去掉 Pareto sampling | 54.1 | 40.1 | 没有差异筛选，RL 信号变弱 |
| ❌ 去掉行为 reward | 58.5 | 43.4 | 正确性影响小，效率下降 |
| ❌ **固定 sigma（最有解释力）** | **48.1** ↓12 | 46.0（不变） | 效率惩罚全程不放松 → 模型**不敢探索** → 正确性大跌，但效率没变（因为模型走极端：少调用但乱猜） |

**⚠️ 局限**：消融能证明组件都有用，但无法区分「飞轮数据质量提升」与「动作空间覆盖变广」各自贡献多少，也无法仅靠 t-SNE 证明 RFT 的多样性就是因果来源。

### 方法定位

ET-Agent = **`self-evolving trajectory data + RFT warm-up + group-relative on-policy RL`**

与 Tool-Light 的本质区别：
- **Tool-Light**：用偏好学习（DPO-style）校正轨迹
- **ET-Agent**：**先扩探索分布 → 再做相对强化**

### 代价与前提

| 局限                              | 原因                                    |
| :------------------------------ | :------------------------------------ |
| 💰 每个 query 需 16 条轨迹 + 飞轮多轮生成筛选 | 计算成本高（实验：4 张 A800，RFT 10.5k 数据，RL 三轮） |
| 🧩 更小的模型可能撑不住                   | 小模型自修正质量差                             |
| 📊 答案评估必须可靠                     | 噪声会破坏组内排序                             |
| 🌐 长程/异质工具场景不保证                 | "少调用"不一定就是更好的行为                       |
| 🔒 只验证了本地 Wikipedia 检索          | 没试过 live web 和更大模型                    |

## 研究卡片

- **问题**：只看最终正确性训练 TIR agent → 漏掉冗余调用、调用不足、执行失败等行为错误。ET-Agent 要在不牺牲正确性的前提下把这三种行为也修好。
- **做法**：飞轮（删冗余 + 补探索）→ RFT（热身）→ Pareto sampling（挑差异够大的题）→ ARPO（组内相对比较，正确性 × 相对效率 + 格式惩罚）。
- **证据**：6 个任务正确率与效率同时最高。固定 sigma（效率惩罚不放松）→ 正确率大跌 → 反证完整设计的必要性。
- **前提**：依赖于"调用少 + 链短 = 行为好"的代理假设；真实场景中工具成本各异、长程探索需更多调用、开放网络需更多检索，该假设会变弱。
- **教训**：不要只给 reward 加效率项。正确做法是先造一个能产生相对学习信号的轨迹分布（飞轮），再设计 reward 和采样去利用这些差异（Pareto + ARPO）。缺一不可。
- **遗留**：能否用工具实际成本、信息增益、step-level verifier 替代粗糙的"调用次数/长度"代理？live web 和更大模型上还能保持同样的 correctness-efficiency 双赢吗？
