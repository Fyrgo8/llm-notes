# 工作经验分享 2026 大模型应用算法还能不能去？ @所有人

## Q：大模型应用都在做些什么，只需要写写skills吗？

A：不同大模型应用方向差距蛮大的，比如用户画像、商品理解这种，那就是拿大模型来打打标签，最多就是做做传统的微调训练，技术壁垒确实低，只能积累一些业务经验；比如ai助手、客服这些业务，之前可能在用workflow，调prompt做one-model后训练，现在harness火了，不少大厂中台工程组会出自己的Agent框架，让不同的业务组接进来，业务算法写写skills就好了。

这个最大的问题是，你不知道这个框架的底层实现，也无法在这个env中进行训练，失去了co-design的意义了，只是做大了harness。

所以还是有不少业务组会自己搭建一套适合自己业务的Agent Loop，在此基础上迭代各种策略，同时也需要进行大规模的post train，并随着模型能力的提升不断优化benchmark体系、工具集合、框架层策略等等，其实从技术栈来看，和基座没啥区别，所以有些大厂的核心C端应用也会有基座团队来参与优化。

## Q：如何选择业务方向，哪个方向最值得去？

A：我个人认为，C端是要优于其他的，拿B端来说，虽然它的业务需求点够复杂，但是SOP够明确，场景很有限，所以这就是为啥到现在为止很多B端还在用workflow，因为它确实好用也最适合！

C端的query分布是无法拟合的，你需要尽可能穷举所有可能的需求点，并且在工具设计时也需要考虑它的通用性，同时你需要保证链路可以满足接入的垂域业务，包括不同的场景、业务需要返回给用户什么样的UI，所以这时候使用Agentic Loop反而是一个更好的选择。特别是评测体系的构建，是一个非常令人头疼的事情。当然目前不少C端应用还在以传统搜索的思路在做这个事情，至少从呈现的UI形式上看，是这样的。从以上这些点来看，个人认为C端应用的scope更大，可以做的事更多，优化空间更大。

## Q：如何看待工程和算法做的事越来越像了，会不会以后就不需要应用算法了？

A：我觉得不会，但应用算法会被重新分层。现在看起来工程和算法做的事越来越像，是因为大模型应用里很多关键问题已经不是单点模型问题了，而是模型、工具、状态、链路、数据和评测混在一起的系统问题。比如一个 Agent Loop 里，什么时候调模型、什么时候调工具、怎么维护 memory、怎么做失败兜底、怎么把结果渲染成业务 UI，这些东西既不是传统意义上的纯工程，也不是以前那种纯算法，而是一个围绕模型能力做系统设计的问题。

但这不代表应用算法不重要了，只是低壁垒的应用算法会变少。比如只是接接工具、写写 skills、调调 prompt、配几个 workflow，这类工作确实会越来越像工程配置，也很容易被中台框架吃掉。尤其如果你既不了解底层 Agent 框架，也不能改 env，更不能基于业务链路做 post-train，那本质上就是在一个别人定义好的 harness 里填空，技术成长空间会比较有限。

真正有价值的应用算法，还是要能做 co-design：把业务问题抽象成模型可决策的环境，设计工具和状态空间，构造高质量训练数据，搭 benchmark 衡量链路收益，再随着基座模型能力变化不断重构策略。所以以后不是不需要应用算法，而是不需要只会"调用大模型"的应用算法。更稀缺的是那种既懂模型能力边界，又懂业务复杂性，还能把 Agent 框架、数据闭环和评测体系一起做起来的人。

## Q：应用算法和基座算法的差距到底在哪？以后还有没有机会反向流向基座？

A：我觉得差距主要不在谁更高级，而在问题定义不同。基座算法更关注模型本身，比如预训练、post-train、RL、对齐、长上下文、多模态、推理能力这些，目标是提升一个通用模型的能力边界；应用算法更关注模型在真实业务环境里怎么用起来，比如 query 怎么理解、工具怎么设计、Agent Loop 怎么跑、链路怎么评测、线上反馈怎么变成训练数据。所以基座更像是在提升模型能力上限，应用更像是在解决模型能力如何转化成业务收益。

但是老实讲，业务算法很难回流到基座，本身现在就是需要match的人，加上现在需要的是招你进来能带我飞的人，所以这是没办法的。但是基座的人过来做应用也不好做，举个例子，一个在业务上效果特别牛逼的Agent系统，不仅仅是主Agent模型本身牛逼，工具如何拆解、如何节省token、首token时延优化、模型输出特定问题时的兜底逻辑、评测体系的构建，这些都是需要结合特定的业务来设计的，基座团队是没有遇到过xx业务场景上的bad case，也没有具体的优化经验和业务理解。

## Q：大模型应用算法的核心能力应该怎么积累？

A：我个人认为，第一层能力不是 prompt，而是业务抽象能力。很多人做大模型应用，一上来就写 prompt、调模型、接工具，但其实最关键的是你能不能把一个复杂业务拆成模型可理解、可决策、可评测的任务。比如用户到底是在找店、找品、比价、决策、售后、闲聊，背后需要哪些供给、哪些工具、哪些状态、最终应该返回什么 UI，这些东西如果抽象不好，后面模型再强也只是乱调工具。

第二层是 Agent 系统能力，也就是你要理解一个大模型应用不是一次模型调用，而是一套 harness。包括 query rewrite、intent routing、planner、tool calling、memory、reflection、fallback、result ranking、UI rendering、safety guardrail，这些模块怎么拆、哪些交给模型、哪些交给规则、哪些应该训练，都会直接决定系统上限。只会写 skills 是不够的，至少要能看懂整个 Agent Loop 为什么这么设计，以及它的瓶颈到底在模型、工具、数据还是框架策略。

第三层是数据和评测闭环能力。大模型应用真正拉开差距的地方，不是第一版 demo，而是后面能不能持续迭代。你要知道怎么从线上日志里挖真实 query，怎么构造 SFT/RL 数据，怎么评估 trajectory-level 的好坏，怎么区分模型能力问题和链路工程问题，怎么让 benchmark 跟业务一起演进。简单说，应用算法要从调 prompt 的人变成定义 env、构造数据、评估链路、反哺模型的人，这才是长期壁垒。

## Q：应用算法未来最值得下注的方向是什么？

A：我觉得最值得下注的方向，一定不是泛泛的大模型应用，而是那些具备复杂需求分布、真实用户反馈、复杂工具环境和可训练闭环的方向。比如 C 端 AI 助手、复杂交易型 Agent、本地生活、电商消费类AI助手，这些方向的特点是 query 长尾、供给复杂、用户目标不稳定，而且最终效果能被真实点击、转化、留存，这种场景才有持续优化空间。

相反，如果一个方向 SOP 很固定、场景很窄、工具很少、结果也很容易规则化，那 workflow 可能就够了，不一定需要很重的 Agentic Loop，也不一定需要很强的应用算法。很多 B 端场景看起来很复杂，但本质是流程复杂，不一定是决策复杂；而 C 端很多场景看起来只是问答，其实背后是意图理解、供给检索、个性化排序、工具选择、UI 组织和用户体验的综合问题，scope 会大很多。

所以我会更看好三类方向：第一是核心 C 端助手，因为它离真实用户最近，数据闭环最好；第二是交易型 Agent，因为它不仅要回答，还要帮助用户完成决策和转化；第三是可训练的 Agent Loop，也就是不只是搭框架，而是能把 trajectory、工具调用、用户反馈、业务目标都转成训练信号。未来真正有价值的应用算法，不是把大模型接到业务里，而是让业务环境持续训练和塑造大模型能力。
