## 摘要

大语言模型（LLM）Agent 在生产环境中的快速部署揭示了一个反复出现的模式：任务执行的可靠性更少取决于底层模型，而更多取决于包裹模型的基础设施层——即 Agent 执行 Harness。本综述提供了一项以实践为基础的、系统性的 Agent Harness 工程研究，围绕三个核心论断组织。第一，Agent Harness 是一个独立的系统层，其工程质量驱动了现实世界可靠性的很大份额。我们通过三阶段工程演进（从 Prompt 工程到 Context 工程再到 Harness 工程）、涵盖成本-质量-速度三难困境、能力-控制权衡以及 Harness 耦合问题的跨层综合分析，以及基于研究空白和生产痛点的开放问题议程来发展这一立场。第二，我们提出 ETCLOVG，一个七层分类法（执行环境 Execution environment、工具接口 Tool interface、上下文管理 Context management、生命周期/编排 Lifecycle/Orchestration、可观测性 Observability、验证 Verification、治理 Governance），它通过将可观测性和治理作为独立的架构关注点，扩展了先前的六组件框架。第三，我们将 170 多个开源项目映射到该分类法上，以揭示生态系统模式、覆盖空白和新兴设计原则，同时还提炼了来自 OpenAI、Anthropic 和 LangChain 生产部署的工程原则，弥合了实践者知识与研究术语之间的差距。

## 1 引言

### 1.1 约束瓶颈：Harness 优先于模型

学术界对基于 LLM 的 Agent 的研究，在很大程度上是对模型本身的研究。研究议程围绕模型能做什么展开：它能否跨多步进行规划、可靠地调用工具、检索并压缩相关记忆，或与其他 Agent 协调。其隐含假设是，Agent 能力主要是模型能力的函数——一个足够强大的模型配合一个足够好的 Prompt 就能产生足够可靠的行为。

近期的经验证据挑战了"更好的模型就能产生更可靠的 Agent"这一假设。三项近期成果确立了这一模式。Bölük (2026a) 仅修改了编辑工具的格式及其周围的工具 Harness，未对模型进行任何修改，便在 15 个模型的编码基准测试中报告了高达 10 倍的性能提升。Trivedy (2026) 通过系统提示重构、中间件上下文注入和自验证钩子，将一个固定的 GPT-5.2-Codex Agent 在 Terminal-Bench 2.0 上的表现从 52.8% 提升到 66.5%，仅通过基础设施变更便获得了 13.7 个百分点的增益。Meta-Harness (Lee et al., 2026) 通过自动化 Harness 优化在 Terminal-Bench-2 上达到 76.4%，超越了所有手工设计的方法，且未修改模型权重。在每个案例中，变量都是执行 Harness（即管理上下文构建、工具交互、编排、反馈和执行约束的基础设施层）；模型保持不变。这些仅通过 Harness 实现的增益中的每一项，都超过了在相同基准测试上被报告为有意义的模型进步的典型 2 到 4 个百分点的改进。这一模式并非偶然：驱动结果的是 Harness 而非模型。

我们将这一模式称为约束瓶颈论（binding-constraint thesis）(Bölük, 2026b)：对于在可比较的前沿模型上评估的长时程任务，基准测试的方差可能由执行 Harness 驱动的程度不亚于模型本身。我们以此论题作为本综述其余部分的框架。

图 3（见 §2）从历史角度说明了同一转变：早期系统将能力集中在单一模型循环中，而后期系统越来越多地将可靠性暴露为跨层基础设施问题。

### 1.2 实践者与研究者之间的差距

实践者的紧迫性与研究术语之间存在张力。OpenAI 明确将"Harness 工程"定义为围绕 Codex Agent 设计环境、约束、文档和反馈循环的学科，并在 2026 年 2 月报告称，一个小团队在五个月内产出了约一百万行代码的内部产品，而未手动编写生产代码 (OpenAI, 2026a)。Anthropic 的 Agent 工程帖子从相邻方向得出了相同原则：有效的 Agent 应使用简单且可检视的架构，工具接口应为 Agent 使用而设计而非从面向人类的 API 中照搬，上下文应渐进式披露而非急切加载，长期运行的工作需要持久化的交接产物和可恢复的执行基础设施 (Anthropic, 2024a; Aizawa, 2025; Anthropic Applied AI Team, 2025; Anthropic, 2025d; 2026b)。Martin Fowler 网站上的一篇文章将 Harness 工程描述为"AI Agent 的控制论调节器"，由前馈引导和反馈传感器构成围绕 LLM 的控制回路 (Böckeler, 2026)。

与此同时，研究社区一直在以越来越高的精度研究 Agent 系统的组成部分：记忆、工具使用、规划和安全。但尚未被系统性研究的是将这些组件整合为可靠运行的系统。其结果是一个实践者-研究者差距：实践者知道 Harness 基础设施很重要，但缺乏正式的术语来描述其重要性的原因，以便实现系统性改进。本综述试图弥合这一差距。

### 1.3 范围与贡献

本综述聚焦于包裹语言模型以管理长期运行、多步骤任务执行的基础设施层。我们不综述作为开发工具的 Agent 框架、作为产品类别的 Agent 平台，也不综述模型能力本身，尽管三者都为我们的分析提供了信息。图 4 总结了构建本综述其余部分的七层分类法。

我们的贡献围绕三个论断组织。

1. **论断 1（概念性）**：基于约束瓶颈论 (Bölük, 2026b)，我们论证 Harness 而非仅模型本身，是现实世界 Agent 可靠性的约束瓶颈。三项近期成果显示，仅通过 Harness 变更便在编码基准测试上实现了高达 10 倍的增益、在 Terminal-Bench 2.0 上获得了 +13.7 个百分点的提升、以及在 Terminal-Bench-2 上达到 76.4%（§1），每项都超过了在相同基准测试上典型的模型驱动增益。我们通过三阶段工程演进（§2）、涵盖成本-质量-速度三难困境、能力-控制权衡和 Harness 耦合问题的跨层综合分析（§11），以及开放问题议程（§12）来发展这一论题。

2. **论断 2（分类性）**：七层 ETCLOVG 分类法将可观测性（Observability）和治理（Governance）作为一等层处理，而非生命周期钩子的副作用。每一层都有自己的生产工具栈（可观测性侧有 Langfuse 和 OpenTelemetry；治理侧有权限引擎、网关和审计流水线），并在生产部署中由不同团队负责。我们还将状态管理置于生命周期与编排（Lifecycle and Orchestration）内部，使状态紧邻读写它的执行流（§2.3）。

3. **论断 3（实证性）**：将 170 多个开源项目映射到 ETCLOVG 上，揭示了生态系统中哪里密集、哪里稀疏，以及早期语料库遗漏了哪些类别。该映射是迄今为止最大的开源 Agent Harness 语料库。执行、工具、生命周期和验证被密集覆盖；可观测性和治理较为稀疏，更多存在于商业平台中；早期语料库中缺失的三个类别——任务运行器、多 Agent 编排器和规格驱动开发工具——现在是一等公民。方法论小节（§2.4–§2.9，附录 A）使编码可复现，该映射支持 §3–§9 中的逐层观察。

## 2 背景与分类法

### 2.1 Agent 系统的演进

从早期思维链提示到自主 Agent 的发展轨迹，可以理解为实践者必须管理的工程表面的渐进式扩展。

**ReAct 时代（2022–2023）。** Yao et al. (2023) 将观察-思考-行动循环确立为基础性原语。早期系统以最小化的基础设施运行：一个 while 循环、一个提示模板和一个小型工具分发表。AutoGPT 和 BabyAGI 通过用任务队列、记忆和工具分发包裹语言模型调用，展示了完全自主运行的雄心，同时也使失败模式（如执行失控、上下文爆炸、状态丢失和未监控的副作用）作为基础设施问题而非仅仅是 Prompt 问题变得可见 (Significant Gravitas, 2023; Nakajima, 2023)。

**工具集成与多 Agent 协调（2023–2024）。** Gorilla、ToolLLM 和 Toolformer 确立了工具使用能力可以被学习或诱导，而非硬编码到固定的 API 包装器中 (Patil et al., 2024b; Qin et al., 2024; Schick et al., 2023)。CAMEL、ChatDev、MetaGPT 和 Mixture-of-Agents 引入了多 Agent 协调模式，从角色扮演对话到软件开发组织再到分层 Agent 聚合 (Li et al., 2023a; Qian et al., 2023a; Hong et al., 2023; Wang et al., 2024)。评估基础设施随 SWE-bench、AgentBench、WebArena 和 GAIA 的出现而成熟 (Jimenez et al., 2024; Liu et al., 2023a; Zhou et al., 2024; Mialon et al., 2023)，协议标准化则随 Anthropic 的 MCP 和 Google 的 A2A 展开 (Anthropic, 2024c; Surapaneni et al., 2025)。

**Harness 转向（2025–2026）。** 到 2025 年，已经积累了足够的部署经验来明确表明，Agent 可靠性的约束瓶颈是基础设施质量而非模型质量。2026 年初的三个独立发展验证了这一转变：OpenAI 明确采纳"Harness 工程"作为一门学科、Stanford/MIT 的 Meta-Harness 表明自动化 Harness 优化超越了手工工程、以及 LangChain 的 DeepAgents 仅通过 Harness 层变更便在 Terminal-Bench 2.0 上从 52.8% 提升到 66.5%，对应 13.7 个百分点的增益和约 26% 的相对改进 (OpenAI, 2026a; Lee et al., 2026; Trivedy, 2026)。

### 2.2 三个工程阶段

2022–2026 年期间揭示了该领域选择工程化对象的一个连贯的三阶段演进。

**Prompt 工程（2022–2024）。** 主要杠杆是输入提示文本。实践者通过精心设计更好的指令、少样本示例和推理模板来进行优化。工程范围较窄：优化单次模型调用的单个文本输入。

**Context 工程（2025）。** 随着 Agent 变得更长期运行，约束瓶颈从"输入是什么？"转移到"模型在每一步应该看到什么信息？"这一阶段聚焦于上下文管理：每轮注入什么内容、如何检索和压缩记忆、如何按相关性对工具结果排序，以及如何处理上下文窗口饱和。范围从单个输入扩展到管理流入上下文窗口的多个信息流 (Anthropic Applied AI Team, 2025)。

**Harness 工程（2026）。** 随着模型变得足够强大以处理长期运行的任务，可靠性越来越依赖于维护状态、中介工具、注入反馈、强制约束和验证进度的基础设施包装器。这一观察与约束瓶颈观点一致，即长时程 Agent 性能由耦合的模型-Harness 系统产生，而非仅由模型产生 (Bölük, 2026b)。因此，Harness 工程追问：必须围绕模型设计什么治理、约束、反馈循环和执行控制才能使 Agent 系统可靠。在我们的分类法中，这一阶段将所有七个 ETCLOVG 层作为一个整体来处理 (OpenAI, 2026a; Böckeler, 2026; LangChain, 2026b)。

每个阶段包含前一个阶段：Harness 工程包含 Context 工程，而 Context 工程包含 Prompt 工程。三个阶段在时间和概念上也相互重叠，而非以清晰的边界依次替代。Prompt 工程至今仍是 Harness 实践的活跃组成部分，Context 工程也在与我们此处综述的 Harness 层面关注点并行地持续成熟。因此，我们的时期划分最好理解为边际工程努力投入方向的转移，而非一系列替代。

### 2.3 ETCLOVG 七层分类法

我们提出了一个用于 Agent Harness 工程的七层分类法。我们以首字母缩略词 ETCLOVG 来指代它，代表执行（Execution）、工具（Tooling）、上下文（Context）、生命周期（Lifecycle）、可观测性（Observability）、验证（Verification）、治理（Governance）。图 4 给出了紧凑的视觉地图；本小节固定了整个综述中使用的解释。

前四层描述了 Harness 的结构核心。执行（E）决定 Agent 代码在何处运行以及什么沙箱约束限制它；工具（T）指定外部能力如何被描述、发现和调用；上下文（C）控制模型在短期、会话级和持久化时程中能看到什么；生命周期（L）组织读写该状态的控制流，从单 Agent 循环到多 Agent 和 Issue 到 Pull Request 的工作流。其余三层描述围绕该核心的控制平面。可观测性（O）捕获轨迹、成本、故障和可靠性信号；验证（V）将任务和轨迹转化为评估、故障归因和回归反馈；治理（G）通过权限、身份、策略、加固、审计和人工监督机制约束行为。第 3–9 节按顺序展开这七层，而第 10–12 节综合不属于任何单一层的跨层权衡和开放问题。

两个设计选择使本分类法与众不同。第一，我们将可观测性（O）提升为独立层，而非将其视为生命周期钩子的副作用。在生产系统中，可观测性拥有专用的工具生态系统（Langfuse、Arize Phoenix、OpenLLMetry）和独特的工程实践（OpenTelemetry 插桩、成本归因、异常检测），值得独立处理。第二，我们引入治理（G）作为一等层，捕获跨三个子层的安全和合规关注点的完整光谱：模型级（护栏、内容过滤器）、系统级（网关、代理、权限模型）和组织级（审计、合规、人在回路中的监督）。

## 2.6 纳入与排除标准

当一个项目满足以下三个条件时即被纳入：它有公开文档记录、它实现或规定了一种具体的 harness 层级机制、且可获取的证据足以为其分配至少一个 ETCLOVG 层。这包括具备可复用编排或工具路由逻辑的智能体框架、实例化了可执行智能体环境的基准、为智能体执行而封装的 sandbox，以及作用于智能体状态、追踪、动作或策略之上的记忆、可观测性、评估或治理系统。我们排除了简单的聊天机器人演示、提示词包、轻量模型客户端包装器、没有智能体运行时的静态数据集或排行榜、非面向智能体的通用基础设施组件，以及那些无法从公开文档检查其技术行为的产品页面。边界案例通过机制而非标签来裁定：一个被命名为"agent"的仓库不足以被纳入，而一个评估或 sandbox 项目则会在它提供了可复用的 harness 机制时被纳入。

## 2.7 编码协议

每个保留的项目都使用公开工件本身作为证据，依据 ETCLOVG 七个层级进行编码：README 文件、文档页面、论文、示例、发布说明，以及在必要时的仓库结构。由于许多系统横跨多个层级，编码采用了多标签方式；主层级标记了对该工件最为核心的机制，而次层级仅在文档暴露出一种独立能力（而非附带依赖）时才予以分配。当前快照采用的是单一主编码者加作者审计的协议，而非正式的多编码者一致性研究，因此我们不报告 Cohen's kappa 或类似的标注者间统计量。在全部集合编码完成后，我们使用一条保守规则重新审视了模糊案例：如果公开证据未能清晰显示一种面向智能体的机制，则保留（不作出）该层级的分配。

## 2.8 语料库的局限性

该语料库应被解读为可见的 agent-harness 生态系统的一张地图，而非所有已部署智能体基础设施的一次普查。它偏向于英语来源、GitHub 可见的项目、开源工件，以及那些维护者发布了足够实现细节以供外部编码的系统。商业生产系统的代表性不足，除非其工程博客、文档或 SDK 暴露了相关机制；而编码智能体基础设施则代表性过高，因为它拥有异常丰富的公开痕迹：仓库、基准、sandbox、issue 到 pull request 的工作流以及发布说明。层级分配同样反映的是公开文档而非私有架构，因此某一层级的缺失意味着"未有公开证据"而非"未实现"。

## 2.9 汇总分析

170 余个项目的映射揭示出一个广泛但不均衡的生态系统。执行、工具接口、生命周期编排和验证拥有最密集的可见覆盖，因为编码、网页、终端和 computer-use 智能体在变得有用之前都需要可运行的环境、工具契约、控制循环和可重复的评估。上下文与记忆出现在许多项目中，但往往嵌入在更大的框架内部，而非作为独立的 harness 组件发布。可观测性和治理在开源覆盖中较为单薄，更多地通过商业平台、SDK 特性或工程文章出现，这表明运营控制的成熟度落后于运行时和基准基础设施。跨层级项目正变得日益普遍：最完整的系统组合了 sandbox、工具协议、编排、追踪、评估和权限控制，这支持了一个核心论点，即 harness 工程是一个集成的系统性问题，而非一组孤立附加组件的集合。

## 3.1 范围与概念

### 3.1.1 定义

智能体的执行环境指的是智能体动作被物理执行所在的基础设施层。在 LLM 智能体的语境中，执行环境与 sandbox 是紧密耦合的概念。因此，生产级智能体系统几乎无一例外地在 sandbox 化的环境中执行动作。

### 3.1.2 为何 Sandbox 化在智能体时代处于核心地位

智能体时代的 sandbox 化不仅仅是从传统多租户代码执行中继承而来的一项安全措施。它同时服务于三个不同的目的，而正是这三者的结合将 sandbox 化从一个运营细节提升为智能体 harness 设计中的一等关注点。

第一个目的是安全。智能体 sandbox 化面临着超越传统多租户代码执行的挑战。LLM 生成的代码在规模化下既不可审计也不可预测，这排除了将静态审查作为主要防御手段的可能。智能体在多个步骤上自主执行，因此在动作执行时刻无法获得人工干预。提示注入（prompt injection）攻击可以将一个原本无害的智能体重新用作针对 sandbox 的定向攻击载体，从而模糊了可信用户意图与恶意输入之间的界限。近期关于 sandbox 逃逸的实证研究表明这些担忧并非假设，我们将定量证据推迟至 §3.3。

第二个目的是可复现性。长时程智能体任务以及衡量它们的评估 harness（§8）需要将执行状态重置到已知基线的能力。一个 Docker 容器或一个 microVM 可以按需销毁并重建，而开发者的工作站则不能，正是这一特性使得基于 sandbox 的评估标准（如 SWE-bench (Jimenez et al., 2024) 和 OSWorld (Xie et al., 2024)）变得可行。在训练时，当单个任务可能在并行轨迹中被重放数百次时，缺乏廉价的重置机制本身就是一个可扩展性瓶颈。

第三个目的是活性（liveness），它是最具智能体时代特性的一个。如果没有 sandbox，智能体希望执行的每一个潜在风险动作——例如文件写入、软件包安装或对外网络调用——都必须经由向人类发出的显式权限提示来把关。在规模化下，这会产生两种失败模式：用户因沮丧而放弃智能体，或者用户条件反射式地批准一切，从而瓦解了这些提示的安全初衷。sandbox 通过定义一个有界区域来打破这一僵局，在该区域内智能体被授权自由行动，从而将权限从一个动作层面的问题转变为一个会话配置层面的问题。Anthropic 报告称，为 Claude Code 引入 sandbox 化在保持安全性的同时将权限提示减少了 84% (Anthropic, 2025b)。在这一框架下，sandbox 同时是一个牢笼和一张许可证，而正是其许可证这一方面首先使得自主的长时程执行成为可能。

在这三个目的中，安全是与传统 sandbox 化共享的，可复现性在智能体场景中被放大，而活性则基本上是全新的。它们的结合使得将智能体 sandbox 化视为一个独立的研究对象（而非容器技术的下游应用）变得合理。

## 3.2 智能体 Sandbox 的类别

智能体 sandbox 基础设施在 2024 至 2026 年间从一小批通用型运行时多样化为若干个不同的产品类别，每个类别针对不同的智能体任务类型进行优化。我们沿着工作负载与用例的维度将这一图景组织为七个类别，我们发现这对正在系统之间进行选择的 harness 设计者最具信息价值。隔离技术这一正交维度——包括容器、诸如 gVisor (Google, 2018) 的用户空间内核、诸如 Firecracker (Agache et al., 2020) 和 Kata Containers (Kata Containers Project, 2017) 的 microVM、WebAssembly，以及诸如 bubblewrap 和 Seatbelt 的操作系统级原语——被作为各小节内的一个设计属性来讨论，而非作为一个顶级类别，因为同一种隔离原语会在不同的工作负载中被复用。这七个类别是：通用型托管 sandbox（§3.2.1）、computer-use 智能体基础设施（§3.2.2）、代码专用 sandbox（§3.2.3）、框架集成运行时（§3.2.4）、浏览器评估环境（§3.2.5）、操作系统级权限 sandbox（§3.2.6）和 sandbox 抽象层（§3.2.7）。以下各小节分别介绍每个类别。

### 3.2.1 通用型托管 Sandbox

通用型托管 sandbox 提供商业或开源的 sandbox 即服务平台，通过 API 接口暴露任意的 OCI 容器镜像，并为未指定的工作负载提供 shell、文件系统、网络和解释器支持。代表性系统包括：Daytona (Daytona Platforms, Inc., 2024)，它从开发者环境转型为智能体 sandbox，具备低于 90ms 的冷启动配置能力；E2B (E2B, 2024)，一个建立在 Firecracker microVM (Agache et al., 2020) 之上的智能体 sandbox；Modal (Modal Labs, 2024)，一个使用 gVisor (Google, 2018) 并具备大规模自动扩缩容的 Python 平台；Northflank (Northflank, 2024)，一个同时支持 Kata Containers (Kata Containers Project, 2017)、Firecracker 和 gVisor 的平台；来自阿里巴巴（Alibaba）的 OpenSandbox (Alibaba, 2026)，一个开源的通用型 sandbox；以及 Docker Sandboxes (Docker, Inc., 2025)，Docker 于 2025 年发布的官方基于 microVM 的产品。

该类别中的设计决策收敛于若干模式：默认临时性（ephemeral-by-default）的语义并附带可选的持久会话、以 Python 和 TypeScript SDK 形式暴露的 API 接口，以及对任意 OCI 镜像的支持。各系统在隔离强度和运营模型上有所分歧。Daytona 默认采用容器隔离并可选 Kata Containers，而 E2B、Modal 和 Docker Sandboxes 默认提供 microVM 或 gVisor 隔离。Northflank 独特地允许为每个工作负载在多个后端之间进行选择，这反映出业界认识到没有任何单一的隔离原语能适配所有威胁模型。我们观察到一个更广泛的趋势，即从内核共享的容器隔离转向专用内核的 microVM 隔离，这一趋势的驱动力在于人们观察到 LLM 生成的代码具有无法事先刻画的不可预测系统调用模式。

### 3.2.2 Computer-Use 智能体基础设施

computer-use 智能体基础设施代表了一种独特的执行模型，其中智能体通过模拟的鼠标、键盘和屏幕观察与图形界面交互，而非通过 API 或 shell 命令。代表性系统包括：Anthropic 的 Computer Use (Anthropic, 2024b)，一个使 Claude 能够直接操作桌面环境的旗舰商业实现；CUA (Cua, 2024)，一个开源的 computer-use 智能体基础设施；以及由 OSWorld (Xie et al., 2024) 提供的基于 VM 的环境，它既充当评估 harness（§8），又充当一个参考性的 computer-use sandbox。

这些系统在 sandbox 内封装了一个完整或近乎完整的桌面环境（通常通过带窗口管理器的 Xvfb 或一个完整的 VM 实现），并向智能体暴露像素坐标和按键动作。其动作空间比基于 API 的类别大得多，但可靠性取决于视觉接地（visual grounding），且完整的操作系统攻击面要求更强的隔离，通常是 microVM 或完整 VM。与 §3.2.1 的托管 sandbox 相比，computer-use sandbox 以密度和启动延迟换取保真度：一个完整的桌面环境比一个无头 shell sandbox 启动更重、更难复用，但它是唯一一种智能体能够操作那些不暴露任何 API 的应用程序的执行模型。

### 3.2.3 代码专用 Sandbox

代码专用 sandbox 是轻量级的环境，针对代码生成、评估和数据分析而非通用 shell 访问进行了优化。代表性系统包括：Judge0 (Došilović, 2024)，一个最初为评测设计、并被广泛复用为编码智能体评估流水线组件的代码评估 sandbox；OpenAI Code Interpreter (OpenAI, 2023)，支撑 ChatGPT 高级数据分析功能的生产规模 sandbox 化 Python 环境；sandboxed.sh (Th0rgal, 2026)，一个面向编码智能体的 shell sandbox；以及 langchain-sandbox (LangChain, 2025c)，它使用编译为 WebAssembly 并在 Deno 运行时下执行的 Pyodide (Pyodide Contributors, 2018)，在本地无需容器即可对智能体生成的 Python 进行 sandbox 化，这一设计也得到 NVIDIA 针对客户端智能体工作流的倡导 (NVIDIA, 2024)。

与 §3.2.1 的通用型 sandbox 不同，这些系统预装编译器和解释器，默认采用无状态的请求级执行，并针对高并发并行性进行优化。这一设计选择以工作负载的通用性换取启动速度、评估吞吐量和更简单的威胁模型。一个值得注意的子趋势是从基于容器的代码 sandbox 转向基于 WebAssembly 的代码 sandbox：WebAssembly 提供基于能力的安全性、确定性执行和微秒级的实例化，代价是受限的 Python 标准库和减少的原生扩展支持。

### 3.2.4 框架集成运行时

框架集成运行时是捆绑在更广泛的智能体框架内部的执行环境，而非作为独立的 sandbox 产品暴露出来。它们与框架的编排循环、工具注册表和提示词约定一起发布，且在不采用周边框架的情况下无法被独立使用。代表性系统包括：OpenHands 运行时 (Wang et al., 2025b)，一个将 bash、IPython、Chromium 浏览器和一个 API 服务器集成到单个镜像中的 Docker sandbox 化环境；agent infra sandbox (Agent Infra, 2024)，一个明确的一体化设计，将浏览器、shell、文件系统、MCP 和 VSCode 封装到一个环境中；以及 smolagents (Roucher et al., 2025) 的执行器层，它提供本地、Docker、E2B、Modal 和 WebAssembly 执行的框架内实现。

该类别中决定性的权衡是在捆绑（bundle）与组合（compose）之间：框架集成运行时优先考虑开箱即用的能力覆盖，代价是更大的镜像体积、更慢的启动以及与某个框架抽象的紧耦合。相比之下，§3.2.1 的通用型 sandbox 采取了相反的方法，即通过外部抽象组合而成的最小化环境。组合方法长远来看是否会取代捆绑式运行时，取决于诸如 MCP 之类的标准是否能够降低在运行时（而非构建时）组装能力所带来的代价。

### 3.2.5 浏览器评估环境

浏览器评估环境兼具 sandbox 和评估 harness 的双重角色。代表性系统包括：WebArena (Zhou et al., 2024)，一个由真实网页应用组成的自托管集群，配以基于 Playwright 的交互；VisualWebArena (Koh et al., 2024)，它用多模态视觉接地任务扩展了 WebArena；以及 BrowserGym (Chezelles et al., 2024)，它与 WorkArena (Drouin et al., 2024) 一起为基于浏览器的智能体执行和基准测试提供了标准化的 gym 风格接口。

这些系统提供隔离的网页执行环境（sandbox 角色），同时供给带自动化评估的可复现任务定义（harness 角色）。这种双重功能将它们与 §3.2.2 的 computer-use 类别区分开来——后者在桌面而非浏览器层面运作——并且它在此处讨论的执行环境与 §8 讨论的评估基础设施之间创造了一个直接接口。浏览器环境还暴露出一个独特的威胁面：由于浏览器会摄入不可信的网页内容，它们是研究针对智能体的间接提示注入和多模态红队攻击的天然基底 (Debenedetti et al., 2024; Greshake et al., 2023; Zhang et al., 2026a; Wei et al., 2026)。

### 3.2.6 操作系统级权限 Sandbox

操作系统级权限 sandbox 使用操作系统原语（Linux 上的 bubblewrap、macOS 上的 Seatbelt，或用于系统调用过滤的 seccomp-bpf）而非容器、VM 或用户空间内核来实现细粒度的文件系统和网络隔离。它们比容器 sandbox 轻量得多，同时提供目录和域的访问控制。代表性系统包括：Anthropic 的 sandbox-runtime (Anthropic, 2025g)，一个开源 npm 包，它通过 bubblewrap 和本地 HTTP/SOCKS5 代理强制执行文件系统和网络白名单，从而包装任意命令；内置于 Claude Code 自身的 sandbox 功能 (Anthropic, 2025b)，它消费该 runtime 以将 bash 工具的文件系统和网络访问限制在配置的边界内；以及 IsolateGPT (Wu et al., 2025)，一个应用 seccomp 和 setrlimit 来跨工具调用 LLM 应用强制执行隔离的研究系统。

这一类别的设计哲学是「权限而非分区」：目标不是为每个会话提供全新的操作系统镜像，而是给予 agent 对现有主机的一个收窄视图，使得被 prompt 注入或幻觉产生的命令无法修改敏感文件或泄露数据。Anthropic 报告称，这类边界在保持安全性的同时将 Claude Code 中的权限提示减少了 84% (Anthropic, 2025b)，这说明了操作系统级 sandbox 的生产力动机：权限提示本身就是长程 agent 执行中的一种 liveness 失败。由于主机内核是共享的，操作系统级权限 sandbox 相比基于 microVM 的托管 sandbox（§3.2.1）针对对抗性代码提供的隔离更弱；当威胁模型是 prompt 注入而非完全对抗性代码时，它们是合适的选择。

### 3.2.7 Sandbox 抽象层

Sandbox 抽象层本身不是 sandbox，而是将多个 sandbox 后端统一到单一 API 之后的接口，允许 harness 在不重写 agent 代码的情况下切换执行基底。代表性系统包括：SWE-ReX (SWE-agent Team, 2024)，由 SWE-agent 团队 (Yang et al., 2024) 开发的 runtime 接口，将 Docker、AWS Fargate、Modal 和 Daytona 统一为可互换的后端；smolagents 的 executor 接口 (Roucher et al., 2025)，将 local、e2b、modal、docker、blaxel 和 wasm 暴露为单一参数化的 executor_type；以及 Kubernetes SIG Apps 下的 Agent Sandbox (Kubernetes SIG Apps, 2025) 项目，它引入了一个 Sandbox CRD 和 controller，通过声明式 API 将 gVisor 和 Kata Containers 暴露为可插拔的隔离后端。

这一类别的出现反映了一种日益成熟的认识，即执行基础设施应当是可替换的。一个硬编码特定 sandbox API 的 harness 会将其编排逻辑耦合到某一厂商的临时性产品上；而抽象层将 agent 运行什么与在哪里运行解耦。我们观察到研究项目（SWE-ReX）、框架内嵌接口（smolagents executors）和新兴基础设施标准（Kubernetes agent-sandbox CRD）之间的趋同，这表明 sandbox 抽象正在成为 agent harness 栈中一个独立的层，而非个别系统的某项特性。

**综合分析。** 在七个类别中，可以看到三条横切性趋势。第一，该领域正沿着隔离强度分化而非趋同：托管 sandbox（§3.2.1）正从共享内核容器迁移向专用内核 microVM，而操作系统级权限 sandbox（§3.2.6）则完全摒弃了独立环境，转而收窄主机视图。普通 Docker 容器所处的中间地带正被两端挤压，而两个极端之间的选择反映的是威胁模型（完全对抗性代码 vs. 被 prompt 注入但其余部分协作的代码），而非某种通用的技术偏好。第二，评估鲁棒性正成为一等关切：诸如 SandboxEscapeBench (Marchand et al., 2026) 这样的容器逃逸基准，以及诸如 CIBER (Ba et al., 2026) 这样针对 code-interpreter agent 的安全基准都表明，当前生产中部署的 sandbox 配置在一定程度上可被前沿模型绕过。第三，该领域正在将基础设施级隔离（本节所涵盖的内容）与工具调用层的语义级或能力级隔离区分开来。CaMeL 的控制流完整性设计 (Debenedetti et al., 2025) 和 Progent 的可编程权限控制 (Shi et al., 2025a) 是后者的范例，它们属于工具使用章节（§4）而非 sandbox 基础设施本身。这两条线是互补的：基础设施 sandbox 约束某个动作一旦被执行后的爆炸半径，而语义隔离则约束哪些动作在一开始就被允许。一个完整的 agent harness 两者都需要。

## 3.3 威胁模型与 Sandbox 逃逸

agent 执行的 sandbox 化既面临传统的容器级威胁（容器逃逸、侧信道、资源耗尽），也面临放大经典关切的 agent 特有威胁类别。第一，prompt 注入攻击允许外部输入（例如检索到的网页、工具响应或文件内容）劫持 agent 行为并发起恶意的 sandbox 操作。第二，目标错位可能导致 agent 自身将主动尝试 sandbox 逃逸作为一个工具性子目标。第三，组合放大发生在具有多工具访问权限的 agent 使单个 sandbox 弱点跨集成能力级联扩散时。

关于 agent 场景下 sandbox 逃逸的实证证据有限但令人担忧。SandboxEscapeBench (Marchand et al., 2026) 在嵌套 sandbox 夺旗（capture-the-flag）设置中评估前沿 LLM，报告针对基于 Docker 的容器有 15% 到 35% 的逃逸成功率，具体取决于容器配置。该基准涵盖了一系列逃逸机制，包括配置错误、权限分配失误、内核缺陷以及 runtime 或编排弱点，其结果确立了这一威胁即便在当前模型能力下也是已实现的而非仅仅是理论上的。防御研究仍处于早期阶段。IsolateGPT (Wu et al., 2025) 为基于 LLM 的 agentic 系统提出了一种执行隔离架构，报告在防止跨应用数据泄露的同时，对四分之三的受测查询性能开销低于 30%。事务性 sandbox 化方法 (Yan, 2025) 提供基于回滚的保护，报告约 14.5% 的开销，并对高风险命令具有较高的拦截率。LLM-in-Sandbox (Cheng et al., 2026) 提供了一个补充视角，它主张采用最小化而非能力最大化的 sandbox 环境既可减少攻击面，也可降低不必要的 agent 复杂度。

综合来看，这些结果暴露了进攻性与防御性进展之间的差距。进攻性评估已产出了一个具体且可复现的基准，而防御性工作仍分散于各个孤立的原型之中，它们在威胁模型、评估协议和基线假设方面各不相同。一个在统一评估方法论下系统性地应对 prompt 注入、目标错位和组合放大的 agent 原生 runtime 安全框架，是一个开放的研究方向，我们将在 §12.1 中重新讨论。

## 3.4 部署模式

agent sandbox 基础设施已在超出最初自托管 Docker 模式之外的多种部署模式上实现多样化。当前实践中并存三种模式。在自托管模式中，开发者直接管理 sandbox 基础设施，这是 OpenHands 和 SWE-agent 的默认模式。在云（SaaS）模式中，sandbox-as-a-service 提供商负责处理基础设施，E2B、Modal 和 Daytona Cloud 即为例证。在混合或自带云（BYOC）模式中，agent 逻辑与 sandbox 执行跨环境解耦；例子包括 OpenHands SDK 的 Local 和 Remote Workspace 抽象 (Wang et al., 2025c) 以及 E2B 和 Northflank 的 BYOC 产品。

这些模式的演进由两种互补的力量驱动。一方面，从业者报告将部署选择沿延迟、安全性和可扩展性三个维度进行刻画 (Anthropic, 2025b;f)。自托管 sandbox 提供最低的延迟和最紧凑的迭代循环，但需承担全部运维负担；云 sandbox 则颠倒了这一权衡，以网络往返为代价提供弹性扩展和托管的安全性；混合模式试图在委托执行容量的同时将敏感数据保留在本地。另一方面，数据驻留、合规和可审计性等组织约束，即便在延迟和可扩展性更有利于单一模式方案的情况下，也会推动部署走向混合架构。

在观察到的实践中，自托管 sandbox 在交互式开发和单租户场景中占主导，而云 sandbox 在多租户和大规模部署中更为常见。混合模式正专门为合规或数据本地性要求与对大量临时执行容量的需求并存的场景而出现。在现实 agent 工作负载下对这些模式进行系统性实证比较仍然缺失，我们将其作为 §12.1 中更广泛 runtime 议程的一部分。

## 3.5 小结

执行环境是 agent harness 的物理基底：它们提供安全边界、用于可复现评估和训练的重置机制，以及长程 agent 可以无需逐条人工批准而行动的有界区域。本节调研的七个类别表明，设计空间如今与其说由单一隔离原语塑造，不如说由工作负载保真度、威胁模型和运维模式塑造。托管 sandbox 和 computer-use 环境强调强隔离和真实执行；代码专用和浏览器环境强调吞吐量和评估结构；框架集成 runtime 为便利性而捆绑能力；操作系统级权限 sandbox 收窄本地权限；而抽象层使基底可替换。

这一设计空间在能力、控制和成本之间制造了反复出现的张力。高风险工作负载倾向于 microVM 和托管云，交互式本地工作流倾向于轻量级权限边界，而大规模训练或评估则倾向于可快速重置的基底。因此，关于 runtime 防御、经济性、可移植性、捆绑 vs. 组合的设计，以及跨层耦合的悬而未决问题，并非孤立的执行层脚注；它们将作为贯穿全调研的开放问题在 §12.1 中再次出现。

工具接口与协议层定义了 agent 如何发现能力、表示可调用的可供性（affordance），以及如何跨异构 runtime 边界执行动作。在实践中，这一层处于两个相互竞争的目标之间的断层线上：通过暴露更多工具来提升能力覆盖，与通过保持动作空间和 prompt 占用较小来维护决策质量。来自生产 agent 系统的近期工程指导反复报告，过大的工具菜单会降低可靠性、增加 token 开销并放大规划错误 (Anthropic, 2025d; OpenAI, 2026c)。

我们将这一层组织为四个互补的方向：协议与接口标准；工具描述、发现与选择；工具增强的模型训练与集成；以及可扩展性与会话管理。

## 4.1 协议与接口标准

MCP 已成为编码和企业 agent 最显眼的工具集成基底，具有明确的 host-client-server 架构以及基于 JSON-RPC 的工具、资源和 prompt 的类型化交换 (Model Context Protocol, 2025b;c;a)。MCP 的实际价值不仅在于 schema 层面的互操作性，还在于生态流动性：agent 构建者可以复用一个不断扩展的 server 目录，而不必为每次部署实现自定义连接器。

A2A 瞄准的是一个不同但相邻的边界。它不是向一个 agent 进程暴露工具，而是标准化不透明 agentic 应用之间的通信，包括通过 Agent Cards 进行发现、支持同步和流式交互，以及长程任务协作 (A2A Project, 2025)。近期的协议综述将 MCP 和 A2A 置于互补角色：MCP 主要用于工具/上下文访问，A2A 用于 agent 间委托与协作 (Ehtesham et al., 2025)。在我们看来，一个更有用的组织原则是按工具/接口标准所跨越的集成边界对其进行分类，而非按厂商谱系或发布时间线。在此视角下，浮现出四个边界：Model↔Function（结构化调用）、Agent↔External capability（runtime 到工具的解耦）、Agent↔Agent（跨进程委托）以及 Agent↔Repo/environment（版本受控的策略）。表 1 总结了这一视角，并明确了为何若干被广泛比较的标准（例如 MCP vs. A2A，或 Function calling vs. OpenAPI）在单一 harness 中占据互不重叠的角色。

Function-calling schema 和 API 描述标准仍是这一层的基础构建块。OpenAI 风格的 function calling 通过 JSON schema 和显式的调用/返回轮次将工具调用操作化 (OpenAI, 2026c)；OpenAPI 提供了一种语言无关、机器可读的 API 契约，许多 agent 框架将其用作工具生成和验证的来源 (OpenAPI Initiative, 2025)。此外，诸如 AGENTS.md 和 AGENT.md 这样的仓库级指令文件提供了一种轻量级的替代方案，可直接在版本控制中编码工具使用和工作流约束，从而减少代码 agent 的配置摩擦 (agentsmd, 2025; agentmd, 2025)。

## 4.2 工具描述、发现与选择

一旦协议定义了如何进行调用，下一个瓶颈就是在每一步应当呈现和选择哪些工具。一条日益增长的工作线研究工具文档质量、检索和动态候选剪枝。EASYTOOL 分析了从庞大库存中选择合适工具的挑战 (Yuan et al., 2025)。AnyTool 和 CRAFT 专注于通过自动构建或精炼工具使用流水线来减少手动规范负担 (Du et al., 2024; Yuan et al., 2023)。MetaTool 这类基准式评估表明，工具检索和调用质量可能在不同领域和查询形式之间产生显著分化 (Huang et al., 2023)。诸如 MCP-Zero、ToolRet 和 ToolRegistry 等更近期的工作强调检索感知的编排和注册表质量是下游 agent 成功的一阶决定因素 (Fei et al., 2025; Shi et al., 2025b; Ding, 2025)。一个密切相关的方向将工具选择扩展到可复用技能（skill），其中 agent 必须识别相关的程序性模块，而不仅仅是紧凑的 API schema。SkillRouter (Zheng et al., 2026) 和 SkillRet (Cho et al., 2026) 都在规模上研究了这一技能选择问题，凸显了从庞大且重叠的技能库中检索出正确技能的需求。

在系统层面，这些结果强化了两条设计原则。第一，「更少但更好的工具」往往优于暴力工具暴露，因为它同时缩小了 prompt 熵和规划器的分支。第二，发现流水线必须是自适应的：静态、全局的工具列表无法扩展到快速演进的仓库或多租户企业部署。

## 4.3 工具增强的训练与集成

第三个方向从 runtime 编排转向模型能力获取。Toolformer 展示了自监督增强，用于学习在生成过程中何时以及如何插入 API 调用 (Schick et al., 2023)。Gorilla 和 ToolLLM/ToolBench 以更大的工具语料库、指令微调流水线以及以执行为中心的 API 使用监督扩展了这一线 (Patil et al., 2024b; Qin et al., 2024)。ToolkenGPT 和 CREATOR 探索 token 级或 controller 式的集成，以提升调用格式保真度和规划稳定性 (Hao et al., 2023; Qian et al., 2023b)。

在生产 harness 中，这些模型侧进展通常与框架级 runtime 栈配对，例如 LangChain、Semantic Kernel 和 smolagents，它们提供记忆抽象、路由中间件和工具适配器 (LangChain, 2026c; Microsoft, 2026; HuggingFace, 2026)。编码 agent 场景还暴露出第二类更具语义性的工具：静态分析器、类型检查器、求解器支持的验证器、证明助手，以及补丁等价性或故障定位检查器。Ugare & Chandra (2026) 将这一空间框定为 agentic 代码推理：agent 探索一个仓库并推理代码行为，而不必执行代码。他们的半形式化推理方法介于非结构化的思维链与完全形式化的验证之间，通过强制 agent 陈述前提、追踪程序路径并推导出明确结论，改进了补丁等价性验证、故障定位和代码问答。对工具层而言，其含义是自动化推理工具应当返回承载证据的产物，例如 trace、证明义务（proof obligation）、反例或结构化证书，而不仅仅是黑盒式的是/否答案。综合证据表明，工具胜任能力由预训练和微调信号、接口 schema 质量以及 runtime 选择策略共同决定；只改进其中一个组件收益有限。

## 4.4 可扩展性与会话管理

一个反复出现的运维挑战是长程下的会话管理。有状态的工具会话改善了连续性，但增加了记账复杂度，尤其是在调用被并行化或跨多个 agent 委托时。失败模式包括陈旧句柄、跨重试的工具状态不一致，以及来自冗长工具 trace 的上下文窗口饱和。因此，有效的 harness 设计需要针对工具会话的显式生命周期控制、有界的工具上下文注入，以及能够将错误归因于规划器逻辑或接口/协议失败的可观测性钩子。

最后，生态级的全景梳理可以帮助从业者快速绘制可用的协议和工具选项，但对于在给定部署中决定应采用哪种协议和工具路由策略而言，以基准为依据的比较仍然必不可少。

该层管理模型在每个执行步骤中所看到的信息，以及知识如何跨轮次和会话持久化。核心工程问题可以简洁地表述为：在每个步骤中恰好为模型提供正确的信息，不多不少。上下文过少，agent 将缺乏正确行动所需的状态。上下文过多，性能将以可测量的、跨模型一致的、且尚未完全理解的方式退化。

我们围绕时间跨度组织本节。第 5.1 节阐述为何上下文需要主动管理而非被动累积。第 5.2 节将 context engineering 定义为区别于 prompt engineering 的独立学科。第 5.3 至 5.5 节涵盖在生产实践中涌现的三个记忆层级：活跃 context window（短期）、session 级持久化（中期）和跨 session 存储（长期）。第 5.6 节讨论 agent 运行数百轮次时产生的协调问题。第 5.7 节以尚未解决的问题作结。

## 5.1 为何上下文必须经过工程化设计

更大的 context window 并不能解决记忆问题。理解其原因需要审视底层架构。

**二次方注意力开销。** Transformer 的自注意力机制计算上下文中每个 token 之间的成对关系 (Vaswani et al., 2017)。对于 n 个 token，这会产生 n² 个成对权重；计算和内存随上下文长度呈二次方增长。FlashAttention 和位置编码插值等工程技术降低了常数因子，但二次方结构是架构性的。将上下文加倍不会使成本翻倍——而是使其翻四倍。这使得 context window 成为一种稀缺资源。

**U 形注意力曲线。** 仅凭架构成本并不能解释为何模型在长上下文中失败，即使计算资源充足。Liu et al. (2024) 提供了关键的实证结果：在包含 20 个输入文档的多文档问答任务中，当相关文档位于上下文中间位置时，准确率相比置于开头或末尾下降了超过 30%。这种 U 形性能曲线在不同模型、任务和上下文长度下均成立，包括专门针对长上下文训练的模型。其实际含义是直接的：信息的放置位置与信息的存在同等重要。一个检索到正确内容但放置不当的 agent 几乎无法从检索中获益。

**规模化下的 Context Rot。** Hong et al. (2025) 在旨在将输入长度与任务难度隔离的受控条件下，评估了 18 个前沿模型，包括 GPT-4.1、Claude Opus 4、Gemini 2.5 和 Qwen3。每个模型都随着输入增长而退化。退化是非均匀且任务特定的。语义模糊的查询——相关段落与问题在词汇上不匹配的情况——表现出更陡峭的退化，指向一种复合失败：模型无法定位相关信息，定位后又无法对其进行推理。退化在窗口填满之前就已开始。一个标称 200K token 容量的模型在 50K 时就可能表现出显著的性能损失。Hong et al. (2025) 将这一现象称为 context rot，这并非边缘情况，而是任何在多步骤中累积工具结果、中间推理和文件内容的 agent 的正常运行条件。

综合这些结果可以确立：上下文管理不能是事后考虑。关于包含什么、放置在哪里、何时移除的每个决策都对 agent 可靠性产生直接影响。

## 5.2 从 Prompt Engineering 到 Context Engineering

从 prompt engineering 到 context engineering 的转变反映了范围的转变 (Karpathy, 2025)。Prompt engineering 优化的是单次模型调用的基本静态文本输入。在视觉领域，同样的框架扩展到连续可学习 token：visual prompt tuning 在冻结视觉 transformer 的 patch 序列前添加一小组可训练向量，使其适应下游任务，同时更新不到 1% 的模型参数 (Jia et al., 2022; Xiao et al., 2025)。文本和视觉两种变体共享 prompt engineering 的核心特征：针对单一输入模态的固定、单次调用优化。Context engineering 则优化模型在多步骤任务中每个推理步骤可用的完整信息状态。Anthropic 应用 AI 团队将其定义为"在 LLM 推理期间策划和维护最优 token 集合的策略集，包括可能出现在其中的 prompt 之外的所有其他信息"(Anthropic Applied AI Team, 2025)。由此得出的指导原则是：在每个步骤中找到最小化的高信号 token 集合，使期望结果的概率最大化。这一原则驱动了本节中的每项技术。渐进式披露按需加载信息而非预先加载。压缩移除已完成使命的 token。记忆检索仅拉取与当前任务最相关的记录。

**上下文包含什么。** 在已部署的 agent 中，推理时的上下文包括：系统提示词和行为指令、工具定义和 schema、先前轮次历史、当前轨迹中的工具调用结果、检索到的文档或记忆记录，以及任何动态注入的工作状态。所有这些都竞争同一有限的注意力预算。Context engineering 意味着在每个步骤中对每个组件做出明确选择，而非让它们不受控制地累积。

这一实践的成熟在生态系统中可见一斑。第一性原理手册 (Kimai, 2025) 和精选综述 (Meirtz, 2025) 现在将 context engineering 视为一门独立学科。涵盖活跃 context window、session 级状态和跨 session 存储的三层记忆架构已成为主导组织框架。它直接映射到操作系统的经典内存层次结构，既提供了有用的直觉，也提供了将正确技术匹配到正确时间跨度的词汇表。

## 5.3 短期：管理活跃 Context Window

短期上下文管理管控模型在单个推理步骤或一个 session 内短序列步骤中所见的内容。这些决策对模型行为有最直接的影响，且在做出错误决策时代价最高。

**系统提示词校准。** 系统提示词建立 agent 的行为包络，并在每次调用中消耗固定预算。Anthropic Applied AI Team (2025) 将设计挑战描述为找到正确的"高度"：过于具体的提示词引入脆弱的、维护成本高的逻辑；过于模糊的则使模型缺乏具体指导。在实践中，有效的系统提示词被组织为清晰划分的部分，包含背景、指令、工具指导和输出格式，使用 XML 标签或 Markdown 标题进行分隔。推荐的工作流程是从最佳可用模型上的最小提示词开始，通过经验识别失败模式，然后添加针对性指令来解决特定缺口。预防性地列举每个边缘情况往往会膨胀提示词而不提高可靠性。

**Token 高效的工具设计。** 工具定义是主要的上下文消耗者。每个 schema、名称、描述和参数类型都在每次调用时被注入。大型工具集在 agent 读取用户请求之前就可能消耗数万个 token。从生产部署中涌现的原则是：优选更少但更有表达力的工具，而非大量狭窄工具的菜单。如果人类工程师无法判断在给定情况下应使用哪个工具，也不能期望模型做得更好。工具应当是自包含的、对错误健壮的、在预期用途上无歧义的，并具有发挥模型优势的描述性参数名。

**即时检索与渐进式披露。** 有效的 agent 不是预先加载所有可能相关的信息，而是维护轻量级标识符如文件路径、存储查询和网页链接，并在任务展开时按需加载数据。Anthropic Applied AI Team (2025) 称这种方法为渐进式披露。它模仿了人类专家的工作方式：我们不记忆语料库，而是建立外部索引系统并按需检索。Claude Code 在实践中使用混合策略。CLAUDE.md 文件在 session 开始时加载以提供项目上下文，而 glob 和 grep 命令让 agent 即时导航和加载特定文件内容。这规避了过时索引问题和大型预填充的成本。环境元数据也携带隐式信号。文件大小暗示复杂度；命名约定提示用途；时间戳代理相关性。Agent 可以逐层构建理解，仅将当前必要的内容保留在活跃窗口中。

**KV-cache 感知的上下文设计。** Prompt caching 是生产 agent 部署中最具成本效益的单一优化，其收益完全取决于上下文的结构方式。经过五次架构迭代后，Manus 团队认定 KV-cache 命中率是"生产级 AI agent 最重要的单一指标"，并指出 Claude Sonnet 上缓存 token 的成本为 $0.30/MTok，而非缓存 token 为 $3.00/MTok (Manus Team, 2025)。从缓存模型中得出三条设计规则。第一，保持 prompt 前缀稳定：系统提示词开头的单个 token 差异会使其后所有内容的缓存失效。第二，将上下文视为仅追加：修改过去的动作或观察会通过创建不同的前缀序列来破坏缓存复用。第三，使用确定性序列化：JSON 序列化中不稳定的键排序会在本应相同的请求间无声地使缓存失效。由于工具定义通常出现在序列化上下文的前部，添加或移除工具会使所有后续轮次的缓存内容失效。Manus 通过使用上下文感知的状态机来解决这一问题——该状态机在解码期间遮蔽 token logits 以防止选择不可用的动作，而不是在运行时修改工具定义列表 (Manus Team, 2025)。Anthropic 的上下文管理发布通过显式 cache_control 断点在产品层面操作化了缓存 (Anthropic, 2025c)。

短期管理的工具生态系统已发展到包括生产级技能库，涵盖 agent 调用间的 KV-cache 压缩模式 (Koylan, 2025)，以及提供以 MCP 为中心的动态上下文组装原语的基础设施项目 (context-space, 2025)。

## 5.4 中期：Session 状态与跨运行持久化

中期上下文管理解决活跃 context window 与完整长期记忆系统之间的差距。具体问题是 agent 如何在 session 内的轮次之间，或在同一任务的多次运行之间，保存和恢复状态。工程价值很高：几百个 token 的结构化工作状态就能弥合上下文重置，否则 agent 将失去所有累积的进展。

**结构化笔记。** 最简单的中期技术是让 agent 维护一个持久化笔记文件，通常是 NOTES.md 或 todo.md，在每次运行开始时读取它，并在上下文清除前更新它。Anthropic Applied AI Team (2025) 通过 Claude 在数千个游戏步骤中玩 Pokémon 来说明这一点。在没有任何关于记忆结构的显式指令的情况下，agent 发展出了已探索区域的地图，维护了针对特定对手的战斗策略及其结果统计，并跨上下文重置跟踪目标。在每个清除主要对话历史的压缩步骤之后，agent 读取自己的笔记并继续多小时的训练序列而不失去连贯性。关键洞察在于笔记将工作记忆外部化。agent 不依赖对话历史来承载任务状态，而是将重要内容写入外部存储并按需读回。

**基于文件的规划与任务外部化。** 结构化笔记的扩展将完整的规划表示外部化到磁盘。如 planning-with-files (OthmanAdi, 2025) 等工具为编码 agent 工作流中的持久化基于文件的规划提供技能包。任务状态、规格说明、中间结果和依赖图在每个里程碑写入文件系统，并在下一次运行开始时选择性加载，完全绕过 context window 处理当前不直接相关的内容。Trellis (mindfold-ai, 2025) 等框架将此模式扩展到包含项目记忆和规格注入，管理结构化项目状态，根据当前步骤的需要选择性注入。

**跨运行注入。** 一个互补模式捕获先前运行的突出输出，并在下一次运行开始时注入。如 claude-mem (thedotmack, 2025) 等工具作为插件式记忆层运行：它们捕获 session 历史，提取在未来运行中最有用的信息，并将其预置于下一个 session 的上下文中。这不需要向量数据库也不需要图存储，同时提供跨运行的实质性连续性。社区仓库 everything-claude-code (affaan-m, 2025) 拥有超过 15 万 GitHub 星标，代表了这些模式最大的开放集合，展示了中间层记忆实践在生产编码 agent 部署中被采用的规模。

**中间层的权衡。** 中期技术提供的连续性显著优于仅窗口内管理，而其基础设施成本仅为完整长期记忆系统的一小部分。局限在于信息从一次运行向下一次运行前向流动，但不从索引存储中检索，且这些技术不能很好地扩展到大量历史。当跨运行历史增长庞大，或当 agent 需要检索特定的先前观察而非注入摘要时，中期技术必须辅以下文描述的长期系统。

## 5.5 长期：持久化记忆系统

长期记忆系统提供跨 session、跨任务、跨 agent 实例持久化的索引化可检索存储。中期技术依赖摘要的前向注入，而长期系统支持任意召回：给定查询，系统检索最相关的存储记忆，无论其创建时间。该层级是 agent 经验随时间累积的场所。

### 5.5.1 基础架构

**操作系统启发的分层记忆。** Packer et al. (2023) 通过将 LLM 的 context window 视为 RAM、将外部存储视为磁盘，引入了关键抽象，赋予模型函数调用能力让其显式地将信息换入换出。与虚拟内存的类比是精确的：正如操作系统通过透明分页为应用程序提供更大内存的幻觉，MemGPT 通过透明的记忆管理为 LLM 提供更长上下文的幻觉。Agent 可以在远超物理窗口限制的上下文上操作，按需从外部存储检索相关历史。该论文明确了 agent 记忆与操作系统内存之间的联系，这一联系支撑了大多数后续的长期记忆系统。

**观察、反思与检索。** Park et al. (2023) 在社会模拟 agent 的背景下引入了 Memory Stream 架构。Memory Stream 将 agent 的所有观察存储为带时间戳和重要性评分的自然语言记录。在每个步骤中，检索模型通过组合三个信号来呈现最相关的记忆：时近性、重要性（由 agent 在写入时评分）和与当前查询的相关性。第二个机制——反思——定期将存储的观察综合为更高层次的洞察。Agent 自问其近期经验中最突出的问题是什么，检索相关记录，并生成被存储回 Memory Stream 的结论。消融实验表明，观察、反思和检索三个组件各自独立地贡献于行为质量；移除任何一个都会产生可测量的退化。观察-反思-检索三元组仍然是 agent 记忆设计的标准模板。

**动态遗忘与人格建模。** Zhong et al. (2024) 在检索架构之上构建了 MemoryBank，增加了两个要素。第一是受 Ebbinghaus 遗忘曲线启发的动态遗忘机制：记忆按指数模型随时间衰减，频繁访问加强记忆，矛盾的记忆被解决。第二是层次化的用户人格摘要，随着新交互的累积每日更新。这些思想——自适应记忆强度和用户建模——后来在 Mem0 和 Honcho 中被规模化操作。

### 5.5.2 生产记忆系统

**混合存储与行业采用。** Mem0 (Chhikara et al., 2025) 是目前生产 agent 中部署最广泛的长期记忆层，拥有超过 1400 万次 Python 包下载、41K GitHub 星标，以及在 CrewAI、Flowise 和 Langflow 中的原生集成。其架构结合了三种存储后端：用于语义相似度搜索的向量数据库、用于关系建模的图数据库，以及用于快速事实检索的键值存储。一个基于 LLM 的提取层处理新交互，识别事实和偏好，并将记录路由到相应的存储。在 LOCOMO 基准测试上，Mem0 比 OpenAI 的原生记忆准确率高 26%，同时比全上下文方法少用 90% 的 token (Chhikara et al., 2025)。Amazon Web Services 在 2025 年选择 Mem0 作为其 Agent SDK 的独占记忆提供商，反映了其从研究原型向生产基础设施的转变。

**动态知识网络。** A-MEM (Xu et al., 2025) 借鉴了 Zettelkasten 知识管理系统。它不是将记忆存储为扁平记录，而是为每条新记忆生成一个结构化笔记，包含关键词、标签、上下文描述和一组与语义相关记忆的动态链接。当添加新记忆时，A-MEM 不仅存储它，还追溯更新现有相关笔记的上下文和属性，使记忆图随着知识的积累而演化。这解决了基于检索系统的一个根本局限：存储记忆的相关性在写入时可能并不明显，其重要性只能在与后续信息的关系中评估。

**从经验中学习。** Hindsight (Vectorize.io, 2025) 持这样的立场：大多数记忆系统专注于记住，而真正的需求是学习。其 API 暴露三个操作：retain 存储新信息，recall 检索相关记忆，reflect 生成整合记忆与当前上下文的倾向感知响应。在内部，retain 操作提取时间实体，去重重叠事实，并构建基于证据的整合知识记录。Hindsight 在 LongMemEval 基准测试上达到最先进性能，由弗吉尼亚理工大学 Sanghani 中心的研究者独立复现。

**推理驱动的个性化。** Honcho (Plastic Labs, 2025) 由 Plastic Labs 开发，构建的是用户模型而非关于用户的事实存储。一个称为"dreaming"的异步推理管道在后台持续处理过去的交互，不仅提取显式事实，还推断关于偏好、沟通风格和演变目标的结论。以实体为中心的数据模型支持多 agent 环境，其中多个 agent 与同一用户交互：每个 agent 维护自己的观察视角，防止交叉污染，而共享的用户模型从所有交互中累积理解。

**集体记忆。** Mozilla AI 的 cq (Mozilla AI, 2025) 解决了其他系统遗留的空白：跨 agent 实例的共享记忆。大多数长期记忆系统是按 agent 或按用户的；当多个 agent 处理相关任务时，每个实例独立重新发现其他实例已经学到的知识。cq 是一个用于共享 agent 学习的开放标准，其中 agent 可以在一个集群中持久化、查询和确认集体知识。其架构是 MCP 原生的，在三个层级运行：存储在开发者机器上的本地知识、团队可访问的组织共享知识，以及跨团队知识。错误后钩子模式——在 agent 遇到错误时自动查询集体知识库——提供了防止重复失败在组织部署中被独立解决的具体机制。

### 5.5.3 学术综述与分类学

Zhang et al. (2025) 的记忆机制综述为该层提供了标准的学术分类法。它区分了短期工作记忆和长期记忆，并将后者细分为语义、程序性和情节性类别。它所形式化的写入-读取-管理循环已成为描述记忆系统行为的标准词汇。更近期的综述 (Du, 2026) 在此基础上，将该循环形式化为 POMDP 风格的 agent 周期，并提出覆盖记忆范围、存储格式和组合结构的三维分类法。它刻画了三种工程模式：模式 A（整体上下文）、模式 B（context window 加检索存储）和模式 C（具有学习控制策略的分层记忆）。这些大致对应于本节描述的短期、中期和长期层级。Gao et al. (2023) 的 RAG 综述为上述生产记忆系统的检索层提供了关于检索增强生成的背景知识。

## 5.6 长周期技术：保持 Agent 在 100+ 轮次中的连贯性

前述各节分别处理单一时间跨度。长周期任务——包括大型代码库迁移、多 session 研究项目和扩展的自主工作流——需要同时协调所有三个层级，并实时决定在每个点应用哪种技术。本节涵盖在此规模下涌现的集成级技术。

**上下文压缩。** 压缩对接近限制的 context window 进行摘要，并以累积状态的压缩表示重新开始执行。Anthropic Applied AI Team (2025) 详细描述了校准挑战。摘要必须保留架构决策、未解决的 bug 和实现细节，同时丢弃冗余的工具输出和已融入后续步骤的中间推理。推荐的调优工作流从最大化召回率开始——捕获轨迹中每一条可能相关的信息——然后迭代以通过移除多余内容来提高精确率。不能从另一个方向开始。早期移除过多会产生一个丢失后续所需上下文的 agent，而这种损失是不可恢复的。工具结果清除是最轻量的压缩形式：在 agent 已对完整工具输出采取行动后，将其替换为紧凑的路径引用。这可以持续应用，现在已是 Anthropic 开发者平台上的产品级功能 (Anthropic, 2025c)。

**子 agent 上下文隔离。** 当任务需要深入探索某个子主题时，探索本身会生成大量中间上下文——这些对子任务有用但会污染协调者对整体任务的视图。子 agent 架构通过将子任务分配给专用 agent 来解决这一问题，每个 agent 拥有全新的 context window，仅向协调者返回 1,000 至 2,000 token 的精炼发现摘要 (Anthropic, 2025e)。详细的探索上下文保持隔离在子 agent 内；协调者仅接收提炼结果。Manus Team (2025) 描述了该模式的改进。对于协调者仅需要输出的简单子任务，上下文完全不共享。对于需要共享状态的复杂子任务，协调者的完整上下文传递给子 agent。Agent 间共享上下文是昂贵的：它产生更大的预填充并消除不同系统提示词间的 KV-cache 复用。隔离成本与协调收益之间的权衡必须针对每种任务类型显式做出。

**混合决策框架。** 生产部署不会统一应用单一技术；它们根据任务结构在执行的不同点选择不同技术。Anthropic (2026c) 将该框架描述为：预加载始终需要的内容，即时检索条件性需要的内容，当窗口接近饱和时压缩历史，当子任务需要会污染协调者上下文的探索时生成子 agent。Anthropic (2025d) 补充了 harness 层视角：检查点和恢复模式允许 agent 在不丢失任务状态的情况下经受瞬态故障，生命周期钩子可以在可配置的阈值处自动触发压缩或记忆整合。这是正确的职责划分。上下文管理应当是基础设施的工作，而非 agent 的。当 harness 自动处理压缩和整合时，模型可以专注于任务推理。

## 5.7 Context Drift 与当前方法的局限性

绑定约束视角将 context drift 框定为控制器侧的失败模式：因为模型仅看到 harness 暴露的状态投影，任务相关上下文的丢失或失真是 harness 的属性，而非仅是模型的属性 (Bölük, 2026b)。上述所有技术在不同程度上、在不同条件下都有效。但它们都没有解决根本问题。Context drift——agent 行为在扩展交互中的渐进退化——仍然是该层中最困难的开放挑战。

**问题的本质。** Context drift 与 context rot 是不同的，尽管两者都源于相同的架构约束。Context rot（第 5.1 节）是单个推理步骤的属性：向固定上下文添加更多 token 会降低检索和推理质量。Context drift 是扩展轨迹的属性。随着 agent 累积轮次，其行为以压缩和检索步骤能减缓但无法阻止的方式偏离原始意图。在 100 轮或更多轮次之后，agent 经常重复已完成的工作、在不自知的情况下与先前决策矛盾、并失去对激励目标的跟踪 (Bowne-Anderson & Huber, 2026)。根本原因不仅是窗口饱和。即使在激进的压缩下，在每个压缩步骤中存活的摘要也带有随时间复合的细微不准确。Agent 累积的假设可能偏离实际任务环境，而当前架构中没有任何机制检测这种偏离。

**评估空白。** 一个次要问题是 context drift 难以测量。标准基准测试在几十个步骤内完成的任务上评估 agent；在 100 轮或更多轮次出现的失败模式未被捕获。近期工作已开始弥合这一差距。Tan et al. (2025) 引入 MemBench，评估多 session 场景下的记忆质量，包括时间推理、知识更新和跨 session 聚合。He et al. (2026) 专门为相互依赖的多 session 任务引入 MemoryArena——这正是 context drift 最具破坏性的条件。Hu et al. (2025b) 提出增量多轮评估方法论，将记忆质量与生成质量隔离。这些基准测试是必要的但不充分的。它们表明漂移确实发生，但尚未提供预防所需的机制性理解。

**当前技术为何不足。** 压缩减少了 token 计数，但无法保证压缩表示准确捕获所有相关状态。每个压缩步骤中丢失的信息一去不返。检索系统可以呈现相关的先前记忆，但查询必须格式良好才能做到这一点。一个正在漂移的 agent 可能不知道它需要检索什么来纠正其航向。子 agent 隔离防止子任务上下文污染协调者，但协调者自身的上下文仍在随时间累积漂移。Bowne-Anderson & Huber (2026) 认为这些局限指向一个边界：仅靠 context engineering 无法解决长周期可靠性问题。需要一个完整的 harness 层，包含验证循环、在战略间隔点的人在回路检查点，以及能够识别行为偏离的异常检测。这促使了第 7 节和第 9 节中讨论的治理和可观测性层成为 context engineering 的必要补充，而非独立关注点。

## 6.1 生命周期状态管理

生命周期状态管理关注跨轮次、session、故障和交接继续 agent 运行所需的操作状态。这包括待处理子任务、工具结果、中间工件、仓库变更、协调元数据、session 持久化和检查点恢复机制。这些机制使编排持久而非瞬态：agent 可以从先前工作继续，而不是将每个步骤视为孤立的模型调用。

核心权衡在无状态和有状态执行之间。无状态设计从先前交互历史重建执行，提高了可重复性和可审计性，但随着轨迹增长而变得昂贵。有状态设计将执行状态持久化在文件、数据库、仓库、任务图或外部服务中，提高了连续性和恢复能力，但引入了一致性和调试挑战 (OpenAI, 2026b; Anthropic, 2026c)。因此许多实际 harness 采用混合设计，将可重放的交互历史与持久化工件结合。

该状态与 §5 中的上下文与记忆层不同，后者关注提供给模型用于推理的信息，如检索到的文档、记忆或对话历史。它也与 §7 中的可观测性不同，后者关注日志记录、跟踪和监控系统行为。这里的重点是 harness 本身用于继续和协调执行的操作状态，如待处理子任务、检查点、重试、共享工件或执行状态。

## 6.2 单 Agent 内循环

单 agent 内循环是 agent 系统中的基本执行单元。单个 agent 通过工具使用和反馈与环境重复交互，没有多个 agent 之间的显式协调。这一抽象支撑了交互式编码、调试和问题解决。

从概念上讲，单 agent 循环遵循 ReAct 范式 (Yao et al., 2023)，交错推理、动作和观察。正如 OpenAI 对 Codex agent 循环的分析所强调的 (OpenAI, 2026b)，这类系统的行为不仅由模型决定，还由构建 prompt、调用工具、管理控制流并将工具输出反馈到后续步骤的 harness 决定。

在此层次，主要执行区分在于无状态重放和混合执行。Codex CLI (OpenAI, 2025) 是基于重放执行的最清晰示例，而实际编码 agent 通常将可重放历史与文件、仓库或 session 状态等持久化工件结合。在表 2 中，OpenCode (Anomaly, 2025)、Claude Code (Anthropic, 2025)、Gemini CLI (Google, 2025)、Codex CLI (OpenAI, 2025)、Aider (Aider-AI, 2025) 和 SWE-agent (SWE-agent, 2025) 因此被归入主要模式"单循环"。尽管具有灵活性，此类系统中的长周期执行可能遭受上下文碎片化、累积错误和薄弱的分解结构 (Anthropic, 2025d)，这激发了更结构化的编排。

## 6.3 多 Agent 编排模式

多 agent 编排通过组合具有专门角色的多个 agent 来扩展单 agent 循环。多 agent harness 不要求一个 agent 独自规划、执行、检查和修订，而是可以将这些责任分配给不同的 agent 或子 agent。这种结构支持任务分解、并行探索、批评、验证和聚合，使其比孤立循环更适合复杂任务。

该设计空间中出现了几种反复出现的模式。**层次化编排**使用更高层次的控制器将工作分配给 agent 或子 agent 并整合其输出。**团队编排**将一组专业化 agent 作为协调团队暴露，通常具有不同的命名职责。**工作流编排**将 agent 和工具组织为显式阶段或控制逻辑。**扇出**并行运行多个 agent 以探索多样化解决方案。**图组合**将 agent、工具或状态表示为交互图中的节点，允许多种协调模式共存。这些标签与表 2 中使用的主要模式类别相匹配。

该层次的执行通常是有状态或混合的，因为多 agent 系统必须维护协调状态、角色分配、任务图、共享工件或中间结果。Anthropic 的规划者-生成者-评估者架构 (Anthropic, 2025e) 说明了更广泛的原则：规划、执行和验证可以分离为显式角色，在增加协调开销的同时改善分解和鲁棒性 (Anthropic, 2026c)。

表 2 中的系统以不同方式实例化这些模式。DeerFlow (Bytedance, 2026)、AutoGen (Microsoft, 2025)、OpenAI Agents SDK (OpenAI, 2026a) 和 DeepAgents (LangChain, 2026b) 被归类为层次化编排。oh-my-claudecode (Heo, 2026) 被归类为团队编排。LangGraph (LangChain, 2026a) 和 Hive (Aden, 2026) 示范了图组合，Semantic Kernel (Microsoft, 2026) 示范了工作流编排，Emdash (Action, 2026) 示范了扇出。

## 6.4 从 Issue 到 Pull Request 的完整生命周期管线

完整生命周期编排管理从规格说明到验证输出的整个任务工作流。在此层次，关注点不仅在于 agent 如何交互，还在于 harness 如何支持跨规划、实现、验证、评审和交付的长周期执行。

核心抽象是**任务运行器**：一个管理调度、状态持久化、重试、验证和完整任务生命周期上迭代改进的 harness (OpenAI, 2026a; LangChain, 2026b)。执行基于持久化工件如仓库、issue、分支、文件、测试和 pull request。典型工作流从 issue 或任务规格说明出发，经过 agent 规划、代码或工件生成、验证、人类评审，最终到 pull request 接受。

由于这些系统在持久化仓库和外部工作流上运行，主导的执行模型通常是有状态的。某些系统在子组件内将持久化基础设施状态与类似重放或 session 本地的执行结合，促成了表 2 中的混合标签。Vibe Kanban (Bloop-AI, 2026)、Symphony (OpenAI, 2026b) 和 GitHub Agentic Workflows (GitHub, 2026) 被归类为完整生命周期管线。从概念上讲，这些系统组合了前述抽象：单 agent 循环提供执行原语，多 agent 模式提供协调，任务运行器将它们整合为端到端工作流，其中人类引导而 agent 执行。
# 7 可观测性与运维 (O)

可观测性与运维（O）是 ETCLOVG 的第五层，也是我们分类法从生命周期钩子提升为一等公民地位的两个层之一（§1 中的论断 2）。支撑系统来自支持论断 3 的 170+ 项目语料库。

该层处理如何在生产环境中监控、调试和确保 agent 行为的可靠性。我们论证可观测性值得独立处理、与生命周期钩子分离，因为它已经催生了一个专门的平台、规范和工程实践生态系统。

## 7.1 追踪与监控平台

Agent 可观测性的基础是结构化追踪收集：将每个 LLM 调用、工具调用、检索步骤和上下文组装操作记录为一棵可检查、可过滤、可重放的 span 树。Langfuse、Opik、Arize Phoenix 和 MLflow 各自提供交互式追踪树，配备延迟火焰图、token 用量细分、成本归因面板和 prompt 版本管理 (Langfuse, 2026; Comet ML, 2026; Arize AI, 2026b; MLflow, 2026)。这些平台通常通过轻量级 SDK 包装器（如 @traceable 装饰器）摄取追踪数据，以最少的代码变更对 agent 的函数调用进行插桩。

在这些平台之下，OpenTelemetry (OTel) 已成为事实上的插桩标准。OTel 社区发布了生成式 AI 语义约定，定义了模型名称、温度、token 计数和延迟的 span 属性 (OpenTelemetry, 2026)。两个开源项目将这些约定付诸实践：OpenLLMetry 为 LLM 提供商（OpenAI、Anthropic、Cohere）和向量数据库（Pinecone、Chroma、Weaviate）提供自动插桩库，发出标准 OTel span 和指标 (Traceloop, 2026)；OpenInference 由 Arize AI 维护，提供与 Phoenix 配套的补充规范和 OTel 对齐的语义约定与自动插桩 (Arize AI, 2026a)。通过构建在 OTel 之上，这些库允许 agent 追踪流入团队已用于传统微服务监控的相同可观测性后端（Prometheus、Jaeger、Grafana、Datadog），降低了采用 agent 专用工具的运维开销。

在更深层的插桩层面，学术工作探索了超越应用层装饰器的技术。AgentSight (Zheng et al., 2025) 使用 eBPF 从应用进程外部监控 agent，在 SSL 边界截获 TLS 加密的 LLM 流量以捕获意图，并监控内核事件（进程创建、文件 I/O、网络调用）以捕获动作。实时关联引擎将 LLM 响应与其触发的系统行为关联起来，辅助"观察者"LLM 执行语义分析以识别风险。该方法与框架无关、无需代码修改、CPU 开销低于 3%。作者指出相比应用层工具的结构性优势：系统级监控不能被受损或配置错误的 agent 绕过，这使其特别适用于安全关键部署。AgentTrace (AlSayyad et al., 2026) 提出互补的基于 schema 的日志框架，捕获跨三个"表面"的结构化日志：认知推理步骤、计划和反思；运维工具调用和 API 交互；以及上下文环境状态和用户输入。这些表面组织在与 OTel 集成导出的统一信封之下。认知表面对 harness 工程特别相关，因为它捕获显式推理工件——计划、反思和自我修正。这些记录为调试源于错误推理而非系统错误的故障提供了原材料。

## 7.2 Agent 专用运维平台

通用追踪平台通常将抽象围绕 LLM 调用、工具调用和检索步骤组织为 span 级事件。Agent 专用平台增加了一层抽象，捕获 agent 级关注点：多步会话跟踪、agent 身份和角色管理、工具选择策略以及跨会话状态交接。AgentOps SDK 引入层次化 span 模型，围绕 agent 生命周期组织会话、agent、操作或任务、工作流、工具调用和 LLM 调用，而非将其视为孤立的 API 调用 (AgentOps AI, 2026)。RagaAI Catalyst 针对 RAG 和多 agent 工作负载，提供呈现检索级和 agent 级异常的质量与安全面板 (RagaAI, 2026)。Laminar 提供开源优先的 agent 可观测性平台，结合追踪、评估和 prompt 管理 (Laminar AI, 2026)。

学术社区已开始形式化这些关注点。研究者 Moshkovich & Zeltyn (2025) 提出六阶段 AgentOps 自动化管线：观察、收集指标、检测异常、执行根因分析、推荐修复和自动化补救。他们进一步将利益相关者空间分解为四种角色：开发者、测试者、SRE 和业务用户，每种角色在生命周期的不同点参与管线。其配套实证研究 (Moshkovich et al., 2025) 将任务流发现作为可观测性原语引入，并通过用户研究证明 79% 的从业者将非确定性执行流视为 agent 系统中最显著的挑战。

Natural-Language Agent Harnesses (NLAH) 框架 (Pan et al., 2026) 及其配套开源实现，采取互补步骤，将 harness 本身视为一等研究对象。通过系统性消融实验，作者量化了各 harness 模块对下游 agent 性能的贡献，这些模块包括工具注册表、权限系统、钩子、技能和上下文折叠。对可观测性的启示是：harness 不仅是监控的对象，还是干预的来源——每个被消融的模块都是可观测性管线可以标记和调优的旋钮。

认知可观测性将 agent 级监控扩展到不仅询问 agent 做了什么，还询问为什么。Watson (Rombaut et al., 2025) 正式引入此概念，部署一个"代理 agent"来复现主 agent 的输出，同时通过 prompt 归因生成逐步推理追踪。在 MMLU 和 SWE-bench-lite 上用 AutoCodeRover 和 OpenHands 进行评估，Watson 证明恢复的推理追踪对手动调试和自动化修正都有用，在任务成功率上产生可衡量的改进。AgentLens (Lu et al., 2024) 以三窗格可视分析系统完成了可视化闭环：Outline View 用于 agent 轨迹，Agent View 用于带因果弧的事件栈，Monitor View 用于同步环境回放。该系统将原始执行日志转化为层次化结构的行为叙事。一项 14 名参与者的用户研究显示，在复杂分析任务上相比纯重放基线有显著改进。更专业的可视化工具如 claude-code-reverse 逆向工程特定编码 agent 的交互链，作为理解 harness 级行为的研究工件 (Yu, 2026)。

## 7.3 成本追踪与优化

LLM 推理按 token 计费，agent harness 放大了成本暴露，因为单个面向用户的任务可能触发数十个 LLM 调用，每个都有自己的 prompt 组装、工具结果注入和响应生成。可观测性需求是双重的：追踪（知道 token 花在哪里）和优化（用更少的 token 达到相同效果）。

在追踪方面，TensorZero 提供统一的 LLMOps 栈，将网关、可观测性、实验和优化捆绑为单一服务，实现按调用和按任务的成本归因 (TensorZero, 2026)。Helicone 采用极简方式作为直通代理，无需代码变更即可添加成本和延迟监控，对想要成本可见性而不承诺完整可观测性平台的团队特别易于接入 (Helicone, 2026)。

在优化方面，FrugalGPT (Chen et al., 2023) 提供了基础性表述，提出三种降本策略：prompt 适配、LLM 近似和 LLM 级联。它表明自适应级联可以匹配 GPT-4 的性能，同时成本降低高达 98%，通过将较简单的查询路由到较便宜的模型。GPTCache (Bang, 2023) 用语义缓存层补充此方法，在重复或释义的查询到达 LLM 之前使用嵌入相似度检索缓存响应进行拦截。路由和缓存已成为生产成本优化栈中反复出现的构建模块，直接适用于 harness 级插桩，其中每个工具调用和上下文组装步骤都可以独立计量和路由。

QC-Opt (Shekhar et al., 2024) 通过质量感知框架扩展路由范式，联合优化模型选择、token 数量和延迟，在预算约束下使用 BertScore 预测器估计输出质量而无需调用目标模型。TALE (Han et al., 2025) 发现的 token 弹性现象揭示了一个重要细微差别：为思维链推理设置过低的 token 预算可能矛盾地增加 token 消耗，因为模型溢出预算产生的 token 比适度预算的 prompt 更多。在基础设施层面，Dual-Pool Token-Budget Routing (Liu et al., 2026b) 从服务端解决成本问题，将同质 vLLM 集群分为短上下文和长上下文池，根据估计的 token 预算路由请求。在 Azure LLM 推理追踪和 LMSYS-Chat-1M 上评估，该方法减少 31-42% 的 GPU 时长（预计在 A100 集群规模下年节省 286 万美元）。

对 harness 工程师而言，启示是成本可观测性必须跨多个层：API 级 token 追踪（Helicone、TensorZero）、应用级路由决策（FrugalGPT、QC-Opt）和基础设施级资源利用（Dual-Pool、KV-cache 占用）。Anthropic 对 agent 编码评估中基础设施噪声的研究 (Anthropic, 2026a) 提供了警示性补充：仅基础设施配置就可以使基准分数偏移 6 个百分点（p<0.01）。这一发现表明成本优化和评估保真度紧密交织，因为削减资源以节省成本可以悄无声息地降低 agent 性能，而没有细粒度可观测性则无从察觉。

## 7.4 可靠性工程

故障恢复、检查点/恢复、重试策略和会话恢复是 harness 级关注点，决定了长期运行的 agent 是否能在瞬态故障中存活。当 agent 跨多个上下文窗口运行时，这些关注点变得尖锐，每个新会话开始时对之前发生的事一无所知 (Anthropic, 2025d)。Anthropic 的 harness 工程工作确定了长期运行编码 agent 中四种反复出现的故障模式：agent 试图一次完成整个任务、agent 过早宣布项目完成、agent 在会话之间留下破碎的环境状态、以及 agent 未经适当测试就将功能标记为完成。其两部分解决方案通过 harness 设计而非模型改进直接解决这些可靠性问题 (Anthropic, 2025d)。它使用初始化 agent 将任务分解为结构化功能列表并设置进度跟踪工件（包括 git 仓库、进度文件和初始化脚本），然后由编码 agent 一次增量处理一个功能同时保持干净的交接状态。

后续工作扩展了此模式。Anthropic 的 GAN 启发的三 agent 架构（规划者、生成者、评估者）引入冲刺合同和分离的自我评估，解决 agent 过度赞美自己工作的倾向 (Anthropic, 2026c)。值得注意的是，作者证明 harness 复杂度应跟踪模型能力：从 Opus 4.5 升级到 Opus 4.6 时，他们完全移除了冲刺结构和上下文重置，将成本从 200 美元降至 125 美元同时维持输出质量。这说明了一个一般原则：**每个 harness 组件都编码了关于模型自身无法做什么的假设，而这些假设随着模型改进会过时。**

在基础设施层面，Anthropic 的 Managed Agents 架构 (Anthropic, 2026b) 将"大脑"（harness + LLM）与"手"（沙箱和工具）和"会话"（持久化事件日志）解耦，使每个部分可独立恢复。如果 harness 崩溃，新实例可以通过 wake(sessionId) 重启并从会话日志的最后事件恢复。如果沙箱失败，harness 将错误捕获为工具调用失败并供应新沙箱。凭证存储在外部保险库中，永不进入沙箱。此架构将所有组件从"宠物"（不可替代的、手动维护的实例）转变为"牲畜"（可互换的、自动重新供应的），这是大规模可靠运行 agent 的前提。

学术文献提供了可靠性工程必须解决的故障模式分类法。MAST (Cemri et al., 2025) 确定了多 agent 系统中 14 种故障模式，聚类为系统设计问题、agent 间失调和任务验证问题（κ=0.88）。Agent Error Taxonomy (Zhu et al., 2025) 按模块（记忆、反思、规划、动作、系统）分解故障，表明错误传播——单一根因故障级联穿过后续决策——是核心可靠性瓶颈。其 AgentDebug 框架通过隔离根因而非治疗表面症状，在任务成功率上产生高达 26% 的相对改进。同时，Vinay (2025) 贡献了 15 种生产 LLM 部署特有的隐藏故障模式的系统级分类法，包括版本漂移（模型更新悄然改变行为）、成本驱动的性能崩溃（激进优化降低质量）和多步推理漂移（长序列上的逐渐连贯性丧失）。QSAF (Atta et al., 2025) 将认知退化形式化为跨五个 LLM 平台验证的六阶段生命周期，揭示幻觉输出可被存储在持久记忆中并在未来会话中复用，创建跨越会话边界的自我强化退化循环。

对于运行时检测，SentinelAgent (He et al., 2025) 将多 agent 交互建模为图，在三个层级（全局、单点、多点）使用 LLM-as-judge 配合人在环路策略优化进行异常分类。AgentFixer (Mulian et al., 2026) 在 IBM 的 CUGA 生产系统上部署 15 个验证工具，实现 64-88% 的问题检测率，发现 38% 的任务失败可追溯到解析错误——一种完全可由确定性检查解决的故障模式。实际启示是 agent 可靠性需要分层检测策略：轻量规则检查用于结构性故障（格式错误的工具调用、schema 违规），统计监控用于性能漂移（延迟、token 用量、成本趋势），LLM 基语义分析用于推理故障（幻觉、计划-动作失调、过早停止）。

## 7.5 讨论：走向统一可观测性

**可观测性-评估缺口。** LangChain 调查中 89%/52% 的分裂反映了结构性脱节：追踪收集工具和评估框架在很大程度上由不同社区用不同接口开发。缩小此缺口需要更紧密的集成，例如从生产故障追踪自动生成回归测试用例，或使用在线评估分数作为告警信号。LangChain 自身的深度 agent 评估分析 (LangChain, 2025a) 和 Anthropic 对基础设施噪声的量化 (Anthropic, 2026a) 都论证评估和可观测性必须作为单一反馈循环而非分离关注点来处理。

**统一 LLMOps 栈。** TensorZero 和 Langfuse 等工具正朝向将网关、可观测性、评估和优化捆绑为单一平台，减少集成开销。NLAH 框架将此推进一步，使 harness 本身模块化且可内省，每个组件的贡献可通过消融量化。Anthropic 的 Managed Agents 架构采取基础设施级视角，将 harness 组件虚拟化为可替换接口（execute()、emitEvent()、getEvents()），能够在不改变周围系统的情况下容纳未来的 harness 设计。

**Harness 即假设原则。** 每个 harness 组件都编码了关于模型自身无法做什么的假设 (Anthropic, 2026c)。随着模型改进，可观测性系统不仅必须检测 agent 何时失败，还要检测 harness 组件何时变成不必要的开销。理想的可观测性系统将包含一个元监控层，追踪哪些干预（上下文重置、评估者反馈循环、工具限制）仍然承载负荷，实现随模型升级的持续 harness 简化。

# 8 验证与评估 (V)

验证与评估（V）是 ETCLOVG 的第六支柱（§1 中的论断 2）。我们讨论的系统和基准来自支持论断 3 的 170+ 项目语料库。

## 8.1 将 Harness 评估视为任务到反馈的生命周期

具有 harness 意识的评估应将报告的分数视为模型-harness 对的属性，而非仅模型的属性。这促使评估协议要么在模型间锁定 harness，要么将 harness 配置作为显式实验因素进行变化 (Bölük, 2026b)。我们将 harness 评估组织为任务到反馈的生命周期：一个从定义 agent 被要求做什么开始、到将评估结果反馈回 harness 改进结束的结构化过程。

此生命周期建立在传统 LLM 评估和 agent 评估之间的关键区别上。传统 LLM 评估主要在固定输入上对输出评分，而 harness 评估衡量一个执行情节：任务在环境中被实例化，agent 随时间与工具和状态交互，追踪被捕获，评估者既判断最终结果也判断达到该结果的路径 (Kapoor et al., 2025; Anthropic, 2026b; LangChain, 2025b)。

此生命周期的核心动机是评估基础设施噪声可能伪装成模型失败：失败的运行可能不仅反映模型限制，还反映工具损坏、陈旧上下文、未重置的沙箱、不稳定测试、基准歧义或不稳定的评判器 (Kapoor et al., 2025; Hu et al., 2025a; Jimenez et al., 2024)。因此，评估应将 agent 行为转化为结构化判断、故障归因和回归反馈，而非仅报告最终分数。

五阶段分解遵循 agent 评估运行的因果路径。在 agent 被评估之前，基准必须指定任务、环境、工具、约束和成功标准。在执行之前，必须验证设置以确保环境或评分器故障不被误认为模型故障。在执行期间，harness 必须捕获使运行可诊断的追踪。在执行之后，系统必须判断结果、轨迹和评估者可靠性，然后将故障归因到可能的 harness 组件。最后，产生的诊断应反馈回回归测试和未来 harness 修订。在此视角下，评估不是终端评分步骤，而是对完整 agent harness 的质量控制循环。

我们围绕五个阶段组织讨论：

- **阶段 1：任务与基准实例化** 定义正在评估什么、在哪个环境中、用哪些工具、约束和成功标准（§8.2）。
- **阶段 2：执行前就绪验证** 检查沙箱、依赖、工具、上下文状态、权限策略、预算和评分器是否在 agent 运行前正确初始化（§8.3）。
- **阶段 3：受控执行与追踪捕获** 在可复现条件下运行 agent，同时记录模型输出、工具调用、状态变化、错误、重试、成本和延迟（§8.4）。
- **阶段 4：多级判断与故障归因** 在结果、轨迹和评估者三个层面评估运行，然后在 harness 栈中定位可能的故障来源（§8.5）。
- **阶段 5：持续回归与部署反馈** 将评估结果转化为回归测试、监控信号和后续 harness 修订的工程反馈（§8.6）。

## 8.2 阶段 1：任务与基准实例化

第一阶段询问正在评估什么。对于 LLM agent，任务不仅仅是自然语言提示，而是由环境状态、可用工具、允许的动作、约束、终止条件和成功标准定义的嵌入问题。因此任务实例化是 harness 评估的基础：没有良好定义的环境和成功条件，后续分数无法可靠解释。

### 8.2.1 软件工程与终端任务

软件工程基准是最成熟的 agent 评估形式之一，因为它们结合了真实任务与可执行的结果验证。SWE-bench 将每个任务置于真实 GitHub issue 和仓库快照中，然后通过运行测试评估生成的补丁是否解决了 issue (Jimenez et al., 2024)。Terminal-Bench 将此想法扩展到命令行工作流，agent 通过编辑文件、运行命令、设置依赖和解释失败与终端环境交互 (Merrill et al., 2026)。此类别的关键洞察是强结果验证器需要强任务实例化——测试只有在仓库状态、依赖和预期成功标准被精确指定时才能作为可靠评估者。

### 8.2.2 Web、浏览器与计算机使用任务

Web 和计算机使用基准将任务置于交互式环境中，成功由浏览器、桌面或应用状态变化定义。WebArena 引入了自主 agent 的真实 Web 环境 (Zhou et al., 2024)；VisualWebArena 添加多模态 Web 任务 (Koh et al., 2024)；BrowserGym 提供通用浏览器-agent 研究基底 (Chezelles et al., 2024)；WorkArena 聚焦于基于 ServiceNow 的知识工作 (Drouin et al., 2024)。OSWorld 进一步将评估扩展到涉及桌面应用、文件和多应用工作流的真实计算机环境 (Xie et al., 2024)。这些基准表明浏览器和计算机使用评估同时是评估问题和环境设计问题。

### 8.2.3 跨领域与企业工作流任务

第三组基准测试跨异构工具、环境和工作流的更广泛 agent 能力。AgentBench 跨操作系统、数据库、网页浏览和游戏评估 LLM agent (Liu et al., 2023a)；GAIA 针对需要推理、工具使用和多模态信息访问的通用助手任务 (Mialon et al., 2023)；The Agent Company 模拟职场式数字劳动 (Xu et al., 2024)。WorkArena 和 WorkArena++ 进一步强调企业工作流，成功取决于导航复杂软件而非回答孤立问题 (Drouin et al., 2024; Boisvert et al., 2024)。此基准族的主要贡献是覆盖度：它测试 harness 是否能跨任务类型、工具接口、状态表示、工作流和成功条件泛化。

## 8.3 阶段 2：执行前就绪验证

第二阶段询问评估是否可以公平且可复现地运行。此阶段在基准排行榜中通常不可见，但对 harness 评估至关重要。在 agent 开始之前，harness 必须验证环境、工具、上下文状态、权限边界、预算约束和评估器是否正确初始化。没有此就绪层，下游故障难以归因。

### 8.3.1 环境与沙箱就绪

环境就绪检查沙箱、仓库、浏览器、终端或 VM 是否从已知基线开始。在软件工程中，这包括仓库快照、依赖、任务设置和测试可执行性；在浏览器和计算机使用设置中，包括网站重置、浏览器配置文件、桌面状态和服务可用性。多个系统使此问题显式化：SWE-bench 依赖可执行仓库状态和基于测试的验证 (Jimenez et al., 2024)；Terminal-Bench 将终端任务与受控环境和验证逻辑打包 (Merrill et al., 2026)；OSWorld 使用真实计算机环境和基于执行的评估脚本 (Xie et al., 2024)。Repo2Run 通过为仓库自动化可执行 Docker 环境来针对依赖和环境设置 (Hu et al., 2025a)，HAL 强调 agent 的标准化、成本感知评估基础设施 (Kapoor et al., 2025)。这些系统共同表明环境重置和依赖验证是测量工具的一部分。

### 8.3.2 工具、上下文与权限就绪

就绪验证是跨层的。工具就绪检查 API、MCP 服务器、浏览器控制、shell 命令和文件操作是否可用且描述一致。上下文就绪检查历史、记忆存储、暂存本和检索文档是否被重置或有意初始化。权限就绪检查文件访问、凭证、网络访问和审批门是否匹配基准规格。

此阶段将评估与工具、上下文和治理层相连。如果工具描述在不同运行间变化，基准不再衡量相同的动作空间。如果记忆或上下文未重置，agent 可能受益于泄露的状态。如果权限过于严格，有能力的 agent 可能因治理原因失败；如果过于宽松，不安全或利用基准的行为可能未被检测。

### 8.3.3 评估器与评分器就绪

评估器本身也必须在执行前被验证。确定性评分器应检查不稳定性、缺失依赖和与环境状态的兼容性；LLM-as-Judge 提示、评分标准和评判模型应被版本化；人工审计协议应提前定义。评估器就绪因此是有意义故障归因的前提。

## 8.4 阶段 3：受控执行与追踪捕获

第三阶段询问执行期间发生了什么。在传统 LLM 评估中，证据可能由单个输入-输出对构成。在 agent 评估中，追踪是多步轨迹：模型观察状态、推理、调用工具、接收工具输出、修改环境、处理错误，最终终止。因此受控执行和追踪捕获是将基准运行转化为可诊断证据的必要条件。

### 8.4.1 受控展开与可复现执行

展开是 agent 评估的基本单位。它包括任务、模型配置、harness 配置、动作序列、中间观察、最终状态和评分结果。受控展开固定意外变化的关键来源，包括环境状态、工具可用性、超时、预算、权限策略和评估器版本。当非确定性存在时，重复展开可以暴露方差而非将其隐藏在单一分数之后。

多个系统说明了可复现执行的需要。SWE-agent 将 agent 暴露于仓库导航、文件编辑和测试执行作为交互循环的一部分 (Yang et al., 2024)。OpenHands 结合代码编辑、命令行执行、浏览器交互、沙箱执行和基准集成 (Wang et al., 2025b)。SWE-ReX 对本地和远程沙箱 shell 环境进行抽象，使执行后端可以在不改变 agent 逻辑的情况下变化 (SWE-agent Team, 2024)。LangChain 进一步将深度 agent 评估分解为单步验证、完整轮次评估和多轮模拟 (LangChain, 2025b)。这些系统共同表明可复现性需要重放完整的模型-工具-环境交互，而非仅重新运行模型调用。

### 8.4.2 追踪原生评估

追踪捕获将二元评分转化为因果诊断。追踪原生 harness 应记录模型输出、工具调用、工具结果、环境状态变化、上下文快照、错误、重试、恢复动作、token 用量、延迟和成本。这些追踪区分表面相似的故障：一个 agent 可能永远找不到相关文件，而另一个可能找到但产生无效补丁。它们还能揭示不良成功，如基准利用、过多工具调用或权限违规。

HAL 将日志和追踪视为标准化 agent 评估的一等工件，使用大规模 agent 日志检查最终分数可能隐藏的行为 (Kapoor et al., 2025)。R2E-Gym 类似地将可执行轨迹和混合验证器与测试时评估和训练时反馈关联 (Jain et al., 2025)。对 harness 工程而言，追踪因此不是辅助调试工件；它们是主要评估数据。

### 8.4.3 成本、延迟与资源追踪

Agent 评估应不仅报告质量，还报告达成质量的成本。长周期 agent 可能通过花费更多 token、调用更多工具、重试更多次或使用更强模型来提高成功率。Harness 级评估因此应追踪 token 用量、模型成本、挂钟延迟、工具调用计数、重试计数和资源消耗。评估应暴露成功-成本-延迟前沿，而非仅按成功率排名 agent。这对部署的 agent 服务尤为重要，其中优化单位是预算和延迟约束下的循环工作负载。

## 8.5 阶段 4：多级判断与故障归因

受控执行和追踪捕获之后，核心问题是运行应如何被判断，以及如果失败，故障应如何定位。我们定义阶段 4 为多级判断和故障归因的组合过程。多级判断在三个层面评估运行：最终结果是否正确，轨迹是否高效且策略合规，以及评估器本身是否可靠。故障归因然后使用这些信号诊断故障最可能源自何处——模型、工具接口、上下文管理器、执行环境、编排循环、基准规格或评估器。此阶段区分了 harness 评估与普通基准评估：目标不仅是分配分数，而是将执行证据转化为可操作的工程诊断。

### 8.5.1 结果级评估

结果级评估衡量最终任务目标是否达成。在软件工程中，这通常通过单元测试、回归测试或 issue 特定验证来操作化（如 SWE-bench）。其优势是可扩展性和可解释性：它提供清晰的任务级成功信号并支持排行榜比较。其局限是压缩——将完整执行情节缩减为一个值，隐藏了 agent 是否稳健、安全或高效地成功。结果评估因此是必要但不充分的。

### 8.5.2 轨迹级评估

轨迹级评估衡量 agent 所走路径的质量。它询问 agent 是否选择了适当的工具、合理排序动作、避免冗余调用、尊重权限边界、从错误中恢复并随时间保持上下文一致性。此层面是 agent 评估与传统输出评估最明显分歧之处：最终答案可以正确而轨迹仍不可接受。

轨迹级判断对 harness 工程特别有价值，因为它提供了从故障到修复的路径。如果 agent 因选择错误工具而失败，工具接口可能需要改进。如果忘记先前约束，上下文层可能需要调整。如果无恢复地循环，编排层可能需要生命周期控制。因此，轨迹评估将基准结果转化为层级特定的工程反馈。

### 8.5.3 评估器级评估

评估器级评估询问评估器本身是否可信。确定性验证器（如单元测试或基于状态的检查器）可复现、廉价且易于跨系统比较，但对开放式任务范围窄且难以设计。LLM-as-Judge 系统对自然语言输出和轨迹评估灵活，但引入偏差、非确定性和额外成本。人工审计对模糊或安全关键案例仍然重要，但无法扩展到持续评估。

G-Eval 表明基于 LLM 的评估器在 NLG 评估任务上能更好地与人类判断对齐，同时也突出了对 LLM 生成文本的偏好偏差 (Liu et al., 2023b)。MT-Bench 和 Chatbot Arena 系统研究了 LLM-as-Judge 并发现位置偏差、冗长偏差、自我增强偏差和有限推理能力 (Zheng et al., 2023)。对 agent harness 而言，评估器级评估不是可选的——如果评分器不稳定、测试非确定性或 LLM 评判有偏差，评估噪声可能被错误归因于 agent 或模型。稳健的 harness 因此应偏好分层评分器。

### 8.5.4 跨 Harness 层的故障归因

故障归因识别运行被判断后应修复什么。失败的展开可能源自模型推理、工具接口设计、陈旧或压缩的上下文、沙箱不稳定性、编排循环、基准歧义或评估器不可靠 (Kapoor et al., 2025; Hu et al., 2025a; Jimenez et al., 2024; LangChain, 2025b)。实践中归因很少是单标签分类问题——模糊的任务规格可能导致 agent 选错工具；过度压缩的上下文可能导致忘记约束；沙箱依赖问题可能触发增加成本的恢复行为；不稳定的评判可能遮蔽有效方案。因此，故障归因应被理解为对完整追踪的诊断过程，而非附加到最终分数的事后标签。

## 8.6 阶段 5：持续回归与部署反馈

最终阶段询问评估结果如何驱动下一次 harness 修订。Agent harness 持续演进：prompt、工具 schema、MCP 服务器、上下文策略、沙箱镜像、编排循环、治理规则和评判 prompt 都可能随时间变化。持续评估将基准结果和生产故障转化为 harness 本身的回归系统。

### 8.6.1 Harness 变更的回归评估

回归评估应由 harness 变更和模型变更触发。工具描述、上下文压缩策略、沙箱镜像、权限规则或评判 prompt 的变更都可能改变 agent 行为或评估分数。由于 harness 组件存在交互，局部改进可能创建全局回归。实际模式是维护分层评估套件：工具 schema 和确定性验证器的类单元测试、局部决策的单步测试、端到端完成的完整展开测试、以及长周期连贯性的多轮模拟。

### 8.6.2 评估框架与 LLMOps 集成

通用评估框架提供持续测试的可复用基础设施。Promptfoo 支持 prompt 和 LLM 应用回归测试；DeepEval 提供类单元测试抽象；RAGAS 聚焦于检索增强生成评估；lm-evaluation-harness 支持标准化语言模型评估 (promptfoo contributors, 2026; Confident AI, 2026; Es et al., 2024; Biderman et al., 2024; Gao et al., 2021)。评估还应与可观测性连接——生产追踪可以成为回归测试，评估失败可以成为可观测性信号。此关联闭合了监控已发生事件和构建受控测试来复现和诊断它之间的循环。

### 8.6.3 基于验证器和训练时评估

更新方向不仅将评估器用作离线指标，还用作训练信号和优化目标。基于验证器的环境和 RL 风格 agent gym，包括 R2E-Gym 和 verifiers，使用环境反馈作为改进 agent 策略和脚手架的奖励或验证信号 (Jain et al., 2025; Prime Intellect, 2026)。这将评估从事后测量步骤转变为 agent 训练、测试时适应和脚手架选择的主动组件。

Meta-Harness 通过将 harness 设计本身作为自动搜索对象来扩展此方向 (Lee et al., 2026)。此视角不仅评估哪个模型在固定 harness 下表现最好，还询问哪个 harness 结构、提示策略、工具接口或控制循环能产生更可靠的 agent 行为。在此框架下，评估不是管线的终点；它是允许 harness 改进的信号。

## 8.7 总结

验证与评估是将 agent 行为转化为工程证据的层。我们将此层组织为任务到反馈的生命周期：阶段 1 将任务实例化于真实环境和成功标准中；阶段 2 验证评估设置就绪、公平且可复现；阶段 3 运行受控展开并捕获追踪；阶段 4 在结果、轨迹和评估器三个层面判断每次运行；阶段 5 将结果反馈回持续回归测试和 harness 改进。

此生命周期将评估从排行榜机制重新定义为 agent harness 的质量控制循环。最终成功率仍有用但不再充分。对长期运行的 agent 而言，核心评估问题不仅是 agent 是否成功，还有为什么成功或失败、路径是否可接受、评估器是否可信、以及下一步应改进哪个 harness 组件。

# 9 治理与安全 (G)

此层处理如何约束、保证 agent 行为的安全性并使其可问责。LLM agent 现在执行 shell 命令、发送邮件、提交代码和调用第三方 API；对于生产部署，核心问题是它们应在什么约束下行动，以及当这些约束失败时谁承担问责。在生产系统中，治理有其自己的工具生态系统，包括权限引擎、策略语言、审计管线和网关控制。此生态系统与第 6、7 节涵盖的生命周期钩子和可观测性基础设施不同，值得作为 ETCLOVG 分类法中的独立层处理。我们围绕五种机制组织讨论：权限模型与身份管理（§9.1）、生命周期钩子（§9.2）、组件加固（§9.3）、声明式宪法（§9.4）和审计基础设施（§9.5）。然后将这些机制置于更广泛的 agent 安全格局中（§9.6），并以开放研究方向结束（§9.7）。

## 9.1 权限模型与身份管理

任何 agent harness 的首个治理问题是访问控制：agent 可以触达哪些工具、文件、网络端点和系统资源？Agent 工作负载使这比传统 RBAC 或 ABAC 更难，因为所需工具集通常取决于部署时未知的自然语言任务。因此文献沿粒度轴发展：部署时固定的静态边界、按工具调用评估的上下文策略、以及用于本地 harness 之外交互的跨边界权限。

**静态权限边界。** 生产编码 agent 通常预定义权限范围并要求超出范围动作的升级。Codex (OpenAI, 2025) 在具有受限文件系统和网络访问的沙箱中运行 shell 命令，Gemini CLI (Mullen & Salva, 2025) 结合工作空间作用域的文件访问与命令允许和拒绝列表。这些边界易于检查，但无法表达任务特定意图。

**上下文相关的特权控制。** Progent (Shi et al., 2025a) 引入领域特定语言（DSL），其谓词覆盖工具名称、参数和环境状态。运行时在每次调用前评估这些谓词，允许最小权限策略按角色、用户或会话变化。Conseca (Tsai & Bagdasarian, 2025) 进一步通过从可信上下文生成任务特定策略并用独立确定性检查器执行来推进此想法。策略生成和执行之间的分离很重要：生成组件可适应任务，而执行器保持可审计。

**身份管理与 agent 间访问控制。** 在授予访问之前，系统必须确定谁在请求。South et al. (2025) 论证 agent 需要认证委托：扩展 OAuth 2.0 和 OpenID Connect 的框架，使用用户 ID Token、Agent ID Token 和作用域委托 Token 将用户意图绑定到 agent 特定权限。此 token 链将创建从人类委托人到 agent 动作的可验证问责追踪。SAGA (Syros et al., 2025) 通过提供者中介架构为多 agent 设置构建此原则——agent 使用加密一次性密钥注册，短期访问 token 在 agent 之间执行用户定义的联系策略。IsolateGPT (Wu et al., 2025) 采用架构方法，采用中心辐射模型，每个第三方应用在隔离的辐射中执行，具有自己的 LLM、记忆和工具访问。辐射间通信通过可信中心传递，使跨应用数据交换成为显式治理决策而非共享上下文窗口的偶然副作用。

**凭证管理。** Agent 常规处理 API 密钥、会话 token 和一次性密码。将这些凭证暴露于 LLM 上下文会产生泄露风险。Skyvern (Skyvern-AI, 2025) 说明了常见模式：在保险库中存储秘密，仅向 LLM 暴露占位符，仅在执行认证动作的自动化层替换原始值。未解决的问题是长周期会话中的秘密生命周期管理，token 可能在轨迹中过期或被吊销，而更新的凭证仍必须保持在模型上下文之外。

**Web 级权限协调。** 上述机制在单个 harness 信任边界内运行。跨组织交互——agent 遍历第三方网站或调用其他方拥有的服务——需要单个 harness 无法独立执行的协调。Marro et al. (2025) 提出 agent-permissions.json，一个类似 robots.txt 的轻量清单文件，网站所有者可声明 agent 可使用哪些 UI 元素、速率限制、并发约束和需要人类确认的动作。

**开放挑战。** 设计既有表达力又可用的权限模型是未解决的问题。过度限制的策略降低 agent 效用；过度宽松的策略否定治理目的。社区尚未收敛于可跨 harness 实现移植的标准权限规格语言。

## 9.2 生命周期钩子

权限模型定义什么是被允许的。生命周期钩子定义策略检查何时触发。许多 harness 在 agent 循环的每个阶段暴露钩子点：规划前、工具调用前、工具执行后和响应交付前。这些钩子允许治理逻辑注入而不修改 agent 的核心推理。

**预执行钩子：输入护栏。** 输入护栏在数据到达 LLM 之前进行验证。PromptShield (Jacob et al., 2024) 和 DataSentinel (Liu et al., 2025) 训练专用分类器以检测用户输入和检索内容中的 prompt 注入载荷。基于规则的检测器严格但脆弱；基于模型的检测器泛化更好但更慢且易受自适应攻击 (Andriushchenko et al., 2024; Nasr et al., 2025)。

**预调用钩子：输出护栏与动作验证。** 在执行工具调用前，输出护栏检查 LLM 的提议动作。ShieldAgent (Chen et al., 2025b) 将安全约束形式化为可验证谓词并对每个动作进行检查。在多 agent 系统中，ControlValve (Jha et al., 2025) 约束 agent 间的控制流图，防止未授权的 agent 转换，解决一个 agent 将执行重定向到另一个 agent 的控制流劫持攻击。

**后执行钩子：信息流控制与污点追踪。** CaMeL (Debenedetti et al., 2025) 实现基于能力的信息流控制。每个数据值携带追踪其来源的元数据标签，区分可信用户输入和不可信 Web 检索。自定义解释器确保不可信数据不能影响控制流决策。CaMeL 分离对可信上下文的规划和对不可信内容的处理，以灵活性换取更强的控制/数据流边界。

**人在环路钩子。** 多个广泛使用的编码 agent（包括 Codex、Gemini CLI、Cursor 和 OpenHands）将破坏性或超范围动作门控于显式用户批准。三个设计维度塑造这些钩子：验证范围（哪些动作需要批准）、警报丰富度（用户看到多少上下文）和重现策略（仅允许一次 vs 永久允许）。Felt et al. (2012) 表明只有 17% 的 Android 用户在应用安装期间注意权限对话框，只有 3% 正确回答了关于所授权限的理解问题；agent 批准对话框可能面临类似的习惯化和理解风险。

**可编程执行基底。** AgentSpec (Wang et al., 2026) 在计划、工具调用和响应阶段定义带触发器、谓词和执行动作的规则。AgentDoG (Liu et al., 2026a) 从单动作分类转向轨迹级诊断，按来源、故障模式和后果组织风险。

**开放挑战。** 当前系统中钩子 API 仍然异构。我们不知道有广泛采用的标准定义第三方治理模块的通用钩子接口。钩子还引入延迟，堆叠钩子之间的交互效应在很大程度上未被研究。

## 9.3 组件加固

生命周期钩子在 harness 层面执行治理。组件加固强化单个 agent 组件——特别是 LLM 和它调用的工具——对抗其特定漏洞。Harness 受益于加固的组件，因为更少的恶意输入能存活到触发治理钩子。

**模型加固。** Prompt 注入的根因是 LLM 以同等优先级处理所有输入文本，使其无法区分系统指令和不可信数据。Wallace et al. (2024) 提出指令层次结构，训练模型优先处理特权指令（系统提示）而非低特权指令（用户消息、工具输出）。模型学习在低级指令与高级约束一致时与之对齐，在冲突时忽略或拒绝。SecAlign (Chen et al., 2025a) 将相同防御目标表述为对 prompt 注入输入、期望响应和不期望响应的偏好优化。

模型级防御补充 harness 级钩子。加固的模型在推理时拒绝更多注入，降低下游护栏的负载。然而，模型加固单独不够：它解决了 Kim et al. 分类法中的"错误指令遵循"但不能防止无约束数据流或未授权动作，这需要系统级执行 (Kim et al., 2026; Wei et al., 2026)。

**基于分类器的运行时加固。** 互补工作线通过部署小型辅助分类器在运行时筛选输入和输出来加固模型边界，而不修改 agent LLM 本身。Llama Guard (Inan et al., 2023) 微调独立模型对用户提示和助手响应按可配置安全分类法进行分类，返回 harness 可路由到允许、阻止或升级决策的每类别标签。此类分类器提供训练时加固缺乏的两个实用属性：分类法可在不重训 agent 模型的情况下编辑，同一检测器可跨异构 LLM 后端复用。

**工具加固与 MCP 安全。** Model Context Protocol (MCP) (Anthropic, 2024c) 已成为 agent 与工具之间的通用接口，但其初始规范缺乏原生安全原语。此缺口吸引了攻击和防御。Radosevich & Halloran (2025) 证明广泛使用的商业 LLM 可被胁迫使用 MCP 工具执行恶意代码、建立远程访问和窃取凭证。Trail of Bits (2025) 表明 MCP 服务器可在用户调用工具之前通过毒化工具描述攻击客户端。防御方面，ETDI (Bhatt et al., 2025) 用加密签名和版本化的工具定义扩展 MCP，确保对工具代码、schema 或权限的任何修改都需要新的签名版本。

**协议级加固。** SAFEFLOW (Li et al., 2025) 为多 agent 系统引入协议级执行，结合细粒度信息流控制与事务性执行语义，使违规工具调用或 agent 间消息可以回滚而非通过共享状态传播。

**供应链风险。** Agent 治理还必须解决 agent 调用的工具和包。Spracklen et al. (2025) 分析 16 个 LLM 的 576,000 个代码样本，发现开源模型以 21.7% 的比率幻觉不存在的包名。攻击者可在公共仓库注册这些幻觉名称并注入恶意代码——被称为"slopsquatting"的技术。

## 9.4 声明式宪法

将治理逻辑直接嵌入应用代码使策略不透明、难以审计且难以更新。越来越多的 harness 将治理规则外部化为声明式配置文件，最常见的是 YAML 格式。这些文件作为 agent 的机器可读宪法。

**训练时宪法：Anthropic 的 Constitutional AI。** Anthropic (2026a) 发布组织为四级优先层次的宪法：安全（保持人类监督）、伦理（诚实、避免伤害）、合规（Anthropic 的指导方针）和有用性（用户请求）。宪法区分硬约束（如对 CBRN 辅助的绝对禁止）和运营商可在定义边界内调整的软编码默认值。在 agent 场景中，此结构建议一个委托人层次，其中 Anthropic 的策略覆盖运营商配置，后者又覆盖终端用户偏好。

**部署时宪法：基于 YAML 的治理。** AutoHarness 仓库 (AIMing Lab, 2026) 在 YAML 文件中编码治理规则，指定管线模式（核心、标准或增强）、风险分类模式、允许和拒绝的工具模式、token 预算限制和审计日志目的地。此声明式风格将治理意图与执行逻辑分离。安全团队和合规官等非开发者利益相关者可在不触碰 agent 代码的情况下审查和修改策略。与训练时宪法不同，YAML 文件可以用标准工具更新、版本化和 diff。

**可编程策略语言。** Progent 的策略语言 (Shi et al., 2025a) 支持布尔谓词、量词和环境引用。Formal-LLM (Li et al., 2024) 更进一步，将计划约束编码为下推自动机。此形式化可以表达顺序排序约束、禁止的工具组合和特定步骤的必需批准门。VeriSafeAgent (Lee et al., 2025) 将用户意图形式化为 UI 状态转换上的 DSL，在执行前验证提议的 GUI 动作是否与用户任务一致。

**开放挑战。** 目前不存在广泛采用的 agent 宪法标准 schema。每个 harness 倾向于定义自己的 YAML 结构，使策略不可移植。训练时和部署时宪法之间的交互特别未被充分探索——YAML 拒绝规则是否能可靠覆盖 RLHF 已强化的行为，以及在什么条件下两层冲突，仍是具有实际安全影响的开放问题。

## 9.5 审计基础设施

治理需要问责。审计基础设施记录 agent 做了什么、为什么这样做以及治理策略是否被尊重。这些记录支持事后调查、法规合规和持续策略改进。

**结构化审计追踪。** AutoHarness (AIMing Lab, 2026) 发出每工具调用的 JSONL 记录，包括工具名称、参数、风险分类、权限决策、执行结果、token 成本和挂钟延迟。这些结构化日志支持实时面板和离线取证分析。SAGA (Syros et al., 2025) 通过记录 agent 之间交换的加密 token 将审计追踪扩展到多 agent 交互，实现跨信任边界的溯源追踪。可重放的审计记录至少需要追踪标识符、委托人身份、工具调用、策略决策和版本、执行结果、资源成本以及对相关输入和输出的完整性哈希。大多数当前系统仅记录这些字段的子集，很少对记录签名或哈希，使审计追踪易受受损 agent 进程的日志篡改。

**异常检测：按动作 vs 轨迹级别。** 检测机制按评估证据的粒度分裂。按动作检测器针对学习或指定的风险概念独立分类每个工具调用。轨迹级检测器联合评估动作序列——AgentAuditor (Luo et al., 2025) 结合行为模式匹配与基于 LLM 的完整追踪推理；SentinelAgent (He et al., 2025) 将多 agent 通信建模为时序图并标记异常交互模式。轨迹级检测以更高延迟和不太精确定位哪个动作触发警报为代价捕获多步攻击。

**成本与资源审计。** 失控的 agent 循环可能在循环密集工作负载中快速耗尽 API 预算。2025 OWASP LLM Top 10 (OWASP Foundation, 2025) 将资源耗尽列为独立风险类别。AutoHarness 追踪每调用 token 消耗并通过宪法声明式执行会话级预算。成本感知治理在多 agent 架构中尤为重要，其中扇出模式可以指数级增加 API 调用。

**分层治理管线。** 新兴设计模式将前述机制（权限、钩子、宪法和审计）整合为单一可配置管线。AutoHarness 以三个层级示范此模式——层级渐进添加更深检查，从解析-风险-权限-执行-审计循环到上下文丰富、输出验证、异常评分、人工升级和形式化约束验证。层级通过 YAML 宪法声明式选择，使治理开销随部署风险缩放。

## 9.6 将治理置于 Agent 安全格局中

治理机制响应具体的威胁格局。近期证据 (Zhang et al., 2026b; Kim et al., 2026; Chen et al., 2026; Wei et al., 2026) 来自开放 agent 对 agent 平台进一步表明 agent 安全不限于 prompt 注入或单 agent 越狱：社会工程、协调的 agent 集群、凭证泄露和平台级参与放大可以共同塑造威胁格局。Kim et al. (2026) 调查 128 篇论文并编目了 agent 栈中 51 种攻击方法和 60 种防御方法。Chen et al. (2026) 通过六维分类法综合 50 篇论文互补此工作，提出安全可构建 agent 平台的参考学说。

**从设计维度到治理机制。** Kim et al. 确定七个设计维度（输入信任、访问敏感性、工作流、动作、记忆、工具、用户界面），每个代表灵活性谱。更大灵活性扩展攻击面，治理在运行时约束该灵活性。权限模型约束工具和动作；生命周期钩子向工作流注入检查点；宪法外部化约束；审计基础设施提供揭示约束过松或过紧的反馈循环。

**实际 agent 中的防御缺口。** Kim et al. 对六个 agent 系统（Codex、Gemini CLI、OpenHands、Browser Use、Nanobrowser、Skyvern）的案例研究揭示没有 agent 完全实现所有防御类别。信息流控制、身份管理和形式化验证在每个被调查系统中都缺失。在所有六种情况下，监控是部分的：agent 记录工具调用但缺乏自动化异常检测。

**纵深防御及其局限。** Kim et al. 论证 agent 安全需要相互补充的分层防御：输入护栏作为第一道防线，输出护栏作为最后防线，信息流控制和监控作为持续运行时保护，访问控制用于认证和授权，人在环路用于关键决策。然而分层并非免费；协调不良的层可能相互干扰，独立治理钩子的可组合性在很大程度上未被测试。

## 9.7 研究方向

表 4 总结了本节讨论系统中的治理状态，揭示了覆盖度的稀疏性，表明治理基础设施通常是安全部署的实际瓶颈，与模型局限性并列。我们确定了八个开放研究方向：

**(1) 标准化策略和审计语言。** 每个 harness 定义自己的 YAML schema 和专有 DSL，碎片化治理生态系统。社区驱动的规范将实现可移植策略、可组合治理模块和跨 harness 审计互操作性。

**(2) 形式化治理保证。** 当前治理机制在形式化保证方面仍有限。Agent 行为的形式化验证处于早期阶段。扩展这些技术以验证治理管线正确性仍未被充分探索。

**(3) 自适应治理。** 静态策略无法预见每个任务上下文。LLM 生成的策略可以弥合此缺口——但机器生成的治理规则的可信度，以及此类策略生成器本身应如何被治理，仍然开放。

**(4) 长周期 agent 的治理。** 执行多小时或多天任务的 agent 引入上下文漂移、跨会话状态和演变的信任假设。许多当前治理管线为单轮或短会话 agent 设计。将其扩展到长周期自主需要有效性可以续期、吊销和跨演变会话审计的治理规则。

**(5) 可用的治理界面。** 可用的治理界面是部署的必要条件。该领域需要权限 UI、审计面板和宪法编辑器的人机交互 (HCI) 研究，使非专业用户可访问。

**(6) 跨层治理一致性。** 训练时对齐（constitutional AI）、部署时配置（YAML 宪法）和运行时执行（生命周期钩子）在不同层以不同保真度运行。它们如何组合，以及运行时治理是否能可靠覆盖训练时倾向，仍是开放问题。

**(7) 端到端供应链治理。** ETDI 等 MCP 安全扩展解决单个 harness 内的工具完整性。然而 agent 还依赖于难以验证来源的外部包、数据集和检索源。跨越完整工具和数据供应链的治理框架仍然有限。

**(8) 治理栈的统一对抗基准。** 现有基准针对窄威胁类别。没有一个在通用对手模型下用可复现攻击追踪评估完整治理栈。社区基准需要联合报告防御有效性、误报率和开销。
# 10 跨层关注点

本节汇集的关注点横跨多个 ETCLOVG 层，并通过展示层级特定推理在何处失效来发展论断 1（agent harness 作为独立系统层，§1）。各层章节分别隔离了执行、工具、上下文、生命周期、可观测性、评估和治理，使每个设计面都能被精确描述。然而，生产环境中的 harness 最常在这些设计面之间的接口处失败。

**层间交互模式。** 七个层并非相互独立。执行环境（E）约束了哪些生命周期和编排策略（L）是可行的；上下文管理（C）影响评估的可复现性（V）；治理（G）施加的身份、权限和审计约束则跨越所有其他层。因此，harness 设计应被理解为一种依赖结构，而非可分离组件的清单。

**成本-质量-速度三难困境。** 更强的评估（V）、更严格的治理（G）、更丰富的可观测性（O）以及更高保真的执行环境（E）通常会增加成本和延迟。优化速度往往会降低诊断深度或安全裕度，而优化质量则可能使迭代成本过高以至于无法常规使用。实际系统必须选择哪些检查是同步的、哪些离线运行、以及哪些故障值得启用昂贵的恢复路径。

**标准化与生态系统动态。** MCP、ACP 和 A2A 等协议反映了工具、agent 和编排状态向共享接口发展的压力。这种压力有利于 agent 无关的基础设施而非单一 agent 集成，但它也将责任转移到治理和可观测性上：标准化的工具调用只有在周围的 harness 能够跨系统保持来源追溯、权限、成本和故障证据时才有用。

**持续存在的生态系统缺口。** 在整个文献语料中，五个缺口反复出现在层边界处：跨工具互操作性、成本归因、故障恢复、多仓库编排和人机交接。这些缺口激发了 §11 中的综合分析：核心问题不再是每个层是否有可用工具，而是组合后的 harness 是否表现为一个可靠的控制系统。

# 11 跨层综合

本节将 §10 的跨层关注点整合为五个系统级效应，是论断 1（§1）的核心支撑。其目的是将讨论从组件覆盖度转向 harness 行为：一旦执行、工具、上下文、编排、可观测性、评估和治理被组合在一起，它们的交互就会产生任何单一层都无法独自解决的约束。

## 11.1 成本-质量-速度三难困境

Harness 的可靠性受到成本、质量和速度之间三方权衡的约束。更强的沙箱和更高保真的环境提升了安全性和可复现性，但增加了启动延迟和基础设施成本；更丰富的上下文和记忆策略可以改善任务连续性，但消耗 token 并引入检索开销；更深入的评估和可观测性改善了诊断能力，但减慢了迭代速度并增加了存储、标注和 trace 处理成本。因此，生产系统不能将质量视为标量目标。它们必须决定哪些风险值得昂贵的控制措施、哪些检查可以异步运行或在回归测试套件中运行、以及在 agent 生命周期的每个阶段哪些遥测数据值得捕获。

## 11.2 能力-控制权衡

更强大的 harness 向 agent 暴露更多权限，但权限的每一次增加都扩大了控制问题。更大的工具菜单拓宽了任务覆盖范围，同时增加了选择错误和 prompt injection 攻击面；持久化记忆有助于长期运行的任务，但带来了来源追溯、过时性和隐私风险；宽松的沙箱使自主执行变得有用，但也扩大了错位或被入侵行为的爆炸半径。因此，能力-控制权衡不是一个附加在功能系统之上的安全补丁，而是一个连接工具 schema、上下文策略、运行时权限、身份、可审计性和人工审批的设计轴。

## 11.3 Harness 耦合问题

Harness 各层以使局部优化变得脆弱的方式相互耦合。执行环境通过影响包可用性、重置语义、延迟和故障模式来改变评估结果；工具描述消耗上下文预算并塑造模型行为；可观测性 trace 只有在身份和权限状态以相同粒度被捕获时才能成为治理证据；评估设计通过奖励某些恢复循环和惩罚其他循环来反馈到编排中。这些耦合意味着 harness 变更应作为系统变更来测试。一个 prompt、工具、记忆、沙箱、验证器或监控器在隔离状态下可能看起来有益，但当与控制循环的其余部分组合时可能会降低整体 rollout 的质量。耦合问题还解释了为什么在不指定周围控制器的情况下，agent 分数无法被干净地归因于模型。在闭环框架下，对上下文策略、工具 schema、验证器或恢复循环的更改会改变控制器 C_H，从而改变同一模型的测量行为 (Bölük, 2026b)。

## 11.4 从 Agent 框架到 Agent 平台

生态系统正在从 agent 框架向 agent 平台演进。框架封装了本地抽象，如 agent、工具、记忆存储和执行循环；平台则增加了持久化工作空间、托管沙箱、身份、计费、可观测性、评估、治理以及跨多次运行和多用户的人机交接。这一转变之所以重要，是因为长期运行的 agent 不再仅仅是调用模型的程序。它们是需要租户管理、合规性、故障恢复、trace 保留和组织所有权的运营系统。因此，核心设计问题从"我如何构建一个 agent？"转变为"我如何运营一个 agent 集群，使其行为在时间维度上保持可检查和可逆？"

## 11.5 开放研究议程

综合分析指向一个以 harness 作为自适应控制系统为中心的研究议程。该领域需要：变化 harness 干预而非仅变化模型权重的基准测试、用于跨层归因故障的 trace 原生方法、在 agent、工具、沙箱、评估器和人类之间传递状态和责任的协议，以及随着模型改进而简化 harness 的优化方法。下一节将这些跨层效应转化为五个开放问题：如何加固和扩展执行环境、维护可靠状态、从 trace 中诊断故障、标准化交接，以及在模型能力变化时保持 harness 的有用性。

# 12 开放问题与未来方向

此处汇集的开放问题源自约束瓶颈论题和跨层综合分析，构成论断 1（§1）证据的前瞻性部分。本节不将七个 ETCLOVG 层视为独立的组件列表，而是追问整个 harness 在何处仍然缺乏科学上的精确规范。核心模式是：agent harness 正在成为长期运行的控制系统，但该领域仍然缺乏关于加固执行基底、保持状态、诊断故障、转移责任以及随模型能力变化更新 harness 的成熟答案。我们将这些缺口组织为五个横跨分类体系的问题。

## 12.1 加固和扩展执行环境

执行环境正在成为安全性、可扩展性和可移植性交汇的控制边界。SandboxEscapeBench 记录了前沿模型在现实配置下可以利用沙箱弱点 (Marchand et al., 2026)，但防御工作仍然分散在具有不同威胁模型和评估协议的系统中 (Wu et al., 2025; Yan, 2025)。与此同时，每任务一容器的模式给大规模训练和评估带来压力，其中数万个并行轨迹需要廉价的重置和重放；SWE-World 指向无 Docker 的代理环境，但学习到的转换相对于真实执行的保真度仍未解决 (Sun et al., 2026)。即使部署可移植性也不是一个已解决的工程细节：基于 Docker 的沙箱继承了 Linux 内核假设，而 macOS、Windows、浏览器、桌面和混合云设置暴露了不同的隔离和可复现性约束。

开放问题是使运行时基底既可测量又可组合。未来的 harness 需要：针对 prompt injection、目标错位和组合放大的通用安全评估；决定何时使用容器、microVM、操作系统级权限边界、完整桌面虚拟机、浏览器环境或学习代理的成本模型；以及在自托管、云和混合部署之间保持语义的可移植性层。框架集成运行时（§3.2.4）和沙箱抽象（§3.2.7）之间的捆绑与组合分裂应被视为经验性设计问题而非产品偏好。MCP 等标准可能降低组合成本，但前提是工具、治理和可观测性层暴露足够的状态以保持运行时选择的可审计性、可恢复性和安全性。

## 12.2 在长期运行的 Agent 中维护可靠状态

最深层的上下文问题不仅仅是如何将更多 token 放入 prompt 中，而是如何在长时间跨度内保持 agent 的工作状态与真实任务状态对齐。长期运行的编码、研究和运维 agent 反复进行总结、检索、压缩和外部化信息；每一次这样的操作都可能删除约束、扭曲优先级或保留过时的假设。近期的上下文工程工作将压缩、工具结果清除、检索和 prompt 缓存感知排序视为管理有限上下文窗口的实用机制 (Anthropic Applied AI Team, 2025; Anthropic, 2025c; OpenAI, 2026b)。然而，上下文腐化和记忆基准测试表明，更长的输入和更丰富的记忆存储并不自动意味着更好的任务状态跟踪 (Hong et al., 2025; Tan et al., 2025; He et al., 2026)。

因此，一个有原则的研究议程应将上下文管理重新定义为状态估计。开放问题是：我们能否刻画每次压缩、检索或遗忘步骤中丢失了多少任务相关信息，以及我们能否约束 agent 内部状态与任务真实状态之间的偏差。近期综述将 agent 记忆形式化为写入-管理-读取循环，并将策略学习管理识别为一个新兴机制族 (Zhang et al., 2025; Du, 2026)。未来系统需要：不确定性感知的摘要、已记忆事实的来源追溯、矛盾处理、显式过时标记，以及让 agent 从持久化制品而非信任其自身压缩历史来重建缺失状态的恢复程序。这也暗示了记忆与评估之间更紧密的联系：记忆策略不仅应通过召回准确率来评判，还应通过它们是否能防止多会话任务中的下游行动错误来评判。

## 12.3 从 Agent Trace 中诊断故障

Agent 评估仍然过于以最终分数为中心：一次运行通过或失败，最终数字被视为关于模型质量的证据。对于 harness 工程而言，这是不够的，因为一次失败的 rollout 可能源自模型推理、误导性的工具 schema、沙箱配置错误、过时的上下文、不稳定的测试、基准歧义、评判器不稳定或编排循环。Anthropic 对 agentic 编码评估的分析表明，基础设施设置可以可测量地改变基准分数 (Anthropic, 2026a)，近期关于 agentic 评估中随机性的工作认为单次运行通过率可能隐藏了大量方差 (Bjarnason et al., 2026)。因此，评估层必须作为测量仪器来研究，而不仅仅用作排行榜生成器。

下一步是 trace 原生评估：trace 应成为系统计算结果分数、轨迹质量、故障归因和回归测试的主要对象。可观测性系统已经捕获了 span、工具调用、成本、重试、异常和中间消息 (OpenTelemetry, 2026; AlSayyad et al., 2026; Koc et al., 2025)，但这些 trace 通常与评估管道断开。LangChain 2026 年的调查报告称 89% 的团队使用可观测性，而只有 52.4% 运行离线评估 (LangChain, 2026a)；这一差距意味着团队可以看到 agent 做了什么，但无法系统地判断行为是否正确。未来工作应通过将异常生产 trace 转换为回归用例、直接在 span 上计算轨迹指标，以及将诊断信号反馈到 prompt、工具、上下文和编排变更中来弥合这一循环。Reflexion 表明 agent 可以在短期设置中从自身 trace 中学习 (Shinn et al., 2023)；将这一思想扩展到长期运行的多会话 harness 仍是开放问题。

## 12.4 Agent、工具和人类之间的标准交接

现代 harness 越来越多地将工作分配给规划器、子 agent、工具、沙箱、评估器和人类，但这些参与者之间的接口仍然是临时性的。已有一些新兴的局部标准：MCP 标准化了工具访问，A2A 针对 agent 间通信，OpenTelemetry 提供了通用的 trace 基底 (Model Context Protocol, 2025b; A2A Project, 2025; OpenTelemetry, 2026; Ehtesham et al., 2025)。缺失的是跨层交接契约。当规划器将工作交给执行器、agent 调用工具、子 agent 返回控制权或系统升级到人工时，交接应传递的不仅是文本摘要，还包括意图、约束、权限、制品、来源追溯、预算状态、风险等级、trace 历史和未解决的决策。

这个问题部分是技术性的，部分是制度性的。OpenAI 的 Symphony 将 issue tracker 和代码仓库框定为 agent 工作的控制平面，而 Anthropic 的长期运行 agent harness 强调持久化进度制品和干净的交接状态 (Kotliarskyi et al., 2026; Anthropic, 2025d; 2026b)。治理工作从相反方向得出相同结论：在 agent 能够安全地代表用户跨系统行动之前，需要 agent 身份、委托、权限清单和可审计性 (South et al., 2025; Marro et al., 2025; Syros et al., 2025)。开放问题是定义足够丰富以支持安全和恢复、但又足够简单以获得广泛采用的交接协议。此类协议应使责任明确：谁授权了该行动、传递了哪些状态、哪些证据支持当前计划、接收方被允许做什么、以及控制权何时必须返回给另一个 agent 或人类。

## 12.5 在模型改进时保持 Harness 的有用性

不应假设 harness 设计会单调地朝着更多脚手架的方向发展。每一个包装器、重置、验证器、规划器、记忆规则和权限门都编码了关于模型无法独立可靠完成什么的假设。随着模型能力的变化，harness 干预应被重新评估而非假定仍然有益。阶乘式的模型×harness 评估可以揭示一个干预何时改善所有模型、仅帮助特定模型族，或逆转模型排名 (Bölük, 2026b)。Anthropic 在长期运行应用开发中报告了这一模式的具体版本：对一个模型有用的上下文重置对更强的模型变得可有可无，移除它们降低了成本而未降低质量 (Anthropic, 2026c)。OpenAI 类似地将 harness 工程框定为保持人类注意力、仓库状态和 agent 执行对齐的学科，而非仅仅添加更多脚手架 (OpenAI, 2026a;b)。

这创造了一个元工程议程：harness 需要优化和简化自身的机制。Meta-Harness 表明 prompt、工具和控制循环可以作为优化目标的一部分被搜索，而非手动固定 (Lee et al., 2026)，而 Natural-Language Agent Harnesses 使 harness 模块显式化且可消融 (Pan et al., 2026)。生产可观测性和成本系统如 TensorZero、Axon 和 AgentOps 指向预算感知的 harness 运营 (TensorZero, 2026; harshkedia177, 2026; AgentOps AI, 2026)，但研究问题比成本最小化更广泛。未来系统应识别哪些干预因果地负责质量、安全或可靠性；跨 harness 变体运行影子模式或 A/B 测试；并在质量-延迟-成本-风险联合约束下进行优化。一个核心风险是基准过拟合：仅针对狭窄测试套件优化自身的 harness 可能变得脆弱。更持久的目标是自适应简化，即 harness 持续追问随着任务、工具和模型能力的变化，哪些控制仍然是必要的。

# 13 结论

本综述将 agent harness 视为独立的工程面，并论证基础设施质量——而非仅模型能力——设定了现实世界 agent 可靠性的上限。围绕这一约束瓶颈论题，我们发展了三个论断。七层 ETCLOVG 分类体系将可观测性和治理从生命周期钩子中分离出来，反映了生产团队已经如何组织其工具和所有权。将 170 多个开源项目映射到该分类体系上，提供了迄今为止最广泛的生态系统快照，并揭示了采用模式、覆盖缺口和新兴设计原则。从 prompt 工程到上下文工程再到 harness 工程的三阶段工程演进，连同涵盖成本-质量-速度三难困境、能力-控制权衡和 harness 耦合问题的跨层综合分析，将 harness 置于更广泛的工程轨迹中。我们的分析有其局限性。语料偏向英语、GitHub 可见的开源项目，以及编码 agent 生态系统；将其扩展到闭源生产系统和非编码 agent 生态系统将使经验图景更加清晰。分类体系本身是描述性的：将 ETCLOVG 转变为能够指导 harness 设计决策而非仅对其分类的规范性框架，是我们希望本综述能够推动的自然下一步。
