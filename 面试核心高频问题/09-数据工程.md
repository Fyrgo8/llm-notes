# 数据工程

> 优先级：P1（高频考察）
> 覆盖范围：预训练数据处理全流程、SFT 数据构造方法、偏好数据工程、合成数据与 Model Collapse、数据飞轮、数据配比与 Scaling、分词、继续预训练
> 核心论点：数据质量是模型能力的天花板，算法和算力只是逼近这个天花板的速度

### 1. 预训练数据处理流水线

预训练数据处理不是一步操作，而是一条多级流水线，每级解决不同层面的问题。典型流程是：原始爬取 → 语言识别 → 规则过滤 → 去重 → 质量打分 → 毒性过滤 → PII 脱敏 → 领域配比混合。下面展开每个关键环节。

**规则过滤（Heuristic Filtering）**。第一道粗筛，用简单规则去除明显垃圾：文档过短（<50 token）或过长（可能是机器生成的垃圾拼接）、特殊字符比例过高（>30% 标点或非文字字符）、重复 n-gram 比例过高（广告页面典型特征）、含黑名单 URL/域名。这步计算成本极低但能砍掉 60-80% 的原始数据——CCNet 处理 CommonCrawl 后只保留约 30% 的页面。工业界实践中还会加入困惑度过滤（用小型 LM 打分，perplexity 过高的文档大概率是乱码或机器生成的垃圾）和语言检测（fastText-based langid 模型，过滤非目标语言的文档）。

**去重（Deduplication）** 是预训练数据处理中最关键的一步。重复数据导致模型在特定模式上过拟合、训练效率下降、且会让模型倾向"背诵"而非"理解"。去重发生在三个粒度：

- **精确去重**（Exact Dedup）：对文档计算 hash（MD5/SHA256），完全相同的文档只保留一份。速度快但只能处理完全一致的副本。工程上通常先做 URL 级去重（同一 URL 爬了多次），再做内容 hash 去重。
- **模糊去重**（Fuzzy Dedup）：处理大量改写、模板套用、镜像站点等"近似重复"。主流方法是 MinHash + LSH（Locality-Sensitive Hashing）——将文档转为 shingle 集合（连续 n-gram，通常 5-gram），计算多组 MinHash 签名估计 Jaccard 相似度，阈值通常设 0.8。LLaMA 用的就是这套方案。SimHash 是另一个选择（将文档映射为固定长度的二进制指纹，汉明距离衡量相似度），更适合检测微小改动。两者的工程差异在于：MinHash 更适合集合相似度、对段落级重复敏感，SimHash 对全文微小修改（如换个标题）更敏感。
- **子串去重**（Suffix Array Dedup）：Lee et al. (2022) 发现即使文档级不重复，也存在大量跨文档的重复段落（如网站 footer、法律声明、版权模板）。用后缀数组找到所有出现 ≥3 次的 50-token 以上连续子串并移除，可以进一步提升训练效率。这一步的计算成本较高（需要对全量语料构建后缀数组），通常只在最终清洗阶段使用。

**质量分类器打分**。规则过滤后仍有大量"格式正确但内容低质"的文本（拼凑文章、SEO 垃圾、低质量翻译）。通用做法是训练一个 fastText/BERT 分类器，以 Wikipedia + 精选书籍为正样本、随机 CommonCrawl 为负样本，对所有文档打 0-1 质量分。GPT-3 用的就是这个方法。更进一步的做法是基于模型的质量过滤——CCI3-HQ-Classifier 就是开源的 SOTA 数据质量过滤模型，能更精细地区分"知识密度高"和"水文但语法正确"的内容。分数阈值是超参——太高导致数据量不足、多样性差，太低则引入噪声。实践中通常取 top 30-50% 的文档。

**毒性过滤与 PII 脱敏**。毒性过滤使用 Perspective API 或训练专门的毒性分类器，识别并移除仇恨言论、色情暴力内容。PII（个人可识别信息）脱敏用正则 + NER 模型识别电话、邮箱、身份证号等，替换为占位符。这两步是安全合规的硬需求。值得注意的是毒性过滤的阈值也需要权衡——过于激进会移除大量涉及敏感话题的正常讨论（如医学文献中的暴力描述、法律案例中的犯罪叙述），影响模型在这些领域的能力。

**数据处理的工程基建**。T 级别的预训练数据处理不可能单机完成。工业界通常用 Spark/Ray Data/Dask 做分布式 ETL，数据存储在 HDFS 或对象存储中。整条流水线需要可复现、可追溯——每个阶段的过滤率、样本分布变化都需要记录，否则出了问题不知道哪个环节的决策有误。

### 2. SFT 数据构造

SFT 数据的核心矛盾是：人工标注质量高但成本贵（一条高质量指令-回答对的标注成本 $5-15），而模型生成的数据量大但可能引入"模型幻觉的自我强化"。主流方法在两者之间寻找平衡。

**Self-Instruct**（Wang et al., 2023）。用少量种子指令（175 条人工编写）引导模型自举生成新指令：模型先生成新的任务描述，再为该任务生成输入-输出对，然后用规则过滤低质量样本（过短、重复、与种子 ROUGE 相似度过高）。Alpaca 就是用 GPT-3.5 通过 Self-Instruct 生成了 52K 条数据，成本仅 $500。核心问题是多样性有限——生成的指令倾向于聚集在模型已经擅长的领域。

**Evol-Instruct**（WizardLM, 2023）。解决多样性问题的关键创新。对已有指令做"进化"——向更深（增加推理步骤）、更广（增加约束条件）、更具体（添加细节）、更抽象（泛化场景）四个方向改写。例如将"写一首诗"进化为"用莎士比亚十四行诗格式写一首关于量子纠缠的诗，要求用至少三个物理学比喻"。通过多轮进化可以生成真正困难的指令，显著提升模型处理复杂任务的能力。

**数据质量 > 数据数量**。LIMA（Zhou et al., 2023）的核心发现：仅用 1000 条精心挑选的高质量数据做 SFT，就能让 65B 模型达到接近 GPT-4 的对话质量。这意味着 SFT 的主要作用是"激活"预训练知识、学习输出格式和交互风格，而非注入新知识。后续的 LESS、Deita 等工作进一步验证了"数据选择"比"数据数量"更重要——用影响函数或聚类方法从大池中选出最有价值的 1-5K 条数据，效果优于全量训练。工业实践中（如美团 QwQ-32B 微调实践），通常采用"规则 + LLM 判断"两阶段数据筛选方法——先用规则过滤格式错误和明显低质样本，再用强模型对剩余样本打分筛选，最终从 4000+ 条原始数据中得到 ~2000 条高质量 CoT 数据用于 SFT。

**多轮对话数据的特殊处理**。SFT 不仅训练单轮问答，还需要训练模型的多轮对话能力。关键技巧是 loss masking——只在 assistant 回复上计算损失，不在 user 输入和 system prompt 上计算。否则模型会"学习生成用户的问题"。ShareGPT 格式的数据（从 ChatGPT 导出的真实对话）因为涵盖自然对话的各种转折模式，是多轮 SFT 数据的重要来源。

**SFT 的局限与 RFT 的兴起**。SFT 本质是"模仿学习"——模型学习复制标准答案的输出模式。它的天花板是标注数据本身的质量：模型最好只能学到和标注一样好，无法超越标注。此外 SFT 存在灾难性遗忘风险——在新任务数据上微调时，模型在相似旧任务上的性能可能下降 30%+。这催生了 RFT（Reinforcement Fine-Tuning）范式——不提供标准答案，而是让模型生成多条候选输出，通过奖励信号（打分器）选择性强化高质量路径。RFT 的核心优势在于模型能"发现创造性解决方案"而非仅仅模仿，适合需要复杂推理的场景。DeepSeek-R1 的纯 RL 路线就是这种思路的极端实践。

### 3. 偏好数据工程

偏好数据是 RLHF/DPO 的燃料——数据的质量直接决定对齐效果的天花板。

**人工标注的流程与挑战**。标准流程是：给定一个 prompt，用策略模型生成 K 个回答（通常 K=4-8，调大 temperature 增加多样性），人类标注者做排序或两两比较。关键挑战在于：标注者间一致性（inter-annotator agreement）通常只有 60-75%，因为"好回答"本身就是主观的；标注者疲劳导致后期质量下降；标注指南的设计直接影响模型行为方向（InstructGPT 的标注指南长达几十页）。

**RLAIF（Reinforcement Learning from AI Feedback）**。用强模型（如 GPT-4）替代人类标注者做偏好判断，大幅降低成本。Constitutional AI（Anthropic, 2022）是典范：定义一组"宪法原则"（如"回答应该无害"、"应该诚实承认不确定性"），让模型按这些原则自我评判、自我修正。具体流程是：模型生成回答 → 模型按原则 critique 自己的回答 → 模型改写为更好的版本 → 改写前后构成偏好对。这绕过了对人类标注的依赖，且原则可以精确编码，一致性高于人类标注。

**偏好数据的质量信号**。不是所有偏好对都同样有价值。margin 小的样本（两个回答差不多好/差）信号弱、噪声大；margin 大的样本（一个明显好于另一个）才提供有效梯度。实践中常用 RM 打分差作为样本权重——差异越大权重越高。UltraFeedback 数据集用 GPT-4 对回答从 helpfulness、honesty、harmlessness 三维度打 1-5 分，提供更细粒度的偏好信号。

**On-policy vs Off-policy 数据**。训练过程中策略模型不断更新，但偏好数据是用旧策略生成的（off-policy）。随着训练推进，旧数据与当前策略的分布偏移增大，RM 的泛化能力下降。解决方案是 iterative RLHF——每隔一定步数用最新策略重新采样、重新标注，保持数据的 on-policy 属性。Online DPO（如 OAIF）就是将这个思路应用在 DPO 框架下。GRPO 更进一步——对同一问题在线采样多个候选回答，利用组内相对排序构造优势信号并归一化，天然实现了 on-policy + 无需独立 RM 的轻量方案。

**偏好数据构造的实操建议**（来自工业实践）。构造 goodcase/badcase 的常见方法：用 SFT 模型调大 temperature 采样多个答案，然后用训练好的 RM 打分排序，高分为 chosen、低分为 rejected。也可以直接用 KTO（Kahneman-Tversky Optimization）——只需要单条数据标"好"或"坏"而不需要配对，省去构造偏好样本对的成本。DPO 的参考模型通常就是 SFT 模型本身——作用是 KL 正则化防止模型被训歪，而不是"让模型向更强的模型看齐"。

### 4. 合成数据（Synthetic Data）

合成数据是 2024-2025 年数据工程最重要的范式转变——从"被动收集真实数据"转向"主动生成训练信号"。当前主流模型几乎都在后训练阶段大量使用合成数据，但其根本性风险也需要深入理解。

**为什么需要合成数据**。高质量 SFT 数据的人工标注成本极高（一条 $5-15），而某些能力维度（复杂推理、多步工具调用、长文本生成）的真实数据天然稀缺。合成数据通过大模型生成的方式弥补这个缺口——用强模型（如 GPT-4）的输出作为弱模型的训练数据，本质上是一种知识蒸馏。Phi 系列模型的成功证明了"教科书级质量"的合成数据可以让小模型在特定维度上媲美大模型。合成数据的核心价值还包括：可控性（精确控制难度分布、领域覆盖、格式多样性）、规模（小时级产出万条）、隐私（敏感领域规避真实数据问题）。

**合成数据的方法体系**。按生成方式分为几种：基于 LLM 生成（最主流，用强模型根据 seed 指令生成训练对）；基于规则/模板（适合结构化场景如数学题、代码题）；基于模拟环境（Agent 场景中让模型在虚拟环境中交互生成轨迹数据）；基于蒸馏（大模型的 logits/reasoning trace 作为小模型的监督信号）。DeepSeek-R1 的训练流程就大量依赖合成数据——先训练一个强推理模型，再用它合成 800K 条高质量 CoT 数据用于后续阶段的蒸馏训练。此外还有几种重要的具体方法：**Rejection Sampling**——让模型生成多个回答，用 RM 或规则验证器筛选最好的，构成 SFT 数据（DeepSeek-R1 用这个方法从 671B 蒸馏到 7B）；**Seed-then-Expand**——从少量真实种子数据出发，让模型生成变体（改变数字、改变场景、增加条件），保持分布接近真实数据；**Self-Play**——模型与自己对弈生成对话数据（SPIN），或模型自己生成问题再自己回答（Self-QA）。

**模型崩溃（Model Collapse）** 是合成数据最严重的系统性风险。当模型在自身生成的数据上反复训练时，会出现"分布收缩"——输出多样性逐渐降低，尾部分布的少数模式被放大，模型趋向于生成越来越同质化的内容。Shumailov et al. (2024, Nature) 正式定义了这个现象并给出了理论证明：纯合成数据的迭代训练会导致分布的尾部信息逐代丢失，最终模型只会输出训练集中高频模式的"平均值"。直觉上类似"复印件的复印件"——每一代都丢失细节。CMU 和北大的研究发现，合成数据的多样性得分和预训练/SFT 后的模型性能呈正相关——多样性比数量更关键。

**缓解模型崩溃的策略**。(1) 始终混入真实数据——合成数据占比不超过总训练数据的 50-70%，保留真实数据锚定分布；(2) 多样性度量监控——用聚类方法度量合成数据的 diversity metric，该分数与下游性能正相关；(3) 来源多样化——不从单一模型合成，而从多个不同模型（Qwen、LLaMA、GPT-4）合成后混合，保持输出风格多样性；(4) 控制合成代数——不做迭代合成（模型 A → 数据 → 模型 B → 数据 → 模型 C），只用原始强模型一次性生成；(5) 每轮合成后做严格的质量过滤和去重检查。

**合成数据的质量控制**。生成后不能直接用，需要多级筛选：规则过滤（格式检查、长度检查、特殊字符检查）→ 验证器验证（代码数据跑测试、数学数据验算答案）→ 模型打分（用 RM 或强模型对质量排序）→ 去重（与已有数据做相似度检查）→ 去除幻觉（合成的"事实"需要与知识库交叉验证）。每一级都会淘汰 30-50% 的数据，最终留下的才是高质量合成数据。好的合成数据还需要确保难度分层——从简单到复杂各级别都有覆盖，避免难度分布偏斜。

### 5. 数据飞轮（Data Flywheel）

数据飞轮是工业界将数据工程从"一次性工作"升级为"持续运转系统"的核心机制——面试中提到这个概念能展示你对工业级 ML 系统的理解深度。

**飞轮的四步循环**。(1) 使用产生数据——用户与模型的每一次交互（查询、反馈、纠错）都是潜在的训练信号；(2) 数据优化模型——收集的数据经处理后用于 SFT、DPO 或更新 RAG 知识库；(3) 模型提升产品——优化后的模型提供更好的用户体验；(4) 产品吸引使用——更好的体验带来更多用户和更多数据。这个循环的价值不在于"存在"，而在于"速度"——完成一次从数据收集到模型更新的循环时间越短，竞争壁垒越深。

**飞轮的战略意义**。NVIDIA 的报告指出，成熟的数据飞轮可将推理成本降低高达 98.6%——通过持续用专有数据训练更小的专用模型替代大型通用模型。随着基础模型逐渐商品化（开源模型质量接近闭源），真正的竞争优势不再来自模型架构，而来自专有数据——它是难以被竞争对手复制的"护城河"。构建数据飞轮的最终目标是：用专有数据训练更小、更快的专用模型，在特定任务上匹配甚至超越通用大模型，同时大幅降低推理成本。这就是"以数据为中心的 AI"(Data-Centric AI) 的核心理念。

**工程实现**。典型的数据飞轮 Pipeline 包含：(1) 数据采集层——埋点捕获用户 query、模型 response、用户反馈（点赞/点踩/修改）；(2) 数据处理层——清洗、去重、质量打分、构造训练格式（基于 Hive 数据仓库做 ETL）；(3) 评估层——自动化 benchmark + LLM-as-Judge 识别模型弱点；(4) 训练层——针对弱点数据做增量 SFT 或 DPO；(5) 部署层——新模型灰度上线，继续收集数据。整个循环自动化程度越高、"循环时间"越短越好。

### 6. 数据配比与 Scaling

预训练数据不是简单堆量，不同来源的数据配比对模型能力有决定性影响。

**领域配比的经验法则**。主流模型的典型配比是：网页数据 ~60%（通用知识）、代码 ~15%（推理+格式能力的意外收获）、书籍+论文 ~15%（深度知识）、对话数据 ~5%（交互能力）、数学/科学 ~5%（精确推理）。代码数据的重要性是一个反直觉的发现——即使目标不是代码生成，加入代码训练也能显著提升模型的逻辑推理和指令跟随能力（Codex 论文的附带发现，后被 LLaMA-2 验证）。

**DoReMi（Xie et al., 2023）**。自动化确定最优数据配比的方法。核心思路是训练一个小的 proxy model，用 distributionally robust optimization（DRO）找到使最差领域损失最小化的采样权重，然后用这组权重训练大模型。实验表明 DoReMi 找到的配比在所有领域上都优于人工调参，且小模型确定的配比可以直接迁移到大模型——这与 Scaling Law 的"小模型预测大模型行为"精神一致。

**数据 Scaling Law**。Chinchilla 给出了 token 数 $D$ 与模型参数 $N$ 的最优比例（$D \approx 20N$），但这是在固定数据质量下的结论。Llama-3 的实践表明，如果持续提升数据质量（更严格的去重、更好的质量过滤、更均衡的领域配比），即使大幅超过 Chinchilla optimal 的数据量（8B 模型用 15T tokens），loss 仍在持续下降。这说明"数据量上限"实际上是"数据质量的上限"——只要能持续注入不重复的高质量数据，过训练就有收益。

**Curriculum Learning 在预训练中的应用**。不是从头到尾用固定配比，而是分阶段调整数据分布。典型策略：早期用大量通用数据建立基础语言能力 → 中期逐步增加代码和数学数据强化推理 → 后期注入高质量对话和指令数据（annealing phase）。Qwen-2.5 明确采用了这种 curriculum 策略，最后阶段用小量极高质量数据做"退火"使各项能力峰值对齐。ACL 2025 的 Keynote 也强调了这个观点："几乎所有高级的模型能力，本质都来自预训练数据"，数据的表征提取方式和阶段性策略是预训练效果的最大影响因素。

### 7. 分词（Tokenization）

分词看似基础但追问很深，面试中"BPE 的合并规则怎么定的"就能筛掉一批人。

**BPE（Byte Pair Encoding）**。训练过程：从字符级词表开始，统计所有相邻 token 对的频率，将最高频的 pair 合并为新 token 加入词表，重复直到词表达到目标大小（通常 32K-128K）。贪心算法，每步只看当前频率最高的 pair。推理时用同样的合并规则对输入文本做分词——按训练时学到的合并顺序依次合并。核心特性是：高频词/子词会被合并为单个 token（"the"、"ing"），低频词被拆分为多个子词，从而平衡词表大小和序列长度。

**WordPiece vs BPE**。WordPiece（BERT 使用）与 BPE 的关键区别在于合并标准——BPE 用 pair 的绝对频率，WordPiece 用似然增益：选择合并后使训练语料似然提升最大的 pair。直觉上 WordPiece 更"全局最优"，但实践中两者效果差异很小，BPE 因实现简单而更流行。

**Unigram Language Model**（SentencePiece 实现）。与 BPE 的"自底向上合并"相反，Unigram 是"自顶向下剪枝"：从一个大词表开始，用 EM 算法估计每个 token 的出现概率，移除去掉后对总似然影响最小的 token，重复直到词表缩减到目标大小。分词时用 Viterbi 算法找到使序列似然最大的切分方式。优势是同一个词可能有多种切分，提供正则化效果。

**Byte-level BPE** 是当前最主流的选择（GPT-4、Qwen2.5、LLaMA-3 都用这个）。关键创新：不从字符级开始，而从 byte 级（256 个基础 token）开始做 BPE。好处是天然处理任何语言和 Unicode 字符，完全消除 OOV（Out of Vocabulary）问题——任何文本最差情况就是退化为 byte 序列。传统字符级 BPE 遇到未见的 Unicode 字符（如特殊 emoji）会失败，byte-level 不存在这个问题。ACL 2025 Keynote 中也提到了新型 byte-level tokenizer 在扩展性上比传统 BPE 更有优势。

**词表大小的权衡**。词表太大：embedding 矩阵（`vocab_size × d_model`）占用大量显存，低频 token 的 embedding 训练不充分（出现次数太少学不好）；词表太小：同样内容需要更长的序列表示（增加计算量和 attention 复杂度），单个 token 的语义粒度太粗。主流选择：32K（LLaMA）到 152K（Qwen2.5），多语言模型倾向于更大词表以覆盖非拉丁文字。经验上中文每个汉字应该平均 1-1.5 个 token——如果一个中文字被拆成 3-4 个 token，说明词表中文覆盖不足。

### 8. 继续预训练（Continued Pre-Training, CPT）

CPT 是垂域模型构建的标准路径——在通用基座上用领域数据继续训练，注入专业知识而不破坏通用能力。

**CPT 的定位**。如果说预训练产出的基座模型是"高中生"，CPT 相当于让他读了"本科"——灌输了领域专业知识，但还不针对特定任务格式。CPT 适合的场景是：目标领域有大量无标注文本（如医疗文献、金融报告、法律法规），且通用模型对该领域术语和知识严重不足。

**关键工程决策**。(1) 数据配比——纯领域数据会导致通用能力退化（"灾难性遗忘"），通常混入 20-30% 的通用数据保持基础能力；(2) 学习率——CPT 的学习率通常设为预训练末期学习率的 1/10 到 1/5，太大会破坏已有知识，太小则学不进去新知识；(3) 数据量——通常需要 10B-100B tokens 的领域数据才能看到显著效果，几百 MB 的数据更适合直接 SFT 而非 CPT。

**Replay Buffer 防遗忘**。在 CPT 过程中周期性地回放通用数据（类似强化学习中的 experience replay），可以有效缓解通用能力退化。Qwen-2.5 的 CPT 策略就包含了这种周期性回放机制。

### 9. 面试高频问题

**Q: 合成数据和真实数据的关系是什么？该怎么选择？**
不是替代关系而是互补关系。真实数据定义了目标分布的"锚点"——告诉模型用户真正关心什么、真实场景长什么样。合成数据负责填充——覆盖长尾场景、增加难度梯度、补充数据不足的领域。经验法则是：SFT 阶段合成数据可以占到 70-80%（因为格式学习对分布精度要求低），但 RL 阶段的 reward signal 必须来自真实用户偏好（否则模型只会学会讨好合成数据的生成模型，而非讨好真实用户）。

**Q: 数据飞轮中最容易出错的环节是什么？**
数据质量控制。飞轮转得越快，垃圾数据进入训练集的风险越大。如果用户的负反馈（点踩）被错误地标为正样本，模型会越训越差形成"死亡螺旋"——这是飞轮的对立面。因此数据校验环节（包括人工抽检 + 自动异常检测）是飞轮中不可跳过的关卡。

**Q: 继续预训练和 SFT 的边界在哪里？什么时候该用 CPT 什么时候该用 SFT？**
关键区分在于：模型"不知道"这个知识（知识缺失）还是模型"知道但不会用正确格式输出"（格式/风格问题）。如果你问模型一个领域问题它给出完全错误的答案（不是格式问题而是知识错误），说明需要 CPT 注入知识。如果模型知道答案但输出格式不对、不够简洁、或不符合业务规范，SFT 就够了。实践中的判断方法：用 few-shot prompting 试一下——如果给了示例后模型能正确回答，说明知识已有，只需 SFT 对齐格式；如果给了示例还是答错，就需要 CPT。

**Q: MinHash 和 SimHash 怎么选？**
看去重目标。MinHash 基于 Jaccard 相似度，擅长发现"内容重叠度高"的文档（如模板文章、镜像站内容），适合段落级和文档级去重。SimHash 基于汉明距离，擅长发现"整体相似但有微小改动"的文档（如只改了标题或日期），适合检测轻微变体。工程上 MinHash 的计算成本较高（需要多组哈希函数），SimHash 更轻量但对局部大段修改不敏感。实践中通常两者结合使用。

**Q: 为什么 DPO 训练容易"训歪"？偏好数据层面怎么防止？**
DPO 的失败模式主要是：(1) chosen 和 rejected 之间 margin 太小导致噪声主导梯度——解决方案是用 RM 打分筛选高 margin 样本；(2) 偏好数据的分布与目标任务不匹配——训练时的 prompt 分布应尽量接近真实用户分布；(3) 数据中存在内在矛盾（同类型 prompt 的偏好方向不一致）——需要做标注一致性清洗。此外 DPO 对参考模型（reference model）的选择敏感——如果 SFT 模型本身质量就差，DPO 的 KL 约束会把优化锚定在一个低水平基线上。
