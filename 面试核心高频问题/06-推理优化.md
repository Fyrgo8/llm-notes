# 推理优化

> 优先级：P1（重要）
> 覆盖范围：Prefill/Decode 两阶段、KV Cache 管理、PagedAttention/vLLM、投机解码、量化推理、解码策略
> 关联背景：理解推理瓶颈是理解 GQA/KV Cache 量化等设计决策的前提

### 1. Prefill 与 Decode 两阶段

LLM 推理分为两个性质截然不同的阶段。Prefill 阶段并行处理整个 prompt，所有 token 的 K/V 一次性计算并缓存，此阶段是 compute-bound——GPU 算力是瓶颈，利用率高。Decode 阶段逐 token 自回归生成，每步只产生一个 token 但需要读取全部已缓存的 K/V，此阶段是 memory-bound——大量时间花在从 HBM 搬运 KV Cache 数据上，GPU 算力严重闲置。

一个直觉性的数据：7B 模型 decode 阶段每生成一个 token 需读取约 14GB 参数权重（模型本身）+ 若干百 MB KV Cache，但只执行一次 token 的矩阵向量乘法。Memory bandwidth utilization（MBU）通常不到 40%。推理优化的核心就是解决 decode 阶段的 memory-bound 问题。

### 2. KV Cache 深入分析

自回归生成第 $t$ 步时，前 $t-1$ 个 token 的 K/V 值不变，缓存它们避免每步 $O(t)$ 的重复计算。但 KV Cache 本身的显存占用不可忽视：

$$
\text{KV Cache} = 2 \times L \times n_{kv\_heads} \times d_{head} \times seq\_len \times \text{dtype\_bytes}
$$

7B 模型（32 层、8 KV heads、head_dim 128、fp16）在序列长度 4096 时，单条请求 KV Cache 约 512MB。并发 32 条请求就是 16GB——接近一张 A100 显存的 20%。这是 GQA（减少 KV heads）、KV Cache 量化（fp16→int8/int4）、PagedAttention（减少碎片浪费）等优化的直接驱动力。

### 3. PagedAttention 与 vLLM

传统 KV Cache 实现为每条请求预分配连续显存块（按最大可能序列长度分配），导致严重的内部碎片——请求实际长度通常远小于预分配长度，实测利用率常低于 50%。

vLLM 的 PagedAttention 借鉴操作系统虚拟内存的分页思想：将 KV Cache 切成固定大小的 block（如每 block 存 16 个 token 的 KV），逻辑连续的序列可存储在物理上不连续的 block 中，通过 block table 映射。需要时按需分配 block，请求完成即释放。

这一设计将 KV Cache 利用率提升到 >95%，直接翻译为同等显存下可支持 2-4 倍的并发请求数。附带好处：多条请求共享相同 prefix 时可共享物理 block（如共同的 system prompt），进一步节省显存。

**Continuous Batching** 是 vLLM 的另一核心：不等整个 batch 都完成才调度新请求，而是哪条请求先完成就立刻释放资源并插入新请求。相比 static batching，吞吐提升 2-5 倍。

### 4. 投机解码

投机解码（Speculative Decoding）用小模型（draft model）快速自回归生成 $K$ 个候选 token，然后大模型一次性并行验证这 $K$ 个 token 的条件概率。验证通过的接受，第一个未通过的位置用大模型重新采样。

加速原理：将 $K$ 步 memory-bound 的 decode 合并为 1 步 compute-bound 的 verify。大模型的 verify 步骤是对 $K$ 个 token 做并行 forward（类似 prefill），GPU 利用率高。

关键性质：通过拒绝采样（rejection sampling）保证最终输出的分布与纯大模型生成完全一致——这是无损加速。实际加速倍数取决于 draft model 的 acceptance rate，通常在 2-3 倍。Medusa 等变体用额外的预测 head 替代独立 draft model，减少额外显存开销。

### 5. 量化推理

量化将模型权重从 fp16 压缩到更低位数，直接减少显存占用和内存带宽需求：

GPTQ 是离线量化方法，逐层求解最小化重构误差的最优量化参数，支持 3/4 bit。AWQ（Activation-aware Weight Quantization）发现不同通道对激活的贡献差异巨大，对重要通道保持高精度，不重要通道激进量化，同等位数下精度优于 GPTQ。

SmoothQuant 解决激活量化难题——激活值中存在少量极大 outlier 通道使得直接量化损失严重。SmoothQuant 通过数学等价变换将激活的量化难度"转移"到权重上（weight 乘以 scale、activation 除以同一 scale），实现 W8A8 全量化推理。

GGUF 是 llama.cpp 的格式，支持 2-8 bit 混合量化，针对 CPU 推理优化，适合端侧部署。

### 6. 解码策略

Greedy decoding 每步取概率最高的 token，确定性但单调。Temperature $T$ 缩放 logits：$T<1$ 使分布更尖锐（更确定），$T>1$ 使分布更平坦（更多样）。Top-k 只从概率最高的 $k$ 个 token 中采样。Top-p（nucleus sampling）从累积概率达到 $p$ 的最小 token 集合中采样——集合大小自适应（简单 token 时集合小，歧义 token 时集合大）。

实践中通常组合使用：temperature 0.6-0.8 + top-p 0.9 是常见配置。Repetition penalty 对已生成过的 token 降低 logit，缓解重复生成问题。

### 7. Prefix Caching

对于有共同 system prompt 的多请求场景（如 API 服务的默认角色设定），prefix 部分的 KV Cache 可以在请求间共享，避免重复 prefill。vLLM 通过对 block 内容做哈希来自动检测可共享的 prefix——如果两条请求的前 N 个 token 相同，则复用对应的 KV block。在 system prompt 较长（数千 token）的场景下，这能显著降低首 token 延迟（TTFT）。
