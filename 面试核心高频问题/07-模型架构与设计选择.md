# 模型架构与设计选择

> 优先级：P2（加分项）
> 覆盖范围：Decoder-only 的胜出原因、LLaMA 架构细节、SwiGLU、MoE、长上下文方案
> 定位：面试中作为追问出现，理解"为什么这样设计"比记住结论更重要

### 1. 为什么 Decoder-only 架构统治了 LLM

三种架构——Encoder-only（BERT）、Encoder-Decoder（T5）、Decoder-only（GPT）——在 2020 年前各有应用。但 Scaling Law 验证后，Decoder-only 成为绝对主流。原因有多个层面：

**低秩表达问题**（苏剑林分析）。Encoder 的双向 attention 矩阵经 softmax 归一化后，尽管输入矩阵满秩，输出实际退化为低秩——信息被过度"平均"。因果 attention 的下三角结构天然是满秩矩阵，每行的归一化不会相互干扰，表达能力更强。

**Zero-shot 泛化**。Decoder-only 的自回归目标天然对应"续写"，用 prompt 就能定义任意任务（GPT-3 的 in-context learning 验证了这一点）。Encoder 模型需要为每个任务设计特定的 head，泛化性差。

**推理效率**。Decoder-only + KV Cache 天然支持增量解码——每步只需计算一个新 token 的 forward。Encoder-Decoder 需要先对整个输入做一次完整编码，cross-attention 还引入额外计算。

**Scaling Law 效率**。Chinchilla 的实验表明，在固定算力预算下 Decoder-only 的参数/数据分配最优点效率最高。

### 2. LLaMA 架构的五个关键设计

LLaMA 相比原始 GPT 架构做了五处改动，几乎被后续所有开源模型沿用：

**Pre-Norm + RMSNorm**。将 LayerNorm 移到 Attention/FFN 之前（Pre-Norm），训练更稳定不易梯度爆炸。进一步用 RMSNorm 替代 LayerNorm——省去减均值步骤，只保留除均方根操作，参数从 $2d$ 减到 $d$，计算快 10-15%。

**SwiGLU 激活函数**。FFN 层用 $\text{SwiGLU}(x) = \text{Swish}(xW_1) \otimes (xW_2)$ 替代 ReLU/GELU。Swish 即 $x \cdot \sigma(x)$，平滑无死区；gating 机制（逐元素乘）提供信息筛选能力。代价是引入两个投影矩阵（参数增加 50%），通常通过将隐层维度从 $4d$ 减到 $\frac{8}{3}d$ 补偿。

**RoPE 位置编码**。相对位置编码，作用在 Q/K 上，支持长度外推（详见位置编码章节）。

**GQA**。Grouped-Query Attention 减少 KV Cache 显存占用，加速推理（详见 Attention 章节）。

**BPE 分词**。SentencePiece 实现的 BPE，平衡词表大小和序列长度，天然处理未见词和多语言。

### 3. MoE（Mixture of Experts）

MoE 在 FFN 层设置多个 expert（每个 expert 是一个独立的 FFN），通过门控网络（router）为每个 token 选择 top-k 个 expert 激活。

Mixtral 8×7B 总参数 46B，但每个 token 只激活 2 个 expert 约 13B 参数——模型容量大但推理计算量仅相当于 13B 密集模型。这是"参数量与计算量解耦"的思路。

MoE 的工程挑战集中在三点：load imbalance（所有 token 涌向少数 expert，其余 expert 浪费），通常用辅助损失强制均衡；expert parallelism（不同 expert 放不同卡，需要 all-to-all 通信调度 token）；训练不稳定（loss spike 频繁），需要更保守的学习率和更强的梯度 clip。

### 4. Causal LM vs Prefix LM

Causal LM（GPT 系列）对整个序列施加因果 mask——每个 token 只看左侧。Prefix LM（UniLM、GLM）则将序列分为 prefix 部分（双向 attention）和 generation 部分（因果 attention），prefix 内部各 token 互相可见。

Prefix LM 在"有明确输入输出边界"的任务上（如翻译、摘要）有天然优势——prefix 部分相当于 encoder，能充分理解输入语义。但它的局限在于：训练时需要标注哪一段是 prefix、哪一段是 generation，灵活性不如纯 Causal LM；zero-shot 泛化能力弱于"一切皆续写"的纯因果模型。T5 的 Encoder-Decoder 可以视为 Prefix LM 的一种极端形式。

当前主流选择是纯 Causal LM + 足够长的 context window——用 prompt 格式模拟 prefix 的效果，同时保持架构统一性和 scaling 效率。

### 5. 涌现能力与 Scaling Law

**涌现（Emergence）** 指某些能力在模型规模突破阈值后"突然出现"的现象。Wei et al. (2022) 发现 CoT 推理、多步算术等任务在小模型上接近随机表现，到 60B-100B 参数时性能跳跃式提升。一种解释是评估指标的"相变"——底层能力是连续提升的，但任务成功率（精确匹配）是 0/1 阈值函数，跨过阈值前看起来没有进步。

**Scaling Law**（Kaplan et al., 2020; Chinchilla, 2022）给出了 loss 与模型参数量 $N$、训练 token 数 $D$、计算量 $C$ 之间的幂律关系：$L \propto N^{-\alpha} + D^{-\beta} + C_{\min}$。Chinchilla 的关键结论是：在固定计算预算下，参数量和训练数据量应等比例扩展。GPT-3（175B，300B tokens）在 Chinchilla 意义上严重"欠训练"——同样算力下训练一个更小但吃更多数据的模型（70B，1.4T tokens）能达到更好效果。这直接催生了 LLaMA（7B-65B，1-1.4T tokens）的设计哲学。

训练 token 数的经验估算：Chinchilla optimal 约为参数量的 20 倍；但实际部署中推理成本与模型大小正相关，因此很多团队选择"过训练"小模型（如 LLaMA-2 7B 用了 2T tokens，远超 Chinchilla optimal），用更多训练计算换取更低的推理成本。

### 6. 激活函数演进

Transformer FFN 层的激活函数经历了从 ReLU 到 SwiGLU 的演进路径，理解每一步"解决了什么问题"是面试要点：

**ReLU** $= \max(0, x)$。简单高效但有"神经元死亡"问题——一旦输入为负则梯度恒为 0，该神经元永久失活。在深层网络中大量死亡神经元浪费模型容量。

**GELU** $= x \cdot \Phi(x)$ 其中 $\Phi$ 为标准正态 CDF。可以理解为"概率性的 ReLU"——不是硬截断为 0，而是以输入值的概率乘以自身。GPT-2/3 和 BERT 的默认选择。优点是全程可导且在零点附近平滑过渡，避免死亡问题。

**Swish/SiLU** $= x \cdot \sigma(x)$。Google 通过 NAS 搜索发现的激活函数，数学上是 GELU 的近似。特点是负半轴有微小的负值区域（"非单调"），提供了自门控能力。

**SwiGLU** $= \text{Swish}(xW_1) \otimes (xW_2)$。PaLM、LLaMA 引入的 gated FFN 变体。关键创新在于 gating 机制：用一组参数（$W_1$）生成门控信号，用另一组参数（$W_2$）生成候选值，逐元素相乘实现信息筛选。这比单纯的非线性变换多了一个"决定保留多少信息"的维度。代价是 FFN 需要三个权重矩阵（$W_1, W_2, W_3$），参数量增加 50%，通常通过将隐层维度从 $4d$ 缩减到 $\frac{8}{3}d$ 补偿。

### 7. 长上下文的架构方案

除 RoPE 外推（NTK/YaRN）之外，还有架构层面的长上下文方案：

**Sliding Window Attention**（Mistral）。每层只 attend 局部窗口（如 4096），跨层叠加后有效感受野线性增长。配合少量 global tokens（如 CLS 或首 token）保持全局信息流。显存从 $O(n^2)$ 降到 $O(n \cdot w)$。

**Ring Attention**。将长序列切分到多卡，每卡处理一段。K/V 在卡间以环形方式传递，每卡依次与所有远端的 K/V 做 attention。通信量可与计算完全 overlap。

**SSM/Mamba**。用状态空间模型替代 attention，推理时维护固定大小的状态向量（不随序列长度增长），实现 $O(1)$ 推理显存。训练时通过 parallel scan 保持并行性。效果接近但尚未超越同规模 Transformer，混合架构（Jamba = Mamba + Attention 层交替）是当前探索方向。

### 8. LLM 的复读机问题

大模型在解码时偶尔陷入重复循环（不断重复同一段短语），称为"复读机"现象。原因是自回归采样的自我强化——模型刚生成的 token 进入 context，提升了相同 token 再次出现的概率。解决方案包括：Repetition Penalty（对已出现 token 的 logit 除以惩罚因子）、Frequency/Presence Penalty（OpenAI API 的做法），以及训练阶段的 Unlikelihood Loss（在训练损失中显式降低重复 token 的概率）。工程上通常设置 repetition_penalty = 1.1-1.3 即可有效缓解。
