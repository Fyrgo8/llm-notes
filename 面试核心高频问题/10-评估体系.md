# 评估体系

> 优先级：P1（高频考察）
> 覆盖范围：主流 Benchmark、评估方法论、数据污染检测、LLM-as-Judge、Agent 评估新基准、业务评测实践、Reward Model 评估
> 核心论点：不可衡量则不可改进——评估体系决定了你优化方向的正确性

### 1. 主流 Benchmark 全景

评估大模型不是跑一个分数就完事——不同 benchmark 测的是不同维度的能力，理解它们的设计思想和局限是面试加分项。

**知识与推理类**。MMLU（Massive Multitask Language Understanding, 57 个学科）是通用知识的"高考"——但它的问题格式全是多选题，模型可以通过排除法或模式匹配拿到虚高分数，不能反映真正的开放式问答能力。MMLU-Pro 增加了 10 个选项并加入更多推理题缓解这个问题。ARC（AI2 Reasoning Challenge）分 Easy/Challenge 两档，测试科学常识推理。HellaSwag 测试常识推理（给定前文选最合理的后续），但由于对抗性 filtering 的设计导致某些选项通过表面统计特征就能排除，高分不一定代表真的理解了常识。

**数学与代码类**。GSM8K（8500 道小学数学应用题）是入门级数学推理 benchmark——但目前主流模型都已刷到 90%+，区分度很低。MATH（竞赛级数学）更有挑战性，目前最好的模型也只在 60-70%。HumanEval（164 道编程题，测试 pass@k）是代码能力的标配评测，MBPP 是其补充。LiveCodeBench 使用比赛新题持续更新，避免数据泄露。

**对话与指令跟随类**。MT-Bench（Multi-Turn Benchmark）用 GPT-4 对模型的多轮对话回复打 1-10 分，覆盖 8 类能力（写作、推理、角色扮演等）。AlpacaEval 2.0 用 LC（length-controlled）win rate 衡量模型输出相比参考模型的优势，去除了"更长回答得更高分"的 bias。Chatbot Arena 是目前公认最可靠的对话模型排名——真人匿名投票的 ELO 排名，没有数据泄露风险（因为问题来自真实用户的即兴提问），但获取排名需要大量用户流量。

**长文本类**。随着上下文窗口从 4K 扩展到 1M+，长文本评测成为新赛道。RULER 测试不同长度下的检索/推理能力，NIAH（Needle in a Haystack）测试"大海捞针"——在超长文档中找到指定信息。InfiniteBench 覆盖 100K+ 的各种长文本任务。这些 benchmark 暴露了一个关键问题：很多模型声称支持 128K context 但实际在 32K 以后准确率就开始显著下降。

**Agent 评估新基准**。2024-2025 年 Agent 能力评估成为新焦点。GAIA（General AI Assistants, Meta 2024）用多模态、多步骤推理题测试模型的"Level 1-3"推理能力——Level 1 只需 1-2 步推理，Level 3 需要联网搜索 + 代码执行 + 多步组合；SWE-bench 让模型修复真实 GitHub issue（给定 issue 描述和代码库，模型需要定位 bug 并生成正确 patch），SWE-bench Verified 是经人工验证的更可靠子集；PaperBench（METR 2025）测试模型复现 AI 顶会论文的能力——理解论文 + 规划实验 + 编写代码 + 分析结果的全流程，反映了对复杂多步任务的规划与执行能力；WebArena/VisualWebArena 测试模型在真实网页环境中完成任务的能力，涉及导航、表单填写、信息提取等。这些 Agent 类 benchmark 的共同特征是：答案不是简单的字符串匹配，而需要验证"目标是否达成"——这本身就让评估变得更复杂。

### 2. 评估方法论

评估不只是跑 benchmark 拿数字——方法论层面的选择对结果影响巨大。

**固定选项 vs 开放生成**。同一模型在多选题格式（MMLU）和开放生成格式下的表现可以相差 5-10 个点。多选题允许模型"猜"——4 选 1 的随机基线是 25%，这给弱模型留了很大的"虚高空间"。开放生成则要求模型真正理解并表达，但自动评测更难（需要语义匹配而非精确匹配）。面试追问点：MMLU 用 5-shot 还是 0-shot？用 CoT 还是直接出答案？这些选择都会显著影响分数，所以"我们模型 MMLU 90 分"这种说法如果不说明具体设置就没有意义。

**Pass@k 的计算**。HumanEval 的标准指标 pass@1 不是"生成一次看对不对"，而是无偏估计量：对每道题生成 n 个样本（n >> k），统计通过了 c 个，然后用组合公式 $pass@k = 1 - \frac{\binom{n-c}{k}}{\binom{n}{k}}$ 计算。这避免了采样方差——如果只生成 1 个样本，运气好碰到正确解也只能得 pass@1=1。直觉上 pass@k 衡量的是"生成 k 次至少有一次正确的概率"。k=1 反映实用场景（用户只看一次输出），k=100 反映模型的能力上界（能力是有的只是概率低）。

**Perplexity 的局限**。PPL 衡量模型对测试文本的预测能力——PPL 低说明模型认为这些文本"理所当然"。但 PPL 低 ≠ 生成质量高：模型可能对所有输入都给出高置信度的平庸回答（安全但无用），获得低 PPL 但用户体验差。此外 PPL 依赖测试集分布——在 Wikipedia 上 PPL 很低的模型不代表能写好代码。面试高频追问："PPL 和人类评价的相关性强吗？"——答案是在生成质量上相关性只有中等（0.4-0.6），因为 PPL 优化的是"像训练数据"，不是"有用"。

**确定性评估 vs LLM-as-Judge**。这是业务评测中最核心的方法论选择。确定性评估（字符串匹配、F1、BLEU、正则提取 + 规则判断）速度快、无波动、成本低，适合有标准答案的场景。LLM-as-Judge 能评估开放式任务（如"回答是否有帮助"、"推理链是否正确"）但有三大问题：(1) 成本高——每条评测需要调用一次强 LLM；(2) 一致性差——同样的输入重复评几次分数可能不同；(3) 有系统性偏差（详见第 4 节）。工业实践中通常两者结合——确定性评估做筛选（快速排除明显差的），LLM-as-Judge 做精细排序（在候选模型间挑最好的）。

### 3. 数据污染检测

数据污染是当前 benchmark 体系面临的最大信任危机——如果测试集泄露到了训练集中，分数就毫无意义。

**为什么污染难以避免**。预训练数据量级达到数万亿 token，来源数千个网站，很难逐一检查是否包含某个 benchmark 的题目。更隐蔽的污染是"间接污染"——训练数据中没有原题，但包含了大量对该 benchmark 的讨论（如论文分析、教程解析），模型从中"间接学到"了答案模式。

**主流检测方法**：

- **n-gram 重叠检测**。最基础的方法：将测试集每道题与训练集做 n-gram 匹配（通常 8-gram 或 13-gram），超过阈值判为泄露。GPT-4 技术报告中用的就是 13-gram overlap。缺点是对改写后的泄露（换个说法表达同样的题目）无能为力。
- **Perplexity 异常检测**。如果模型对某道测试题的 PPL 异常低（远低于同难度其他题目），很可能在训练时见过。具体做法是对比被测模型与参考模型（确认没见过测试集的模型）在每道题上的 PPL 差异——差异过大的标记为可疑。
- **Canonical Form 规范化**。将测试题归一化到标准形式（去除格式差异、打乱选项顺序），然后检查规范化后的文本是否出现在训练集中。这能捕获"只是换了选项顺序"或"加了无关前缀"的轻微变形泄露。
- **LLM-assisted 检测**。最新的方法是用模型自身来判断——如果模型不仅能做对题目、还能"背出"该题的上下文信息（比如知道这道题来自 MMLU 的哪个子集、原始出处是什么教材），说明它在训练时见过这道题而不是"真的会做"。

**面试追问**。"你怎么确认你模型在 MMLU 上的分数没有被污染影响？"——标准答案是多管齐下：n-gram 检测 + PPL 异常分析 + 在全新的 held-out 测试集上验证一致性。如果分数只在已知 benchmark 上高、在同类型但全新的测试集上显著下降，大概率存在污染。LiveBench 和 LiveCodeBench 等"持续更新"的 benchmark 正是为了从根本上解决这个问题——题目在评测后才公开，永远领先于训练数据。

### 4. LLM-as-Judge

用一个强 LLM 当评委给另一个 LLM 的输出打分——这是目前开放式任务评估的主流方案（MT-Bench、AlpacaEval 都用这个），但它有严重的系统性偏差需要了解。

**核心偏差（面试必知三大 bias）**：

- **位置偏差（Position Bias）**。在 pairwise comparison 中，模型倾向于选择出现在第一个位置的回答，即使两个回答完全相同。有的模型（如 GPT-4）偏好第一个，有的偏好最后一个。解决方案：做两次评判，交换两个回答的位置，只有两次一致才算有效评判。
- **冗长偏差（Verbosity Bias）**。更长的回答倾向于得到更高分——即使长度来自无意义的重复或废话。AlpacaEval 2.0 专门引入了 Length-Controlled win rate 来去除这个偏差。实践中可以在 prompt 中明确要求"不要因为长度而给出更高分"并做 length normalization。
- **自我增强偏差（Self-Enhancement Bias）**。模型给自己生成的内容打分高于同等质量的其他模型输出。如果用 GPT-4 评判 GPT-4 vs Claude 的输出，GPT-4 倾向于选自己。解决方案是避免用被评模型的同系列模型做 judge，或使用多个 judge 取共识。

**多维度打分 vs 整体打分**。只给一个 1-10 的总分区分度太低且不可解释。更好的做法是多维度独立打分——如 helpfulness、accuracy、relevance、creativity 各打一个分，再加权汇总。UltraFeedback 数据集就是按多维度标注的。这也让 debug 更容易：如果模型在"helpfulness"上得分高但"accuracy"上得分低，说明它倾向于编造答案。

**提升 LLM-as-Judge 可靠性的方法**。(1) 结构化评分——要求 judge 先输出评判理由（Chain-of-Thought），再给分数，理由可以帮助人工审核评判是否合理；(2) 参考答案辅助——给 judge 提供一个标准答案或评分标准（rubric），减少主观性；(3) 多 judge 投票——用 3-5 个不同 LLM 分别评判，取多数票；(4) 校准（Calibration）——先让 judge 在一组已有人工标注的样本上跑一遍，检查与人类评判的一致性（Cohen's kappa > 0.7 才可信赖）。

**面试追问**："什么时候不应该用 LLM-as-Judge？"——当任务需要精确事实判断时（如代码是否能编译通过、数学答案是否正确），应该用确定性验证器（编译器/测试用例/符号计算器），不用 LLM 评判。LLM-as-Judge 更适合无标准答案的开放式质量评估。

### 5. Reward Model 评估

Reward Model（RM）是 RLHF 的核心组件——RM 的质量直接决定了对齐训练的方向是否正确。但如何评估 RM 本身的好坏，是一个经常被忽略但面试会追问的话题。

**RM 的评估维度**。(1) **准确率**——给定偏好对 (chosen, rejected)，RM 能否正确判断 chosen > rejected？在标准偏好数据集上衡量。(2) **一致性**——同一样本多次评分是否稳定？RM 不应该对微小的无关改动（如加一个空格）产生显著分数变化。(3) **区分度**——RM 的分数分布是否有足够的方差？如果所有回答都给 0.5-0.6 分，这个 RM 没有区分能力。(4) **泛化性**——在训练分布外的新 prompt 上是否仍然可靠？

**RM 的常见失效模式**。(1) **长度 hacking**——模型学会生成更长的回答来获得高分，因为 RM 的训练数据中长回答往往是 chosen（包含更多信息）。这不是真正的质量提升而是利用了 RM 的 shortcut。(2) **格式 hacking**——模型学会用 markdown、列表、代码块来获得高分，因为 RM 将格式丰富与质量高关联。(3) **reward overoptimization**——当 PPO 训练过度优化 RM 给出的分数时，模型的真实质量（人类评价）反而下降——这就是 Goodhart's Law（"当衡量指标变为目标时，它就不再是好的衡量指标"）。

**RewardBench**。Evaluating Reward Models 的标准 benchmark——包含多个类别（chat、safety、reasoning）的偏好对，测试 RM 的综合判断能力。不同 RM 在不同类别上的表现差异很大——有的 RM 在 safety 维度上非常准但在 reasoning 维度上接近随机。这意味着实践中可能需要多个专门的 RM（safety RM + helpfulness RM + reasoning RM）而不是一个通用 RM。

### 6. 业务评测实践

学术 benchmark 和业务评测是两码事——模型在 MMLU 上高分不代表在你的业务场景上好用。

**为什么需要业务专属评测**。通用 benchmark 测的是模型的"通用智力"，但实际业务场景有特定的输入分布（用户怎么问问题）、特定的质量标准（什么算"好回答"在不同业务里完全不同）、特定的约束（响应时间、输出格式、安全合规要求）。一个模型在 HumanEval 上 pass@1=85% 不代表它能在你的项目中正确调用你的 API——因为 HumanEval 只测通用 Python 编程，不测你特定的 SDK 接口。

**构建业务评测集的方法论**。核心步骤：(1) 场景分析——明确业务涉及的能力维度（如实体识别、意图分类、信息摘要、推理规划）；(2) 数据采集——从线上真实用户 query 中采样，确保覆盖常见场景和边角 case（长尾分布的尾部很重要）；(3) 标注标准制定——多个标注者达成一致的评分细则（rubric），明确什么得 5 分、什么得 1 分，每个维度有具体示例；(4) 标注与质检——多标注者独立标注 + 一致性检查（kappa > 0.7）；(5) 持续更新——业务演进后评测集也要更新，否则会出现"模型在旧评测集上分数很高但新场景一塌糊涂"的问题。

**评测维度设计**。业务评测通常需要多维度，典型划分：正确性（事实是否准确）、完整性（是否遗漏关键信息）、相关性（是否答非所问）、安全性（是否包含有害内容）、格式合规（是否满足指定的输出格式）。不同维度的权重根据业务优先级调整——金融场景正确性权重极高（一个错误的数字可能导致巨额损失），而创意写作场景更看重多样性和流畅度。

**评测框架（OpenCompass）**。OpenCompass 是目前中文社区最主流的开源 LLM 评测框架，支持 100+ 标准 benchmark，提供统一的评测流程（模型接入 → 数据加载 → 推理 → 评判 → 报告生成）。它解决了评测的"基建问题"——如果每个团队都自己写评测脚本，结果不可比、不可复现。除 OpenCompass 外，lm-evaluation-harness (EleutherAI)、HELM (Stanford) 也是常用框架。

### 7. 评估中的面试陷阱

**陷阱 1：只看平均分不看方差**。模型 A 在 MMLU 上 85%、模型 B 84%——能说 A 更好吗？不一定。如果 A 的方差大（有些学科 95% 有些 60%），B 的方差小（所有学科都稳定在 82-86%），在实际应用中 B 可能更可靠。面试中提到"除了平均分还要看各子任务的分布"是加分答案。另外评测结果的置信区间很少被报告——在只有 100 道题的子集上差 1-2 个点很可能在统计噪声范围内。

**陷阱 2：Benchmark Saturation（基准饱和）**。当多个模型都在某 benchmark 上达到 95%+ 时，这个 benchmark 失去区分度，不再有价值。GSM8K、HellaSwag 目前都处于这个状态。面试追问"GSM8K 分数有什么局限"——答"它已经饱和了，无法区分当前 SOTA 模型"是满分回答。新的、更难的 benchmark（MATH-500、GPQA、ARC-AGI-2）正在替代这些饱和的旧基准。

**陷阱 3：混淆能力维度**。"模型 X 在推理上超过了人类"——要追问具体是什么推理。数学推理（GSM8K）、代码推理（HumanEval）、常识推理（HellaSwag）、因果推理（COPA）是完全不同的能力。一个模型在数学推理上很强不代表它的因果推理也强。面试中把"推理"拆细是展示深度的方式。

**陷阱 4：忽略评测设置差异**。同一模型在不同评测设置下分数可能差 10+ 个点：0-shot vs 5-shot、greedy decoding vs sampling、system prompt 有无、CoT 有无。比较不同模型时必须在完全相同的设置下进行。如果看到技术报告中某模型声称"超过 GPT-4"但没有说明用的什么设置，要高度怀疑。特别是 temperature=0（greedy）和 temperature=0.7（sampling）在推理任务上可能差 5-15 个点。

**陷阱 5：过度依赖单一评测**。ACL 2025 Keynote 特别强调了这一点——单一 benchmark 无法全面反映模型能力，需要多维度、多任务的评测矩阵。Chatbot Arena 虽然是"最接近真实用户评价"的 benchmark，但它也有偏差：(1) 用户群体不代表所有使用场景（偏技术人群）；(2) 难以评估长期交互中的一致性（每次只评一两轮对话）；(3) 不覆盖安全性等需要专门测试的维度。

### 8. FAQ

**Q: 如何判断一个新 benchmark 是否值得关注？**
看三个维度：(1) 是否已经被主流模型"刷爆"（saturation）——如果 top 模型都 95%+ 则无价值；(2) 是否有数据泄露风险——如果测试集早已公开在网上，所有新模型都可能"背答案"；(3) 是否衡量了有意义的能力——有些 benchmark 只是在测模型对某种特定格式的适应性，不反映真正能力。好的 benchmark 特征是：有清晰的能力定义、有足够的区分度、有防泄露机制（如持续更新题目）。

**Q: Chatbot Arena 的 ELO 分值差 50 意味着什么？**
ELO 差值反映两个模型对战时的胜率预期。差 50 分大约意味着强者有 57% 的胜率；差 100 分约 64% 胜率；差 200 分约 76% 胜率。需要注意 ELO 会随用户群体和提问分布变化——不同时间段的 ELO 不能直接比较（就像棋手的 ELO 不能跨时代比较一样）。

**Q: 什么时候需要做 A/B Test 而不是离线评测？**
当离线评测无法捕获的指标对业务至关重要时——比如用户留存、会话时长、任务完成率等"端到端"指标。模型在离线评测中"正确率高 2%"可能在线上完全没有体感，而"回答格式稍微调整"可能大幅影响用户满意度。A/B Test 是评估的最终仲裁——但成本高（需要真实流量和时间），所以通常先用离线评测大量筛选，再把少数候选拿到线上 A/B。

**Q: Agent 评估的核心难点是什么？**
Agent 的评估比传统 NLU/NLG 评估难一个量级，因为：(1) 过程重要性——Agent 达成同一目标的路径多样，不能只看最终结果，中间步骤的效率和安全性也要评估；(2) 环境依赖——Agent 需要与外部环境交互（API 调用、文件操作、网页浏览），评测需要模拟或真实环境，且环境状态不可精确重现；(3) 组合爆炸——多步任务中每步都有多种选择，正确路径空间巨大，评判标准难以穷举；(4) 安全边界——Agent 有真实世界的操作权限，评估不仅要衡量"能不能完成"还要衡量"过程中是否有危险操作"。目前 Agent 评估还处于早期阶段，没有统一范式。

**Q: Reward Model 和 LLM-as-Judge 有什么区别？**
RM 输出的是一个标量分数（或一对样本的偏好概率），用于训练时的优化信号；LLM-as-Judge 输出的是评判理由 + 分数/排序，用于评测时的质量衡量。RM 追求的是训练效率（梯度信号清晰），可以接受一定的噪声因为 RL 训练有鲁棒性；LLM-as-Judge 追求的是与人类评价的一致性（可解释、可审计）。两者的训练数据源可以相同（都来自人类偏好标注），但优化目标不同。在实践中两者也有融合趋势——用 LLM-as-Judge 的输出构造偏好对来训练 RM（RLAIF），或用 RM 分数来辅助 LLM-as-Judge 做初筛。
